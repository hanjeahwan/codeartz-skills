# Agentic Design Navigator 现实风险矩阵

| 现实场景                                                                   | 关键变量                                 | 预期行为                                                                 | 禁止行为                                 | 失败后果                               | 证据与场景                                                        |
| -------------------------------------------------------------------------- | ---------------------------------------- | ------------------------------------------------------------------------ | ---------------------------------------- | -------------------------------------- | ----------------------------------------------------------------- |
| 程序员已经给出明确的改写、查询或实现请求，只想马上得到结果                 | 是否存在会反转执行结果的意图歧义         | 直接执行，不启动导航                                                     | 因 Skill 可用而强制澄清目标              | 简单任务被拖慢，用户失去控制权         | 现有路由；拟建 `direct-task-bypass` smoke                         |
| 一次性任务仍只有模糊症状，错误理解会导致完全不同的交付                     | 一次性范围、竞争理解、返工成本           | 提供认知增量，必要时最多问一个高价值问题                                 | 因任务只做一次就跳过意图判断             | Agent 快速交付错误对象，返工由用户承担 | 拟建 `vague-one-off-intent` smoke                                 |
| 用户把排行榜、按钮或 AI 助手等当前手段说成目标                             | 目标变化、方案机制、成功变化             | 区分期望变化与当前手段                                                   | 直接把功能清单扩写成需求                 | 方案锁定过早，真正问题被隐藏           | 拟建 `goal-versus-mechanism` full                                 |
| 用户输入不完整，存在两个会导向不同问题定义的合理解释                       | 信息增益、方向反转风险                   | 最多展示两个暂定理解，只澄清会反转方向的未知                             | 为完整而补造主体、价值或边界             | 暂定推断被包装成事实                   | 与 `vague-one-off-intent` 合并覆盖                                |
| 用户没有反对暂定理解，但也没有明确确认                                     | 确认状态、后续决策依赖                   | 保持“暂定”或“有争议”                                                     | 把沉默、继续讨论或接受下一步当成确认     | 错误基线逐轮固化                       | 拟建 `unconfirmed-probe` full                                     |
| 用户说“不知道”或无法回答抽象问题                                           | 可回答性、探针类型                       | 改用一种具体判断探针，仍最多问一个问题                                   | 重复抽象追问、混用多种探针或展开正式方案 | 用户承担诊断劳动，探索停滞             | 与 `unconfirmed-probe` 合并覆盖                                   |
| 用户已给出目标、主体、边界和可观察成功变化，并要求比较方向                 | 基线成熟度、关键取舍                     | 展示基线后比较少量实质不同方向                                           | 无限澄清、生成固定数量功能方案或改写边界 | 决策窗口被错过，方案不可比较           | 拟建 `comparable-directions` full                                 |
| 用户明确替换主体、目的或边界                                               | 变化范围、明确撤销信号                   | 归类为明确转向，更新受影响基线并保留历史方向                             | 把用户转向判成意图漂移或强迫回旧目标     | 用户被旧基线绑架                       | 拟建 `direction-state-change` full                                |
| 用户只修正一项内容，或要求保留主线再探索另一方向                           | 局部修正、并行分支、共享边界             | 只更新受影响项；并行方向使用命名分支                                     | 全量重建基线或用新分支覆盖主线           | 有效决策丢失，分支相互污染             | 与 `direction-state-change` 合并覆盖                              |
| 新机制违反仍有效的不排名、隐私或权限边界                                   | 冲突对象、撤销信号、执行影响             | 显示冲突；没有明确撤销时保持为有争议，必要时请求一次裁决                 | 静默接受机制或自行撤销边界               | 方案造成隐私、权力或信任伤害           | 拟建 `boundary-conflict-and-retry` full                           |
| 用户重复发送相同输入，或要求重新执行上一轮                                 | 语义是否变化、已有状态                   | 没有语义变化时复用现有状态，不重复创建问题、基线或分支                   | 把重试当作新确认或新方向                 | 状态膨胀，同一主题出现互相竞争的副本   | 与 `boundary-conflict-and-retry` 合并覆盖                         |
| 用户开始无关新主题，或明确停止维护当前基线                                 | 主题相关性、结束信号                     | 结束旧主题状态，把新请求独立处理                                         | 把无关任务当成方向变化                   | 旧边界错误套入新任务                   | 拟建 `topic-switch` full                                          |
| 会话压缩、恢复或上下文缺失后继续探索                                       | 可见基线、证据完整性                     | 只使用当前可见且可追溯的状态；关键状态缺失且会反转方向时最多询问一个问题 | 凭模型记忆补造历史确认                   | 用户无法发现基线已被悄悄改写           | 当前 runner 无法制造真实上下文丢失，标记为未覆盖                  |
| 用户未要求持久化设计基线                                                   | 持久化授权                               | 只在当前对话输出                                                         | 默认创建 `.codeartz` 或其他文件          | 仓库出现未经授权的长期状态             | 核心静态检查；无需独立 live scenario                              |
| 用户或宿主明确要求写入共享文件，但文件不可写、写入超时或工作区有并发脏改动 | 明确路径、写权限、目标快照、超时、脏状态 | 保留对话产物并准确报告未写入；不得覆盖并发内容                           | 声称已持久化、回退用户改动或创建替代路径 | 基线丢失或覆盖他人工作                 | 当前 Skill 只规定授权，不负责通用文件事务；标记待裁决是否纳入核心 |
| 多个 Agent 同时维护同一对话外基线                                          | 共享状态所有权、冲突策略                 | 默认不适用；只有宿主提供共享持久化合同时再定义串行或合并策略             | 在无状态合同下自行合并                   | 基线分叉且无法确定权威版本             | 不适用：当前产物默认属于单一对话                                  |
| 导航过程需要外部事实，但检索工具失败                                       | 外部事实是否会反转方向、失败可恢复性     | 将事实标记为待验证；不影响方向时继续，影响方向时停止相关结论             | 把未检索信息写成事实                     | 方向建立在虚假约束上                   | 通用事实边界已由项目规则覆盖；无需复制到 Skill                    |
| 用户要求形成跨项目长期规则                                                 | 当前主题基线、项目规则授权               | 结束导航后交给对应沉淀流程处理                                           | 把设计基线直接升级为长期项目规则         | 暂定探索污染项目规范                   | 不适用：属于 Agent Evolve 或项目知识流程                          |

## 初步结论

- 当前核心已覆盖清晰请求绕过、目标与方案区分、未确认状态、单问题、判断探针、方向变化和边界冲突。
- 优先验证的疑点是重复输入、主题切换和上下文恢复；没有失败证据前不修改核心。
- 文件并发、写权限和超时只在明确持久化分支中成立。是否把该分支纳入 Skill 核心需要单独裁决，不能为填满工程维度自动加入。
- 等价行为合并为八个拟建场景；后续根据风险和可观察性继续删减，不保留只验证措辞的场景。
