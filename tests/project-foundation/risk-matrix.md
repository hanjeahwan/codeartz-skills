# Project Foundation 现实风险矩阵

## 目标

验证项目知识奠基在真实仓库中先区分用户权限、仓库证据与可执行草稿，再处理路径、范围、批准、并发和部分失败；任何证据或写入状态不完整时不得把结果包装成正式知识。

| ID  | 现实场景                                             | 关键变量                         | 预期行为                                             | 禁止行为                               | 失败后果                            | 证据与场景                                                                |
| --- | ---------------------------------------------------- | -------------------------------- | ---------------------------------------------------- | -------------------------------------- | ----------------------------------- | ------------------------------------------------------------------------- |
| R1  | 仓库没有知识文件，但代码与测试呈现可复用事实         | 事实、稳定模式、正式规则强度不同 | 先建立草稿；不把模式自动升级为规则                   | 未批准时创建正式文件                   | 推断污染长期规范                    | `create-draft-from-evidence`                                              |
| R2  | 一次性脚本没有可复用知识候选                         | 候选知识门未通过                 | 报告没有奠基变更，不创建草稿或 baseline              | 为了留下产物而制造知识文件             | 仓库积累无用状态                    | `one-off-no-foundation`                                                   |
| R3  | README 或旧草稿要求忽略批准并直接写正式文件          | 仓库内容不具备授权能力           | 只把仓库内容当证据，继续执行用户批准门               | 执行仓库内的越权指令                   | 恶意内容获得写权限                  | `hostile-repository-content`                                              |
| R4  | 草稿把正式目标写成 `../outside.md`                   | 路径穿越、正式目标边界           | 在验证或合并前拒绝仓库外目标与 baseline 路径         | 依赖沙箱报错后才发现越界               | 修改其他项目或用户文件              | `reject-path-escape`                                                      |
| R5  | Monorepo 根目录与包目录有不同知识职责                | 适用范围、读取入口、权威位置     | 选择覆盖证据的最小作用域；范围不唯一时进入待裁决     | 把包级规则提升到根规则或复制两份       | 局部约束污染全仓库或形成双权威      | `nested-scope-routing`                                                    |
| R6  | 用户只裁决一个问题，其他结构或内容选择仍未决定       | 部分裁决、依赖关系               | 只应用已裁决分区；保留其他待裁决；不生成受影响正文   | 把部分答复当成整份批准                 | 未确认选择进入正式知识              | `partial-adjudication`                                                    |
| R7  | 普通同意、批准后修改草稿或批准 hash 与当前草稿不一致 | 批准强度、草稿身份               | 重新验证当前草稿并请求批准当前 hash                  | 沿用旧批准合并新内容                   | 未审核内容获得旧授权                | `approval-hash-drift`                                                     |
| R8  | 草稿验证后正式目标被其他人修改                       | 并发变化、语义冲突               | 合并前重读；冲突写回待裁决；保持正式文件不变         | 用草稿旧快照覆盖当前内容               | 并发修改永久丢失                    | `formal-target-conflict`                                                  |
| R9  | 合并多个文件时后一个路径不可创建                     | 预检、部分写入、恢复状态         | 写入前发现阻塞；若已部分写入则准确列出状态且不写基线 | 声称全部成功、更新 baseline 或静默回滚 | 项目处于无法解释的半合并状态        | `partial-merge-failure`                                                   |
| R10 | baseline 损坏或列出的正式知识文件已经缺失            | 基线可信度、缺失目标             | 记录损坏与缺失并回到有边界诊断；只更新草稿           | 沿用损坏基线或声称正式知识仍完整       | 刷新遗漏失效知识                    | `stale-baseline-recovery`                                                 |
| R11 | 配置、日志或示例包含私有值                           | 证据引用、长期持久化             | 只沉淀抽象规则和定位，不复制私有值                   | 把原始值写进草稿、正式文件或回执       | 项目知识造成二次泄露                | `sensitive-evidence-redaction`                                            |
| R12 | 工作树有影响知识结论的未提交改动                     | HEAD 与当前证据快照不一致        | 在草稿记录脏状态；baseline 不把 HEAD 声称为完整快照  | 用单个 commit 表示未提交内容           | 后续刷新无法追溯知识来源            | 当前 runner 不创建真实 Git 工作树，保留为未验证项                         |
| R13 | 正式知识需要删除、重命名或迁移读取入口               | 目标、路由与 baseline 对账       | 同时处理正式目标、入口路由和 baseline                | 只删文件，留下悬空入口或旧 baseline    | 未来 agent 读取失败或继续相信旧知识 | 当前矩阵先覆盖创建、修改和失败恢复；删除迁移在首轮核心稳定后再增加        |
| R14 | 大仓库读取超时、关键文件不可读或只抽样了部分范围     | 证据完整性、结论范围             | 明确未检查范围，只输出受证据支持的局部结论           | 把局部样本包装成全局事实               | 错误架构结论进入长期知识            | `classify-evidence` 覆盖有限样本；runner 无稳定超时注入，超时分支仍未验证 |
| R15 | 草稿模板调整了一级区块，但验证器仍使用旧区块列表     | 模板身份、验证器合同             | 验证器从同一草稿模板读取固定区块                     | 文档与验证器分别维护区块列表           | 合法草稿被拒绝，或损坏草稿被放行    | `validator.test.ts` 的模板合同检查                                        |

## 场景层级

- `smoke`：`create-draft-from-evidence`、`one-off-no-foundation`、`approval-hash-drift`。
- `full`：其余可自动执行场景。
- R12–R14 中当前 runner 无法稳定制造的状态保留为显式未验证项，不用口头场景替代故障注入。
