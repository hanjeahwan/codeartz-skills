# 手册写作风格规范

面向「给 agent 看的规则手册」。覆盖两层:

- 系统层:整套手册怎么组织、agent 怎么聚焦。
- 规则层:单条规则怎么写、整篇怎么排。

目标:每条规则都能被独立勾选,分支不会被漏掉,注意力不被深缩进稀释。

一句话总纲:**默认扁平,分支显式,不全篇深嵌套;同一条规则只存在一处。**

---

## 一、系统层

### 1. 文件分工(Skill 与手册关系)

- `SKILL.md` 只放:触发条件、阶段路由、全局边界、禁止动作。
- 阶段手册只放:该阶段的执行规则、检查项、产物规则、输出模板。
- 单一权威位置:同一条规则只存在一处。
- 移动而非复制:内容移到手册后,从 `SKILL.md` 删掉详细内容。
- 禁止用复制代替移动(双份规则会导致后续漂移)。
- 多个阶段手册结构保持一致;内容只服务各自阶段。

### 2. 专注机制

- agent 执行某阶段时,只读该阶段手册。
- 禁止预读后续阶段手册。
- 阶段切换必须由明确条件触发。
- 条件未满足 → 不进入下一阶段产物。

### 3. 编辑纪律

- 编辑 skill 前先判断:内容属于入口路由,还是阶段执行?→ 据此决定放 `SKILL.md` 还是阶段手册。
- 改结构时:检查同类手册的层级,保持一致。
- 改规则内容时:只改权威位置那一处。
- 优先动结构:优先调整分段、标题、排列顺序。
- 保留原文:原有检查句尽量保留原句,新增内容只作分段标题或归类说明,不重写已稳定的句子。
- 语言一致性:项目维护的规则手册、Skill 与面向 agent 的自然语言默认使用中文;协议字段、固定标识、命令、代码符号和无法本地化的专有名词保留原文。

---

## 二、规则层

### 4. 三条铁律

1. **一条规则 = 一个可独立勾选的项。** 读到这一行,不依赖上文就能判断「现在该做什么、不该做什么」。
2. **复合句必须拆。** 「条件 + 动作 + 禁止」不能塞进一句话,否则模型会只做前半句、漏掉尾部禁止。
3. **压平不丢规则。** 把嵌套改扁平时,逐条核对原规则,每条都要能追溯到,不允许在重写中蒸发。

### 5. 规则分两类,写法不同

先判断:这条规则是否存在「满足 X 时做 A,否则做 B」这种**互斥分流**?

- 否(只有一个动作或一个约束)→ **简单规则,扁平写。**
- 是(有条件判断、多个互斥后果)→ **分支规则,用命名槽位写。**

#### 5.1 简单规则:扁平,一行一条

- 缩进 0–1 层。
- 不为了「看起来结构化」而包装。

示例:

```
- 用户给出路径时,优先使用用户路径。
- 方案文件只写方案内容,不写交接文件、探索链路、已查/未查路径。
- 写入失败时,不把本轮说成已产出项目方案,只说明失败原因。
```

#### 5.2 分支规则:命名槽位,而非缩进堆叠

把 if / then / else 显式拆成有名字的槽位,模型无法跳过某个分支。

常用槽位(按需取,不必全用):

- **检查项 / 触发面**:看什么。
- **放行条件**:满足什么就通过。
- **失败条件**:满足什么算命中问题。
- **处理方式**:命中后怎么做。
- **裁决条件**:无法处理时降级到哪。
- **禁止**:明确不能怎么做。

示例(同一规则,正确写法):

```
- 同入口变体检查
  - 检查项:入口/路由/页面/模块级目标行为是否已枚举同入口变体。
  - 放行条件:已枚举且都在已授权分区。
  - 失败条件:存在未授权变体。
  - 处理方式:写入未知 / 冲突 / 待裁决。
  - 禁止:不写成普通风险。
```

对照:**错误写法**(复合句,尾部禁止易被漏)

```
- 路由、页面、模块或入口级目标行为已枚举同入口变体;未授权变体写入未知、冲突或待裁决,不写成普通风险。
```

### 6. 复合句拆分

凡是一句里同时出现下面两类成分,就必须拆开成独立行:

- 条件词:当 / 时 / 若 / 满足……
- 禁止词:不 / 禁止 / 不要 / 不写成……

拆分后:

- 条件单独成行。
- 每个动作单独成行。
- **每个禁止单独成行**,绝不挂在动作句尾。
- 过长的检查句拆成分段或子项,保证每条只检查一个重点。

### 7. 嵌套深度

- 简单规则:0–1 层。
- 分支规则:最多 2 层(规则名 + 槽位),槽位内不再继续向下嵌套子树。
- **超过 2 层 = 信号**:说明你在用缩进表达分支,改成 5.2 的命名槽位。

不要把所有规则都套重结构。简单规则被包装成多层缩进,会拖低整篇的可执行性。

## 三、整篇质量

### 8. 段落顺序

- 单篇顺序:先写目标,再写执行规则,最后写限制与输出要求。
- 层级、标题格式、步骤顺序在同类手册间保持统一。

### 9. 指令质量

- 可执行:多用动词(检查 / 生成 / 禁止 / 保留),避免抽象描述。
- 聚焦:一条指令只表达一个明确动作。
- 无隐含假设:不要让 agent 猜目标、对象、格式或成功标准。
- 优先级清楚:规则可能冲突时,写明哪个指令优先。
- 边界明确:写清何时做、何时不做、不确定时怎么处理。
- 输出明确:指定格式、长度、语气、是否需要引用或列表。
- 避免重复:删除语义相近、作用重叠的内容,防止 agent 不知道优先执行哪条。

### 10. 示例规则

- 只保留能消除歧义的示例;示例过多会干扰主规则。
- 示例只用于解释规则,不能替代规则本身。
- 写通用规则,不要只适配某个特定案例。

### 11. 收尾约束(给「没有则写不适用」这类字段)

- 允许为空的字段,显式写「不适用」,不要留空、不要为填模板补造内容。
- 不要把未授权 / 不在范围内的东西,降级写成「风险」或「不适用」来回避——该进待裁决的进待裁决。

## 四、落地自检

写完后逐项过一遍。

系统层:

- 这条内容放对文件了吗?(路由进 `SKILL.md`,执行进阶段手册)
- 这条规则在别处还有第二份吗?有就删到只剩权威位置。
- 同类手册的层级、标题一致吗?

规则层:

- 这一行能独立勾选吗?还是必须翻上文?
- 句子里有没有「条件 + 动作 + 禁止」挤在一起?有就拆。
- 有分支吗?每个分支后果都显式命名了吗?
- 缩进超过 2 层了吗?超了就改槽位。

整篇:

- 顺序是不是「目标 → 执行 → 限制/输出」?
- 有没有语义重叠、会让 agent 不知道先执行哪条的内容?
- 示例是在解释规则,还是在替代规则?
- 改写后,原来的每条规则还都在吗?
