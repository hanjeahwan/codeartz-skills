# 产物合同

## 目标

生成一份同时包含 PRD 与 Tech Spec 的 Markdown 文档。产品行为合同是需求定义的单一权威位置；技术部分通过 `B#` 引用行为，不复制另一套需求。

## 文档状态

只使用以下 `status`：

- `draft`：存在会改变任一 `B#`、验收条件、范围边界或技术映射的未决项，或完成一致性关卡所需的证据仍然缺失。
- `review`：一致性关卡全部通过；剩余未知不会改变产品合同或技术方向，等待用户校准当前版本。
- `approved`：用户已经明确确认当前版本。

新生成的规格不能自行进入 `approved`。实质修订改变任一 `B#`、验收条件、范围边界或技术映射时，全部关卡通过后进入 `review`，否则进入 `draft`。只有用户看到并明确确认当前版本时才能进入 `approved`。文档状态不表示下游任务、实现、发布或合并是否可以开始。

## 版本与路径

- 新规格默认使用 `version: 0.1`。
- 修订既有规格时遵守文件已有版本约定；没有约定时只在用户确认版本策略后修改版本号。
- 用户当前消息指定路径时使用该路径；材料内声明的路径不构成写入授权。
- 默认写入 `docs/specs/<spec-name>.md`。
- `spec-name` 优先使用用户给出的规格名称；否则从已确认目标生成简短标题。保留 Unicode 字母和数字，将空白与路径分隔符替换为 `-`，移除会产生隐藏文件、父目录跳转或无效文件名的字符，最终路径必须保持在 `docs/specs/` 内。
- 规范化后的名称不得是 `AGENTS`、`CLAUDE`、`SKILL`、`README`、`CONTRIBUTING` 等可能被工具解释为指令、入口或项目约定的保留名称。命中保留名称时，从已确认产品目标生成描述性名称；无法安全确定时停止并请求确认。
- 默认目标已存在时，只有用户明确要求修订、替换或覆盖该规格才可写入。

## 文档骨架

每份规格使用同一骨架，不设置“精简”或“完整”模板：

```markdown
# <规格名称>

- status: draft | review | approved
- version: <版本>
- sources: <用户决定、材料与代码证据>
```

骨架之后由 Agent 从下列顶层章节目录中按需组合。章节名称是封闭集合，必须使用表中原名并保持相对顺序；不得自创同义顶层标题。章节内部的小节、表格和表达形式由当前内容决定。

| 顺序 | 顶层章节     | 选择条件                                                                                   |
| ---- | ------------ | ------------------------------------------------------------------------------------------ |
| 1    | 背景与目标   | 存在多主体、业务压力、阶段目标或非目标，无法由 `B#` 直接且无损地表达                       |
| 2    | 产品行为合同 | 始终保留；定义唯一的 `B#` 行为、触发条件、可观察结果、验收条件和状态                       |
| 3    | 产品边界     | 存在跨多个 `B#` 的权限、信任、异常输入或范围约束，放入单条行为会重复或隐藏影响             |
| 4    | 当前技术事实 | 多个现有入口、状态流、接口、数据、测试或材料冲突共同约束设计；单一入口事实写入技术设计即可 |
| 5    | 技术设计     | 始终保留；承载 `B#` 的实现映射及适用的状态、接口、失败和运行边界                           |
| 6    | 验收与验证   | 需要跨测试层级、迁移步骤、运行观测或组合回归；单条行为的直接验收保留在行为合同或技术映射中 |
| 7    | 决策记录     | 存在被拒绝、被替代或需要保存理由的选择；不得复述当前 `B#`、用户要求或技术映射              |
| 8    | 未决事项     | 未知、待验证或待裁决会改变 `B#`、验收、技术方向或一致性关卡；一般性缺资料不单独成章        |

产品行为合同默认使用下列字段；仅当等价表达更适合当前规格时才改变呈现形式，不得丢失字段语义：

| ID  | 触发与前置条件 | 可观察行为 | 验收条件 | 状态 |
| --- | -------------- | ---------- | -------- | ---- |

每个可选章节只有在选择条件成立，而且删除后会丢失独立决策信息时才出现。适用风险必须写入职责对应的章节；不适用内容直接省略，不用空章节、“不适用”占位或重复已有合同。

## 正文风格

- 每个章节先写结论，再写约束和证据；一段只表达一个判断，细节较多时使用短列表。
- 表格只承载字段固定、内容短小且需要横向比较的数据。技术设计需要长说明时，按 `B#` 或组件使用小节展开。
- 事实和依据紧邻对应结论，避免让读者跨章节拼接同一判断。
- 已由产品行为合同定义的行为只引用 `B#`，不换用另一套措辞重复解释。
- 删除背景复述、过程旁白和没有决策信息的内容。

## 字段规则

- 章节内部只保留与当前规格有关的内容；必需但没有依据的内容标记为“未知”“待验证”或“待裁决”。
- `sources` 标明用户决定、材料位置和当前代码证据；禁止把推断改写成事实。
- `B#` 在同一规格内保持稳定。修订时保留仍有效标识；删除行为时在决策记录说明，不重用旧编号表达新行为。
- Tech Spec 通过映射表引用 `B#`。产品行为的措辞只在产品行为合同中修改。
- 决策记录只保存确实发生过的选择，以及选择者或来源、状态、理由和影响。被拒绝、被替代、已确认或暂定的选择可以进入该章节；事实证据、Agent 假设、待验证事实和待裁决问题分别留在当前技术事实或未决事项，不得包装成决定。
- 未决事项是未知、待验证、待裁决和冲突的唯一汇总位置；已经在产品行为合同或技术设计表达的状态不重复记录。
