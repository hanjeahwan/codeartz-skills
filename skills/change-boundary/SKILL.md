# Change Boundary

## 使用时机

当任务同时涉及需求资料和既有系统行为时，使用该技能。典型输入包括用户需求、PRD、spec、issues、会话记录、旧方案、review 评论、截图、日志、diff、仓库路径或测试输出。

本技能的目标是输出方案。不要直接写实现计划、测试矩阵、交付说明或代码。

## 默认视角

以高级全栈程序员视角工作，不绑定任何语言、框架或平台。分析前端、后端、数据、权限、异步任务、第三方集成、构建部署、测试观测和运行时环境之间的边界。

把用户资料当作目标和假设来源，把代码事实、外部事实和运行证据当作系统现实来源。不要把 PRD、issue、旧方案或用户总结直接当成当前系统事实。

## 核心术语

- 非结构化资料：用户需求、PRD、spec、issues、会话、截图、现有方案、review 评论、事故记录、日志或仓库线索。
- 资料假设：从非结构化资料中抽出的可验证声明。资料假设不是系统事实。
- 代码事实：由代码入口、调用链、类型、测试、配置、运行脚本、迁移、日志或依赖封装层证明的当前系统行为。
- 外部事实：由官方文档、源码、协议、平台限制或项目固定版本依赖证明的外部约束。
- 宽范围对象：资料或代码里被上层描述覆盖的对象集合，例如模块、功能、流程、对象类别、用户群体、状态集合或路径集合。
- 窄语义行为：具体且可观察的行为，例如阻塞、跳过、捕获、弹窗、安装、保持原行为、直接通过、发送通知、写入数据或触发副作用。
- 语义分区：宽范围对象下面语义不同的子集合，来源可以是状态、类型、生命周期、权限、失败类型、副作用归属、环境、旧兼容路径或用户原话。
- 反证分区：明确不适用、不得改变、保持原行为、跳过、例外、直接通过或只在某范围适用的分区。
- 未知分区：没有足够资料或代码证据判断授权状态的分区。
- 冲突分区：资料、代码、外部事实或用户目标之间互相矛盾的分区。
- 目标合同：目标行为适用分区、不适用分区、保持原行为、输入输出、状态变化、副作用、失败行为、用户可见变化和兼容约束。

## 核心规则

- 先抽资料假设，再读代码事实。
- 声称当前系统行为、系统现实或根因前，先读取仓库证据。
- 涉及外部库、SDK、协议、平台限制或版本差异时，先读本仓库依赖版本、封装层、配置和类型定义；仍影响方案时，再查官方文档、源码或协议。
- 上层归属、同一路径、同一入口、同一模块、同一 owner 或一致性诉求，不等于子分区被授权使用同一行为。
- 专门搜索反证分区和未知分区。发现反证时，不用相邻分区授权覆盖它；发现未知时，不用假设覆盖它。
- 风险只能提高证据深度和约束方案，不能授权新目标行为。
- 外部事实只能约束可行性、平台行为、风险和发布条件，不能授权本仓库新增分区行为。
- 重要结论必须能沿“证据 -> 资料假设或代码事实 -> 语义分区 -> 目标合同 -> 方案”追溯；链路中断时，结论降级为未知、冲突或候选。
- subagent 默认只读。主 agent 必须复核原始证据，才能把 subagent 输出升级为代码事实、边界、目标合同或方案。

## 单轮执行

每次触发技能或收到用户新消息时，按以下顺序执行。轻量任务可以压缩展示，但不能压缩判断。

1. 判断运行模式和本轮停靠点。
2. 确认当前步骤、证据等级、宽范围对象、窄语义行为、反证分区、未知分区、冲突分区和下一步。
3. 判断新消息是否改变目标、分区、代码事实、外部事实、风险或输出深度；改变时回到最早受影响步骤。
4. 执行当前步骤所需的最小证据动作。
5. 检查 gate。gate 失败时，只输出缺口、限制和下一步证据动作。
6. 响应前检查每个结论是否超过当前证据等级；超过时删除、降级或改成候选。

## 运行模式

先判断用户要什么深度。模式只决定停在哪里，不降低证据门槛。

- 资料整理模式：用户只要求理解资料。停在资料假设、初始分区、未知和待代码验证清单。
- 代码分析模式：用户要求判断现有行为、影响范围或可行性。至少推进到系统现实；涉及 bug、事故、异常或行为偏差时，再推进到根因判断。
- 方案设计模式：用户要求方案、路径比较、技术取舍或代码变更建议。推进到目标合同和方案。
- Review 模式：用户要求 review 方案、diff、PR 或半成品。先从现有方案或 diff 反推目标行为、触及分区和隐性副作用，再回到资料假设、代码事实、语义边界和目标合同。

用户请求不清楚时，选择不会越权的最浅模式。快速判断可以给临时结论，但必须标明证据等级、限制和下一步最小证据动作。

## 证据等级

- L0 未知：缺资料或代码证据，只能列补证据动作。
- L1 资料假设：有用户、PRD、spec、issue、会话或旧方案证据，但没有代码验证。
- L2 代码事实：已读取入口、调用链、类型、配置、测试、运行脚本或副作用路径，能说明当前系统行为。
- L3 可执行合同：已有语义边界、代码事实、目标合同和方案约束。

证据等级不能上跳。没有代码证据，不能从 L1 写成 L2 或 L3；没有验证结果，不能使用“已验证”“已修复”“完成”这类结论。

## 证据记录

记录能改变判断的证据，不做形式化账本。每条关键证据至少保留：

- 来源类型：用户原话、PRD、spec、issue、会话、代码、测试、配置、运行脚本、日志或官方文档。
- 定位信息：文件路径、符号名、测试名、配置名、命令、用户原句或资料位置。
- 证据内容：该证据实际说明了什么。
- 影响对象：它影响哪个宽范围对象或语义分区。
- 影响行为：它支持、反驳或限制哪个窄语义行为。
- 证据状态：支持、反证、限制、冲突或未知。

资料证据只能形成资料假设。代码、测试、配置和运行脚本才能确认系统事实。日志和指标可以支持运行时判断，但不能替代代码边界分析。

## 执行流程

按顺序推进。用户只要状态时，只输出当前步骤、已知事实、缺口和下一步。

1. **非结构化资料**
   收集用户需求、PRD、spec、issues、会话、旧方案、review、截图、日志和仓库线索。标出来源、定位、时间或版本、可形成结论和证据等级，不合并成结论。

2. **资料假设**
   抽出用户目标、明确排除项、宽范围对象、窄语义行为、拟议目标行为、冲突点和待验证声明。资料假设不是系统事实。

3. **代码事实**
   读取入口、调用链、类型、schema、状态、配置、测试、运行脚本、依赖封装和副作用归属。记录搜索词、已查路径、未查路径、命中、未命中和停止原因；只把有仓库证据支撑且搜过反证的判断写成代码事实。

4. **语义边界**
   枚举宽范围对象下的语义分区，包括状态、类型、生命周期、权限、环境、失败类型、兼容路径、副作用归属和用户原话分区。定义允许变更面、禁止扩散面、未知默认和冲突处理；专门找反证分区和未知分区。

5. **系统现实**
   合并资料假设和代码事实，说明当前系统真实行为、哪些资料被证实、哪些被推翻、哪些仍未知。

6. **根因判断**
   如果任务是 bug、事故、异常或行为偏差，判断根因。根因必须来自代码事实和系统现实。不是故障类任务时，写“不适用”。

7. **目标合同**
   定义目标行为适用哪些分区、不适用哪些分区、哪些保持原行为，以及输入输出、状态变化、副作用、失败行为、用户可见变化和兼容约束。每个字段写清变化前、变化后、保持不变或未知；失败行为和兼容约束不能留到实现时再决定。

8. **方案**
   输出推荐方案。只说明怎么改、改哪里、为什么这样改、哪些不改、风险是什么。不要进入实现计划、测试矩阵或交付说明。

## 证据获取策略

- 先从用户点名资料和仓库入口开始。
- 用领域词、类型名、状态名、错误名、配置名、测试名、事件名、权限名和数据字段扩展搜索。
- 从入口向下追踪调用链，从测试向上追踪行为契约。
- 优先读取会改变判断的文件：类型、schema、状态、权限、配置、测试、迁移、副作用和外部调用。
- 对每个授权分区至少寻找一个支持证据和一个反证搜索结果。
- 对每个禁止、未知或冲突分区记录为什么不能继续扩展。

可以停止搜索的条件：

- 证据已足够支撑当前运行模式的 gate。
- 新增证据只重复已有结论，不再改变分区、边界、根因或目标合同。
- 搜索空间过大，需要先向用户汇报证据缺口和下一批范围。
- 仓库或依赖不可访问，相关判断已降级为假设或未知。

不能停止搜索的情况：

- 只找到正向证据，尚未搜索反证分区。
- 只读到调用链入口，尚未确认状态、数据契约或副作用。
- 只读到实现，尚未读测试、配置或现有行为约束。
- 已发现代码和资料冲突，但未定义冲突处理。
- 变更涉及高风险分区，但未检查回滚、观测、发布或补偿影响。

## Gate

- 没有资料假设，不输出代码事实结论。
- 没有代码事实，不输出系统现实、根因判断、目标合同或方案。
- 没有语义边界，不输出目标合同。
- 没有目标合同，不输出方案。
- 不适用范围写不出来时，说明语义边界未建立，不输出方案。
- 资料、代码、外部事实或用户目标冲突未解决时，不把冲突分区纳入方案。
- 目标合同没有覆盖适用分区、不适用分区、保持原行为、输入输出、状态变化、副作用、失败行为、用户可见变化和兼容约束时，不输出推荐方案。
- 推荐方案无法追溯到代码事实、语义边界、根因判断和目标合同时，只能输出方案候选、合同缺口或回流步骤。

Gate 失败时，只输出缺口、缺失证据、被阻止的结论和下一步证据动作。

## 降级输出

- 无仓库或代码不可访问时，只输出 L1 资料假设、初始分区、候选代码路径和补证据清单。用户要求方案时，只能给资料层候选方案，并说明哪些判断依赖代码验证。
- 用户要求快速判断时，输出“资料上看”“代码证据支持”“仍未知”三类结论，不把快速判断包装成最终方案。
- 代码事实不足时，可以输出候选系统现实和待验证根因，但不能输出确定根因、目标合同或推荐方案。
- 目标合同不完整时，可以输出合同缺口、冲突选项和保守默认，不输出方案模板的推荐结论。
- 外部事实无法确认时，把受影响的根因、目标合同或方案字段标为候选或未知，并写明需要确认的版本、封装层或官方来源。

## 风险和外部事实

遇到以下风险时，提高证据深度，并把风险落到影响分区、必查代码证据、方案限制和待裁决点：权限和身份、数据完整性、副作用、异步一致性、兼容性、运行环境、性能和容量、安全和隐私、可访问性和国际化、可观测性。

风险不能授权目标行为。不得因为风险规避新增阻塞、弹窗、捕获、安装、跳过、通知、数据暴露、外部写入、扣费或后台任务。

涉及外部事实时，先定位本仓库使用方式：锁文件、依赖声明、封装层、调用点、配置、类型定义和测试。只有接口语义、版本差异或平台限制会影响方案时，再查官方文档、源码或协议。无法确认外部事实时，把相关根因、目标合同或方案降级为候选或未知。

## 回流规则

发现新资料、新代码事实、新反证、新未知分区、外部事实冲突、风险变化或用户目标变化时，回到最早受影响步骤，重建后续结论。不要只在最终回答补一句。

- 用户追加需求、例外、取消分支或改变目标：回到非结构化资料；只改变目标行为时回到目标合同。
- 新代码入口、状态、类型、权限、兼容路径、skip、no-op 或 fallback：回到代码事实；影响边界时继续回到语义边界。
- 推荐方案需要未授权分区、未知分区、反证分区或新副作用：回到方案；如果合同变化，回到目标合同。
- 外部事实或风险改变目标行为、边界或方案可行性：回到最早受影响步骤，并降低证据等级。

多个回流来源同时命中时，采用最早受影响步骤、最高风险、最低证据等级和最保守输出。

## 方案选择规则

- 先生成最小可行方案。它必须满足目标合同，改动面最小，并让反证分区、未知分区和保持原行为分区不受新行为影响。
- 推荐方案只能选择已授权目标行为的实现路径。方案不能新增目标行为、扩大适用分区、删除保持原行为分区或改变兼容承诺。
- 备选方案只保留会改变执行结果、风险、成本、发布时间、验证义务或长期维护的路径；纯风格差异不列为备选。
- 被拒方案必须写具体故障模式：越界分区、破坏合同字段、无法验证、回滚困难、兼容破坏、风险不可接受或维护成本失控。
- 取舍必须说明代价归属和爆发时机。用户需要承担产品范围、兼容破坏、发布窗口或成本变化时，输出用户裁决点。
- 推荐方案不能写成已实现、已验证或生产安全。

## 方案输出模板

输出方案时使用以下结构。信息不足时，不填充猜测；写明缺口并停在当前 gate。

```markdown
## 方案

### 推荐结论

[一句话说明推荐怎么做；如果证据不足，标为候选方案。]

### 适用范围

- [哪些分区适用该方案。]
- [哪些入口、模块、对象、状态或用户路径会受影响。]

### 不适用范围

- [哪些反证分区不改。]
- [哪些未知分区暂不纳入。]
- [哪些保持原行为。]

### 依据

- 资料假设：[来自需求、PRD、issue、会话或旧方案的关键假设。]
- 代码事实：[来自入口、调用链、类型、状态、测试、配置或副作用归属的事实。]
- 系统现实：[资料和代码合并后，当前系统真实表现。]
- 根因判断：[如果是 bug/事故/异常任务，写根因；否则写“不适用”。]
- 证据等级：[L1/L2/L3，以及不能声称的内容。]

### 目标合同

- 目标行为：[新增、修改、删除或保持什么行为。]
- 输入输出：[输入、输出、状态变化。]
- 副作用：[写数据、发请求、发通知、缓存、权限、异步任务等影响。]
- 失败行为：[失败、跳过、fallback、直接通过或报错方式。]
- 兼容约束：[旧路径、旧数据、旧 API、旧 UI 或旧行为如何保持。]

### 变更方向

- [建议改动的模块、边界或抽象层。]
- [为什么改这里，而不是改相邻路径。]
- [如何避免把行为扩散到未授权分区。]
- [方案需要保留的实现路径约束、验证前置或发布/回滚/观测前置。]

### 被拒方案

- 方案 A：[被拒方案。]
  故障模式：[会误改什么、漏掉什么、破坏什么边界。]

### 风险和待裁决

- 风险：[仍可能爆发的问题。]
- 待裁决：[需要用户确认的范围、行为或取舍。]
- 待补证据：[还缺哪些代码事实或外部事实。]
- 不能声称：[由于证据等级或 gate 限制，本轮不能声称的结论。]
```

## 禁止动作

- 不把资料假设写成代码事实。
- 不用一致性、完整覆盖或风险规避补足语义授权。
- 不把未知分区默认纳入目标行为。
- 不用相邻分区授权覆盖反证分区。
- 不把用户表象当根因，不把止血包装成根因修复，不把局部修复说成全局方案。
- 不用测试通过、同路径、同模块、owner 归属或框架最佳实践证明语义边界正确。
