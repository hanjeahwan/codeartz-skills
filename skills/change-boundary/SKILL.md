# Change Boundary

## 使用时机

当任务同时涉及需求资料和既有系统行为时，使用该技能。典型输入包括用户需求、PRD、spec、issues、会话记录、旧方案、评审评论、截图、日志、diff、仓库路径或测试输出。

## 产物流

本技能有两个文件产物和一个确认停靠点：

1. 方案阶段：完成证据、边界、目标合同和方案后，写入 `.swe/<topic>/change-boundary.md`。
2. 确认停靠点：写完方案后，停止在对话里，要求用户明确确认三件事：接受方案、理解风险、裁决待裁决点或确认无待裁决点。
3. 交接阶段：只有确认停靠点完成后，才读取方案文件并写入 `.swe/<topic>/handoff.md`。

用户催促、询问流程、要求继续或问是否该生成 handoff，不等于接受方案、理解风险或裁决待裁决点。缺确认时，只输出确认清单，不生成交接文件。对话回复只给路径和摘要，完整内容以项目文件为准。

## 分析范围

按高级全栈程序员接手真实项目的顺序工作，不绑定任何语言、框架或平台。定位前端入口、后端入口、数据契约、权限、异步任务、第三方集成、构建部署、测试观测和运行时环境之间的边界。

把用户资料当作目标和假设来源，把代码事实、外部事实和运行证据当作系统现实来源。不要把 PRD、issue、旧方案或用户总结直接当成当前系统事实。

## 核心术语

- 非结构化资料：用户需求、PRD、spec、issues、会话、截图、现有方案、评审评论、事故记录、日志或仓库线索。
- 资料假设：从非结构化资料中抽出的可验证声明；不能当作系统事实。
- 代码事实：由代码入口、调用链、类型、测试、配置、运行脚本、迁移、日志或依赖封装层证明的当前系统行为。
- 外部事实：由官方文档、源码、协议、平台限制或项目固定版本依赖证明的外部约束。
- 宽范围对象：资料或代码里被上层描述覆盖的对象集合，例如模块、功能、流程、对象类别、用户群体、状态集合或路径集合。
- 窄语义行为：具体且可观察的行为，例如阻塞、跳过、捕获、弹窗、安装、保持原行为、直接通过、发送通知、写入数据或触发副作用。
- 语义分区：宽范围对象下面语义不同的子集合，来源可以是用户原话、代码路径、类型、状态、生命周期、对象类别、权限、环境、失败类型、副作用归属、旧兼容路径或文档约束。
- 同入口变体：同一路由、页面、模块、命令、任务或事件入口下，由运行模式、对象类别、权限层级、来源渠道、生命周期、状态族、环境、feature flag、合同分支、失败/恢复上下文或副作用归属切出的语义分区。
- 反证分区：明确不适用、无需参与、不得改变、保持原行为、不得阻塞、不得安装、不得弹出、不得捕获、跳过、例外、直接通过或只在某范围适用的分区。
- 未知分区：没有足够资料或代码证据判断授权状态的分区。
- 冲突分区：资料、代码、外部事实或用户目标之间互相矛盾的分区。
- 目标合同：目标行为适用分区、不适用分区、保持原行为、输入输出、状态变化、副作用、失败行为、用户可见变化和兼容约束。

## 核心规则

- 先抽资料假设，再读代码事实。
- 声称当前系统行为、系统现实或根因前，先读取仓库证据。
- 涉及外部库、SDK、协议、平台限制或版本差异时，先读本仓库依赖版本、封装层、配置和类型定义；仍影响方案时，再查官方文档、源码或协议。
- 用户要求“直接给方案”“不用分析”“先按经验来”时，不降低证据门槛。没有代码事实和语义边界时，只输出候选方向、缺口和下一步证据动作。
- 上层归属、同一路径、同一入口、同一模块、同一归属方或一致性诉求，不等于子分区被授权使用同一行为。
- 用户给出路由、页面、模块或入口级目标行为时，先枚举同入口变体。未被用户或仓库证据授权的变体保持未知或待裁决，不降级成普通风险。
- 给分区应用窄语义行为前，先写出授权证据。授权只来自用户明确点名该分区使用该行为，或仓库代码、测试、配置、迁移、类型或文档合同把同一行为绑定到该分区。
- 未枚举到的子分区保持未知，不随上层对象、同一路径、同一入口或同一归属方自动纳入方案。
- 反证分区优先于上层边界。上层归属允许分析该范围，不等于允许对每个子分区应用同一窄语义行为。
- 专门搜索反证分区和未知分区。发现反证时，不用相邻分区授权覆盖它；发现未知时，不用假设覆盖它。
- 风险只能提高证据深度和约束方案，不能授权新目标行为。
- 外部事实只能约束可行性、平台行为、风险和发布条件，不能授权本仓库新增分区行为。
- 重要结论必须能沿“证据 -> 资料假设或代码事实 -> 语义分区 -> 目标合同 -> 方案”追溯；链路中断时，结论降级为未知、冲突或候选。
- 默认只读取仓库并写 `.swe/` 产物。不要创建 branch、worktree、提交或修改实现文件；用户明确切换到实现任务时，先完成或引用方案边界，再进入开发流程。
- subagent 默认只读。主 agent 必须复核原始证据，才能把 subagent 输出升级为代码事实、边界、目标合同或方案。

## 单轮执行

每次触发技能或收到用户新消息时，按以下顺序执行。轻量任务可以压缩展示，但不能压缩判断。

1. 判断运行模式、产物模式和本轮停靠点。
2. 确认当前步骤、证据等级、宽范围对象、窄语义行为、反证分区、未知分区、冲突分区和下一步。
3. 判断新消息是否改变目标、分区、代码事实、外部事实、风险或输出深度；改变时回到最早受影响步骤。
4. 执行当前步骤所需的最小证据动作。
5. 检查关卡。关卡失败时，只输出缺口、限制和下一步证据动作。
6. 响应前检查每个结论是否超过当前证据等级；超过时删除、降级或改成候选。

## 运行模式

先判断用户要什么深度。模式只决定停在哪里，不降低证据门槛。

- 资料整理模式：用户只要求理解资料。停在资料假设、初始分区、未知和待代码验证清单。
- 代码分析模式：用户要求判断现有行为、影响范围或可行性。至少推进到系统现实；涉及故障、事故、异常或行为偏差时，再推进到根因判断。
- 方案设计模式：用户要求方案、路径比较、技术取舍或代码变更建议。推进到目标合同和方案。
- 评审模式：用户要求评审方案、diff、PR 或半成品。先从现有方案或 diff 反推目标行为、触及分区和隐性副作用，再回到资料假设、代码事实、语义边界和目标合同。

用户请求不清楚时，选择不会越权的最浅模式。快速判断可以给临时结论，但必须标明证据等级、限制和下一步最小证据动作。

## 产物模式

- 方案文件模式：按 `references/change-boundary.md` 的方案阶段手册写入 `.swe/<topic>/change-boundary.md`。
- 交接文件模式：按产物流的确认停靠点和 `references/handoff.md` 的交接文件阶段手册生成 `.swe/<topic>/handoff.md`。
- 对话降级模式：项目不可写、用户只要求对话回答，或仓库/代码不可访问时，只在对话输出；仓库/代码不可访问时按证据等级降级。

## 证据等级

- L0 未知：缺资料或代码证据，只能列补证据动作。
- L1 资料假设：有用户、PRD、spec、issue、会话或旧方案证据，但没有代码验证。
- L2 代码事实：已读取入口、调用链、类型、配置、测试、运行脚本或副作用路径，能说明当前系统行为。
- L3 可执行合同：已有语义边界、代码事实、目标合同和方案约束。

证据等级不能上跳。没有代码证据，不能从 L1 写成 L2 或 L3；没有验证结果，不能使用“已验证”“已修复”“完成”这类结论。

## 证据记录

记录能改变判断的证据，不做形式化账本。每条关键证据至少保留：

- 来源类型：用户原话、PRD、spec、issue、会话、代码、测试、配置、运行脚本、日志或官方文档。
- 定位信息：文件路径、符号名、测试名、配置名、命令、用户原句或资料位置。
- 证据内容：该证据实际说明了什么。
- 影响对象：它影响哪个宽范围对象或语义分区。
- 影响行为：它支持、反驳或限制哪个窄语义行为。
- 证据状态：支持、反证、限制、冲突或未知。

资料证据只能形成资料假设。代码、测试、配置和运行脚本才能确认系统事实。日志和指标可以支持运行时判断，但不能替代代码边界分析。

## 执行流程

按顺序推进。用户只要状态时，只输出当前步骤、已知事实、缺口和下一步。

1. **非结构化资料**
   收集用户需求、PRD、spec、issues、会话、旧方案、评审评论、截图、日志和仓库线索。标出来源、定位、时间或版本、可形成结论和证据等级，不合并成结论。

2. **资料假设**
   抽出用户目标、明确排除项、宽范围对象、窄语义行为、拟议目标行为、冲突点和待验证声明。资料假设保持为待验证声明，不能当作系统事实。

3. **代码事实**
   先定位用户点名入口，再追调用链、类型、schema、状态、配置、测试、运行脚本、依赖封装和副作用归属。记录搜索词、已查路径、未查路径、命中、未命中和停止原因；只把有仓库证据支撑且搜过反证的判断写成代码事实。

4. **语义边界**
   枚举宽范围对象下的全部已知语义分区，包括用户原话、代码路径、类型、状态、生命周期、对象类别、权限、环境、失败类型、兼容路径、副作用归属和文档约束。路由、页面、模块或入口级对象还要枚举同入口变体。不要只列本轮想修改的分区；定义允许变更面、禁止扩散面、未知默认和冲突处理；专门找反证分区和未知分区。

5. **系统现实**
   合并资料假设和代码事实，说明当前系统真实行为、哪些资料被证实、哪些被推翻、哪些仍未知。

6. **根因判断**
   如果任务是故障、事故、异常或行为偏差，判断根因。根因必须来自代码事实和系统现实。非故障类任务写“不适用”。

7. **目标合同**
   定义目标行为适用哪些分区、不适用哪些分区、哪些保持原行为，以及输入输出、状态变化、副作用、失败行为、用户可见变化和兼容约束。每个字段写清变化前、变化后、保持不变或未知；失败行为和兼容约束不能留到实现时再决定。

8. **方案**
   读取 `references/change-boundary.md`，按方案阶段手册完成写入前检查、产物规则和输出模板。通过时写入 `.swe/<topic>/change-boundary.md`；失败时只输出合同缺口、证据缺口或回流步骤。方案只说明怎么改、改哪里、为什么这样改、哪些不改、风险是什么；不要进入实现计划、测试矩阵或交付说明。

9. **交接文件**
   到达产物流的确认停靠点后，读取方案文件和 `references/handoff.md`，按交接文件阶段手册生成 `.swe/<topic>/handoff.md`。

## 证据获取策略

按最短能改变判断的路径取证。每轮记录搜索词、已查路径、未查路径、命中、未命中和停止原因。

1. 从用户点名资料和仓库入口开始；用领域词、类型名、状态名、错误名、配置名、测试名、事件名、权限名和数据字段扩展搜索。
2. 找入口：路由、页面、组件、API、命令、任务、队列、事件、定时器、迁移或测试入口。
3. 找合同：类型、schema、接口、权限、配置、feature flag、环境变量、迁移、文档或测试断言。
4. 找状态、数据和副作用：状态流、生命周期、缓存、持久化字段、读写路径、旧数据、兼容分支、外部请求、通知、写入、安装、扣费、权限变更、异步任务、日志和指标。
5. 找反证：不适用、无需参与、不得阻塞、不得安装、不得弹出、不得捕获、skip、no-op、fallback、直接通过、保持原行为、例外、只在特定分区适用的分支。
6. 给每个授权分区记录支持证据和反证搜索结果；给每个禁止、未知或冲突分区记录不能继续扩展的原因。

满足以下条件时停止搜索：

- 证据已支撑当前运行模式的关卡，新增证据只重复已有结论，不再改变分区、边界、根因或目标合同。
- 搜索空间过大，需要先向用户汇报证据缺口和下一批范围。
- 仓库或依赖不可访问，相关判断已降级为假设或未知。

以下情况继续搜索或回流，不输出更高等级结论：

- 只有正向证据，还没有搜索反证分区。
- 只读到入口或实现，还没有确认状态、数据契约、副作用、测试、配置或现有行为约束。
- 已发现代码和资料冲突，但未定义冲突处理。
- 变更涉及高风险分区，但未检查回滚、观测、发布或补偿影响。

## 关卡

- 资料假设缺失时，不输出代码事实结论。
- 代码事实缺失时，不输出系统现实、根因判断、目标合同或方案。
- 语义边界缺失时，不输出目标合同；不适用范围写不出来，视为语义边界缺失。
- 目标合同缺失时，不输出方案；合同必须覆盖适用分区、不适用分区、保持原行为、输入输出、状态变化、副作用、失败行为、用户可见变化和兼容约束。
- 冲突分区未解决、未知分区阻塞目标行为或反证分区被目标行为覆盖时，不输出推荐方案。
- 推荐方案不能追溯到代码事实、语义边界、目标合同和适用时的根因判断时，只输出方案候选、合同缺口或回流步骤。

关卡失败时，只输出缺口、缺失证据、被阻止的结论和下一步证据动作。

## 降级输出

- 无仓库或代码不可访问时，只输出 L0/L1 资料假设、初始分区、候选代码路径和补证据清单。资料不足时保持 L0；资料足够时才给资料层候选方案，并说明哪些判断依赖代码验证。
- 用户要求快速判断时，输出“资料上看”“代码证据支持”“仍未知”三类结论，不把快速判断包装成最终方案。
- 用户要求跳过分析或直接给方案时，保持关卡。代码事实、语义边界或目标合同缺失时，不写方案文件，只输出候选方向、被阻止的结论和最小补证据动作。
- 代码事实不足时，可以输出候选系统现实和待验证根因，但不能输出确定根因、目标合同或推荐方案。
- 目标合同不完整时，可以输出合同缺口、冲突选项和保守默认，不写方案文件或推荐结论。
- 外部事实无法确认且会影响根因、目标合同或推荐方案时，不输出推荐方案；只输出受阻字段、需要确认的版本、封装层或官方来源。不影响推荐结论时，写入非阻塞待补证据。

## 风险和外部事实

遇到以下风险时，提高证据深度，并把风险落到影响分区、必查代码证据、方案限制和待裁决点：权限和身份、数据完整性、副作用、异步一致性、兼容性、运行环境、性能和容量、安全和隐私、可访问性和国际化、可观测性。

风险不能授权目标行为。不得因为风险规避新增阻塞、弹窗、捕获、安装、跳过、通知、数据暴露、外部写入、扣费或后台任务。

涉及外部事实时，先定位本仓库使用方式：锁文件、依赖声明、封装层、调用点、配置、类型定义和测试。只有接口语义、版本差异或平台限制会影响方案时，再查官方文档、源码或协议。外部事实缺失会影响目标行为、边界、根因或推荐方案时，回流到最早受影响步骤；不影响推荐结论时，写入非阻塞待补证据。

## 回流规则

发现新资料、新代码事实、新反证、新未知分区、外部事实冲突、风险变化或用户目标变化时，回到最早受影响步骤，重建后续结论。不要只在最终回答补一句。

- 用户追加需求、例外、取消分支或改变目标：回到非结构化资料；只改变目标行为时回到目标合同。
- 新代码入口、状态、类型、权限、兼容路径、skip、no-op 或 fallback：回到代码事实；影响边界时继续回到语义边界。
- 推荐方案需要未授权分区、未知分区、反证分区或新副作用：回到方案；如果合同变化，回到目标合同。
- 外部事实或风险改变目标行为、边界或方案可行性：回到最早受影响步骤，并降低证据等级。

多个回流来源同时命中时，采用最早受影响步骤、最高风险、最低证据等级和最保守输出。

## 方案选择规则

- 先生成最小可行方案。它必须满足目标合同，改动面最小，并让反证分区、未知分区和保持原行为分区不受新行为影响。
- 推荐方案只能选择已授权目标行为的实现路径。方案不能新增目标行为、扩大适用分区、删除保持原行为分区或改变兼容承诺。
- 备选方案只保留会改变执行结果、风险、成本、发布时间、验证义务或长期维护的路径；纯风格差异不列为备选。
- 被拒方案必须写具体故障模式：越界分区、破坏合同字段、无法验证、回滚困难、兼容破坏、风险不可接受或维护成本失控。
- 取舍必须说明代价归属和爆发时机。用户需要承担产品范围、兼容破坏、发布窗口或成本变化时，输出用户裁决点。
- 推荐方案不能写成已实现、已验证或生产安全。
- 变更边界只能说明归属方、边界、状态归属、改动层级和实现约束；不要写按顺序执行的文件级任务清单。

## 阶段手册

- 方案阶段：写入 `.swe/<topic>/change-boundary.md` 前，读取 `references/change-boundary.md`。该手册包含方案写入前检查、产物规则和输出模板。
- 交接文件阶段：用户完成确认停靠点后，读取 `references/handoff.md`。该手册包含交接文件写入前检查、产物规则和输出模板。
- 阶段手册不可访问时，输出缺失路径和被阻止的产物，不临时重建模板。

## 禁止动作

- 不把资料假设写成代码事实。
- 不用一致性、完整覆盖或风险规避补足语义授权。
- 不把未知分区默认纳入目标行为。
- 不用相邻分区授权覆盖反证分区。
- 不把用户表象当根因，不把止血包装成根因修复，不把局部修复说成全局方案。
- 不用测试通过、同路径、同模块、归属方或框架最佳实践证明语义边界正确。
- 不因为第三方平台支持某能力，就授权本仓库所有组件、路由、状态或用户群使用新行为。
- 不把推荐方案写成已实现、已验证、可直接上线或生产安全。
- 不在方案阶段写实现步骤、文件级任务清单、测试矩阵或交付总结。
- 不在方案阶段创建 branch、worktree、提交、安装依赖或修改实现文件。
- 不在用户同意方案前生成交接文件。
- 不把交接文件内容混入方案文件。
- 不在未读代码时声称根因、系统现实或变更边界。
- 不让 subagent 写文件、安装依赖、格式化文件或提交 git。
