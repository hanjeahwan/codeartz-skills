# 目标边界执行手册

## 阶段职责

- 本手册负责资料分析、代码取证、语义分区、目标合同、方案检查和方案文件更新。
- 对话分析停在用户要求的最浅结论。
- 方案文件只写入 `.codeartz/<topic>/target-boundary.md`。
- 本阶段不进入实现。

## 执行规则

### 分析顺序

| 顺序 | 阶段     | 检查动作                                                                 | 阶段结果                                 |
| ---- | -------- | ------------------------------------------------------------------------ | ---------------------------------------- |
| 1    | 资料假设 | 提取用户目标、明确排除项、拟议行为、冲突点和待验证声明。                 | 可验证的资料假设。                       |
| 2    | 代码事实 | 定位入口、调用链、类型、状态、配置、测试、运行脚本、封装层和副作用归属。 | 当前系统行为及其反证。                   |
| 3    | 语义边界 | 枚举已知分区、同入口变体、授权状态和越界后果。                           | 允许工作面、禁止扩散面、未知和冲突。     |
| 4    | 系统现实 | 合并资料假设、代码事实和外部事实。                                       | 已证实、已推翻和仍未知的当前行为。       |
| 5    | 目标合同 | 定义当前行为、目标行为、保持不变和未知。                                 | 可验证的行为合同。                       |
| 6    | 方案     | 选择只覆盖已授权分区的最小落点。                                         | 推荐方案、边界、风险、裁决点和开工入口。 |

- 分析顺序不可跳级。
- 新证据改变前序结论时，回到最早受影响阶段。
- 用户只要求状态时，只输出当前阶段、已知事实、缺口和下一步。

### 来源分类

| 来源类型                                     | 可形成的内容 | 定位要求                         |
| -------------------------------------------- | ------------ | -------------------------------- |
| 用户原话、PRD、spec、issue、会话或旧方案     | 资料假设     | 原句、章节、issue 或资料位置。   |
| mockup、wireframe、UI/UX spec、截图或设计稿  | 资料假设     | 页面、画板、frame 或截图位置。   |
| 代码、测试、配置、迁移、类型或运行脚本       | 代码事实     | 文件、符号、测试、配置或命令。   |
| 官方文档、源码、协议、平台限制或固定版本依赖 | 外部事实     | 版本、页面、源码位置或协议章节。 |

- 当前代码边界仍由仓库入口、调用链、类型、配置和测试确认。

### 目标授权

- 目标授权检查
  - 检查项：每个目标行为是否符合 `SKILL.md`“全局边界”中的授权规则。
  - 失败条件：目标行为只由仓库现状、相邻分区、外部事实或最佳实践推导。
  - 处理方式：按“方案不确定性路由”处理。
  - 禁止：不把缺少目标授权的行为写入推荐方案。

### 语义分区

- 分区授权状态只使用：已授权、禁止、保持原行为、未知、冲突、不适用。
- 已授权表示证据明确允许目标行为作用于该分区。
- 禁止表示证据明确要求目标行为不得作用于该分区。
- 保持原行为表示该分区继续执行当前行为。
- 未知表示现有证据不足以判断授权状态。
- 冲突表示资料、代码、外部事实或用户目标互相矛盾。
- 不适用表示证据已经证明该分区不属于当前目标。
- 同入口变体包括同一路由、页面、模块、命令、任务或事件下的类型、状态、权限、环境、生命周期、来源、失败上下文和副作用分支。
- 反证分区包括明确不适用、无需参与、不得改变、保持原行为、跳过、例外或直接通过的分区。

### 目标影响面

- 前端/UI：页面、组件、视觉对象、交互、状态、设备、a11y、i18n、设计系统。
- 后端/API：接口、handler、service、schema、错误码、兼容合同。
- 数据：表结构、迁移、索引、旧数据、读写路径、事务。
- 权限/安全：身份、角色、租户、访问控制、隐私、审计。
- 异步任务：队列、cron、worker、重试、幂等、补偿。
- 第三方集成：SDK、webhook、外部 API、限流、版本约束。
- 配置/运行环境：env、feature flag、部署环境、区域差异。
- CI/CD/构建部署：build、test、release、rollback、artifact。
- 系统边界：模块、共享层、服务、仓库、归属方、共享合同、版本和部署依赖。
- 变更演进：一次切换、灰度、双轨、迁移、回填和回滚。
- 运行约束：SLO、容量、成本、日志、指标、监控、合规和数据驻留。
- 只检查被用户目标、资料假设、代码事实或风险命中的影响面。
- 每个命中影响面只记录是否涉及、证据、目标行为和边界处理。
- 无法判断的影响面写入未知或待补证据。

### 复杂变更

- 只检查被目标行为、代码事实、外部事实或风险命中的复杂变更维度。

- 变更拓扑检查
  - 检查项：目标行为是否跨越模块、共享层、服务、仓库或归属方。
  - 检查项：每个命中边界是否存在共享合同、版本差异或部署依赖。
  - 放行条件：每个命中边界都有可定位证据和边界处理。
  - 失败条件：跨边界关系无法确认。
  - 处理方式：按“方案不确定性路由”处理。

- 演进方式检查
  - 检查项：目标行为是否需要一次切换、灰度、双轨、迁移、回填或回滚。
  - 放行条件：每个命中方式都写明触发条件、影响分区、兼容边界和退出条件。
  - 失败条件：数据、公开合同、共享状态或跨服务版本发生变化，但没有演进方式。
  - 处理方式：回到目标合同阶段。

- 运行约束检查
  - 检查项：目标行为是否命中 SLO、容量、成本、合规或数据驻留约束。
  - 放行条件：每个命中约束都写明来源、影响分区和方案限制。
  - 失败条件：运行约束会改变目标合同或推荐方案，但现有证据不足。
  - 处理方式：按“方案不确定性路由”处理。

### 证据获取

- 从用户点名的资料、路径、入口或对象开始。
- 搜索入口、合同、状态、数据、副作用、反证和同入口变体。
- 外部能力影响方案时，先检查仓库固定版本、封装层、调用点、配置、类型和测试。
- 仓库证据仍不能确认外部约束时，再查官方文档、源码或协议。
- 工作过程中记录搜索词、已查路径、未查路径、命中、未命中和停止原因。
- 方案文件只保留会改变结论的关键证据。

- 授权分区证据检查
  - 检查项：每个已授权分区是否记录支持证据。
  - 检查项：支持证据是否把目标行为绑定到该分区。
  - 检查项：每个已授权分区是否记录反证搜索结果。
  - 失败条件：任一已授权分区缺少支持证据或反证搜索结果。
  - 处理方式：继续补证据。

- 语义边界证据检查
  - 禁止：不把测试通过当作语义边界的证明。
  - 禁止：不把同一路径、模块或归属方当作语义边界的证明。
  - 禁止：不把框架最佳实践当作语义边界的证明。

- 证据停止检查
  - 放行条件：现有证据已支撑当前结论，新增证据只会重复已有判断。
  - 暂停条件：搜索空间过大，需要先确认下一批范围。
  - 降级条件：仓库、依赖或外部来源不可访问。
  - 继续条件：只有正向证据，尚未搜索反证。
  - 继续条件：尚未确认状态、合同、副作用、测试、配置或现有行为约束。
  - 继续条件：代码与资料冲突尚未处理。
  - 继续条件：高风险分区尚未检查回滚、观测、发布或补偿影响。

### 风险和外部事实

- 检查权限、数据完整性、副作用、异步一致性、兼容性、运行环境、性能、安全、隐私、可访问性、国际化和可观测性风险。
- 每项风险写明影响分区、爆发时机、现实后果、代价归属和方案限制。
- 外部事实影响目标行为、边界或方案可行性时，回到最早受影响阶段。
- 外部事实不影响推荐结论时，写入待补证据。

### 方案选择

- 备选方案检查
  - 保留条件：备选方案会改变结果、风险、成本、发布、验证或维护方式。
  - 无内容处理：没有实质性备选方案时写“不适用”。
  - 禁止：不为填充模板编造备选方案。

- 被拒方案检查
  - 检查项：每个被拒方案是否写明具体故障模式。
  - 检查项：每项取舍是否写明代价归属和爆发时机。
  - 无内容处理：没有被拒方案时写“不适用”。

- 推荐方案检查
  - 禁止：推荐方案不新增未授权行为。
  - 禁止：推荐方案不扩大已授权分区。
  - 禁止：推荐方案不删除保持原行为的分区。
  - 禁止：推荐方案不改变未经授权的兼容承诺。

### 方案不确定性路由

- 不确定性检查
  - 检查项：证据、分区、范围、行为或取舍的不确定性是否改变目标合同或推荐方案。
  - 阻塞条件：不确定性会改变目标行为、适用分区、兼容约束或推荐方案。
  - 阻塞处理：停止输出推荐方案，回到最早受影响阶段。
  - 裁决条件：不确定性属于用户范围、行为或取舍。
  - 裁决处理：写入待裁决。
  - 待补条件：证据缺口不影响当前推荐结论。
  - 待补处理：写入待补证据。

### 写入前检查

- 全局边界检查
  - 检查项：结论是否符合 `SKILL.md` 的全部全局边界。
  - 失败处理：回到最早受影响阶段。

- 依据和合同检查
  - 检查项：方案是否区分资料假设、代码事实、外部事实和系统现实。
  - 检查项：目标合同是否覆盖目标行为、输入、输出、状态变化、副作用、失败行为、用户可见变化和兼容约束。
  - 放行条件：每项结论都能追溯到对应证据和目标合同字段。

- 复杂变更检查
  - 检查项：方案是否覆盖全部命中的变更拓扑、演进方式和运行约束。
  - 放行条件：每个命中维度都有可定位证据、边界处理和方案限制。
  - 无内容处理：没有命中复杂变更维度时写“不适用”。

- 目标影响面检查
  - 检查项：推荐方案是否覆盖全部命中影响面。
  - 放行条件：每个命中影响面都有可定位证据、目标行为和边界处理。
  - 失败条件：命中影响面缺少代码、配置、测试、脚本或文档证据。
  - 处理方式：先补证据。

- 推荐结论范围检查
  - 检查项：推荐结论是否只覆盖已授权分区。
  - 放行条件：未知、冲突、禁止和不适用分区没有被带入目标行为。

- 分区覆盖检查
  - 检查项：分区授权矩阵是否覆盖全部已知分区。
  - 放行条件：反证、未知、冲突和保持原行为分区均已列出。
  - 禁止：不只列被推荐方案修改的分区。

- 同入口变体检查
  - 检查项：入口级目标行为是否已枚举同入口变体。
  - 失败条件：存在未授权变体。
  - 处理方式：写入未知、冲突或待裁决。
  - 禁止：不写成普通风险。

- 粗粒度命中检查
  - 检查项：方案是否命中共享入口、共享层、状态分支、类型分支、事件、拦截器、配置或公共抽象。
  - 放行条件：命中面只覆盖已授权分区。
  - 失败条件：命中面会覆盖未知、禁止、不适用、保持原行为或冲突分区。
  - 处理方式：先收窄命中条件。
  - 裁决条件：不能收窄时写入待裁决。

- 风险检查
  - 检查项：已授权方案内仍可能爆发的问题是否写清代价归属、爆发时机和现实后果。

- 待裁决检查
  - 检查项：范围、行为或取舍问题是否已列出。
  - 检查项：每个待裁决项是否写清现实后果。
  - 无内容处理：写“不适用”。

- 证据收尾检查
  - 检查项：证据停止点是否写清停止原因。
  - 检查项：待补证据是否只包含不阻塞方案结论的代码事实或外部事实。
  - 检查项：不能声称是否覆盖未实现、未验证、未测试或未检查的内容。
  - 无内容处理：没有内容时写“不适用”。

- 开工入口检查
  - 检查项：先读是否引用证据索引中的关键证据编号和定位。
  - 检查项：先复核是否列出开工前必须重新确认的事实。
  - 检查项：先补是否列出实现前必须补齐的证据。
  - 检查项：过期触发是否覆盖方案、代码、分支、依赖、配置、数据或设计资料变化。
  - 禁止：不写文件级任务清单。

- 方案写入关卡
  - 放行条件：全部写入前检查通过。
  - 失败处理：不写推荐方案文件。
  - 失败输出：只输出事实、缺口、限制、待裁决点和下一步证据动作。

## 产物规则

### 方案文件

- 用户给出路径时，优先使用用户路径。
- 用户未给路径时，从用户点名的功能、issue、变更、模块、路由或主要对象生成 `<topic>`。
- `<topic>` 使用小写 kebab-case。
- 主题无法判断时，使用 `.codeartz/target-boundary/target-boundary.md`。
- 方案通过写入关卡后，写入 `.codeartz/<topic>/target-boundary.md`。

- 已有文件处理
  - 检查项：`.codeartz/<topic>/target-boundary.md` 是否存在。
  - 缺失处理：创建 `.codeartz/<topic>/` 目录和方案文件。
  - 已有处理：先读取现有方案，再决定更新或另存。
  - 禁止：不静默覆盖已有方案。

- 对话降级
  - 触发条件：项目不可写或用户只要求对话回答。
  - 处理方式：只在对话输出。
  - 输出要求：说明没有写入项目文件的原因。

- 无代码降级
  - 触发条件：没有仓库或代码不可访问。
  - 处理方式：只输出资料假设、初始分区、候选代码路径和补证据动作。
  - 禁止：不写推荐方案文件。

- 写入失败处理
  - 触发条件：方案写入失败。
  - 输出要求：只说明失败原因或等待用户确认路径。
  - 禁止：不把本轮说成已产出项目方案。

- 方案状态更新
  - 待裁决条件：方案仍有待裁决点。
  - 待裁决状态：写“待裁决”。
  - 可采纳条件：方案关卡通过，但用户未明确同意采用当前方案。
  - 可采纳状态：写“可采纳”。
  - 已采纳条件：待裁决点已处理或明确无待裁决点，风险已展示，用户明确同意方案或要求按当前方案进入实现。
  - 已采纳状态：写“已采纳”。
  - 禁止：不因用户催促、询问流程或询问是否可以开始而写“已采纳”。

- 对话回复只包含方案路径、推荐结论和待裁决摘要。
- 完整方案以项目文件为准。
- 方案文件只写方案、关键证据和开工入口。
- 禁止把完整探索链路写入方案文件。
- 禁止把已查路径和未查路径写入方案文件。
- 禁止把方案写入未被用户点名的 ADR、issue、PR description、README 或实现文件。

## 输出要求

### 模板使用规则

- 写入方案文件时，使用“方案输出模板”。
- 允许为空的字段写“不适用”。
- 未授权或不在范围内的内容写入未知、冲突或待裁决。

### 方案输出模板

```markdown
## 方案

### 推荐结论

- 方案状态：[待裁决 / 可采纳 / 已采纳]
- 推荐方案：[一句话说明推荐路径和目标边界。]
- 待裁决摘要：[待裁决项数量和一句话摘要；没有则写“不适用”。]

### 待裁决

| 编号 | 待裁决问题                             | 需要用户裁决的内容   | 不裁决会阻塞的结论               | 现实后果                           |
| ---- | -------------------------------------- | -------------------- | -------------------------------- | ---------------------------------- |
| D1   | [未授权同入口变体、范围、行为或取舍。] | [用户需要选择什么。] | [无法确定的目标合同或方案边界。] | [触发流程、影响对象和可观察后果。] |

[没有待裁决项时写“不适用”。]

### 目标合同

| 合同项       | 当前行为                                             | 目标行为                         | 保持不变或未知     |
| ------------ | ---------------------------------------------------- | -------------------------------- | ------------------ |
| 目标行为     | [当前行为。]                                         | [新增、修改、删除或保持的行为。] | [保持不变或未知。] |
| 输入         | [当前输入。]                                         | [目标输入。]                     | [保持不变或未知。] |
| 输出         | [当前输出。]                                         | [目标输出。]                     | [保持不变或未知。] |
```
