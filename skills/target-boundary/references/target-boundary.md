# 方案阶段手册

## 阶段职责

方案阶段负责生成 `.codeartz/<topic>/target-boundary.md`。

生成前必须把以下内容收敛成方案：

- 资料假设。
- 代码事实。
- 语义边界。
- 目标合同。

本阶段不生成交接文件，不进入实现。

## 写入前检查

### 方案写入前自检

写入 `.codeartz/<topic>/target-boundary.md` 前，逐项确认：

#### 证据和合同

- 依据检查：
  - 区分资料假设、代码事实、外部事实、系统现实和根因判断。
  - 不把用户表象写成系统事实。
- 目标合同检查：
  - 覆盖输入输出、状态变化、副作用、失败行为、用户可见变化和兼容约束。

#### 目标影响面

- 目标影响面检查：
  - 检查项：推荐方案是否命中 `SKILL.md` 定义的目标影响面。
  - 放行条件：每个命中影响面都有可定位证据、目标行为判断和边界处理。
  - 失败条件：命中影响面缺少代码、配置、测试、脚本或文档证据。
  - 失败条件：把未检查影响面默认为不涉及。
  - 处理方式：先补证据。
  - 回流条件：无法补证据且影响推荐方案时，回到代码事实、语义边界或目标合同。
  - 裁决条件：无法补证据且涉及用户范围、行为或取舍时，写入待裁决。
  - 待补条件：不影响推荐方案时，写入待补证据。
  - 禁止：不因为影响面属于同一项目、同一模块、同一入口、同一团队或同一发布链路，就默认授权目标行为。

#### 分区和授权

- 推荐结论范围：
  - 只覆盖已授权分区。
  - 未知、冲突、禁止和不适用分区没有被带入目标行为。
- 不适用范围：
  - 列出反证分区、未知分区、冲突分区和保持原行为分区。
- 分区授权覆盖：
  - 覆盖每个已知宽范围对象下的已知子分区。
  - 不能只列被推荐方案修改的分区。
- 同入口变体：
  - 路由、页面、模块或入口级目标行为已经枚举同入口变体。
  - 未授权变体写入未知、冲突或待裁决。
  - 不写成普通风险。
- 上层目标覆盖变体：
  - 同时满足：被上层目标覆盖但没有明确排除。
  - 同时满足：缺少目标行为依赖机制。
  - 处理方式：必须进入待裁决。
  - 禁止处理：不写成不适用、保持原行为或普通风险。
- 粗粒度命中检查：
  - 命中面：入口、共享层、状态分支、类型分支、事件、拦截器、配置或公共抽象。
  - 放行条件：命中面只覆盖已授权分区。
  - 失败条件：命中面会覆盖未知、禁止、不适用、保持原行为或冲突分区。
  - 失败处理：先收窄命中条件。
  - 裁决条件：不能收窄时写入待裁决，不写成风险或不适用。

#### 目标边界

- 目标边界：
  - 只写边界、归属、约束和取舍。
  - 不写文件级任务清单、测试矩阵或交付总结。

#### 风险和裁决

- 风险：
  - 写清已授权方案内仍可能爆发的问题、代价归属、爆发时机和现实场景。
  - 未授权分区不写成风险。
- 待裁决：
  - 列出未授权同入口变体、范围、行为或取舍。
  - 写清需要用户裁决的现实场景。
  - 没有则写“不适用”。

#### 证据收尾

- 证据停止点：
  - 写清因重复、范围过大、不可访问或不改变结论而停止的证据。
  - 没有则写“不适用”。
- 待补证据：
  - 只写不阻塞方案结论的代码事实或外部事实。
  - 实现、验证或发布前仍需补。
  - 没有则写“不适用”。
- 不能声称：
  - 写清证据等级限制、未实现、未验证、未测试或未覆盖的内容。

任一项失败时：

- 不写推荐方案。
- 输出合同缺口、证据缺口或需要回流的步骤。

## 产物规则

### 方案产物

方案通过关卡后，默认写入当前项目的 `.codeartz/<topic>/target-boundary.md`。

- 用户给出路径时，优先使用用户路径。
- 用户未给路径时：
  - 从用户点名的功能、issue、故障、模块、路由或主要对象生成 `<topic>`。
  - 使用小写 kebab-case。
- 主题无法判断时：
  - 使用 `.codeartz/target-boundary/target-boundary.md`。
  - 不要为了命名停止推进。
- 写入方案文件前：
  - 证据等级必须达到 L3。
  - L0/L1/L2 输出：只能在对话里输出资料假设、代码事实、候选系统现实、合同缺口或补证据动作。
  - 不写 `.codeartz/<topic>/target-boundary.md`。
- 写入前处理目录和已有文件：
  - 创建 `.codeartz/<topic>/` 目录。
  - 已有同名文件时，先读取现有内容。
  - 再决定覆盖、更新或另存。
  - 不要静默丢失已有方案。
- 对话回复只给：
  - 方案文件路径。
  - 推荐结论。
  - 证据等级。
  - 待裁决点摘要。
  - 完整方案以项目文件为准。
- 项目不可写或用户只要求对话回答时：
  - 降级为对话输出。
  - 说明没有写入 `.codeartz/` 的原因。
- 没有仓库或代码不可访问时：
  - 按降级输出处理。
  - 不要把资料层候选方案写成 `.codeartz/<topic>/target-boundary.md`。
- 方案写入失败时：
  - 不把本轮说成已产出项目方案。
  - 只说明失败原因或等待用户确认路径。
- 不要把方案写入 ADR、issue、PR description、README 或实现文件，除非用户明确点名。
- 方案文件只写方案内容。
- 不要把以下内容塞进方案文件：
  - 交接文件。
  - 完整探索链路。
  - 已查路径和未查路径。

## 输出模板

### 方案输出模板

#### 使用规则

- 写入 `.codeartz/<topic>/target-boundary.md` 时，使用此模板。
- 只在方案关卡通过且证据等级达到 L3 时写入项目文件。
- 关卡失败时：
  - 不使用此模板写文件。
  - 只在对话里输出缺口、被阻止的结论和下一步证据动作。
- 字段信息不足时：
  - 不填充猜测。
  - 写明缺口、未知或非阻塞待补证据。
- 方案文件只写方案内容。
- 不要写入：
  - 交接文件内容。
  - 完整探索链路。
  - 已查路径或未查路径。
- 没有会改变执行结果的被拒方案时：
  - 写“不适用”。
  - 不要为了填模板补造备选。

```markdown
## 方案

### 推荐结论

[一句话说明推荐怎么做。]

### 范围和授权

- 适用分区：[哪些分区、入口、模块、对象、状态或用户路径应用该方案。]
- 不适用分区：[哪些反证、未知、冲突、不适用或保持原行为分区不应用该方案。]
- 覆盖声明：[是否覆盖所有已知分区；未覆盖时写原因和处理方式。]
- 授权明细：[分区] - [已授权 / 禁止 / 保持原行为 / 未知 / 冲突 / 不适用] - [证据和处理方式。]
- 反证和未知：[为什么不能把目标行为套用到这些分区。]
  现实场景：[误套目标行为时，真实用户流程或系统流程会怎样出错。]

### 目标影响面

- 填写规则：按 `SKILL.md` 的目标影响面清单逐项填写。
- 禁止：不新增、不删减影响面。
- [目标影响面]：[涉及 / 不涉及 / 未知]
  证据：[涉及时写可定位证据；不涉及时写理由；未知时写缺口。]
  目标行为：[该影响面是否需要新行为；没有则写“不适用”。]
  边界处理：[允许改 / 保持原行为 / 待裁决 / 待补证据 / 不适用。]

### 依据

- 资料假设：[来自需求、PRD、issue、会话或旧方案的关键假设。]
- 代码事实：[来自入口、调用链、类型、状态、测试、配置或副作用归属的事实。]
- 外部事实：[来自官方文档、源码、协议、平台限制或固定版本依赖的约束；没有则写“不适用”。]
- 系统现实：[资料和代码合并后，当前系统真实表现。]
- 根因判断：[如果是故障、事故或异常任务，写根因；否则写“不适用”。]
- 证据等级：[L3，以及不能声称的内容。]

### 目标合同

- 目标行为：[新增、修改、删除或保持什么行为。]
- 输入输出：[输入、输出、状态变化。]
- 副作用：[写数据、发请求、发通知、缓存、权限、异步任务等影响。]
- 失败行为：[失败、跳过、fallback、直接通过或报错方式。]
- 兼容约束：[旧路径、旧数据、旧 API、旧 UI 或旧行为如何保持。]

### 目标边界

- [建议触达的归属方、边界、状态归属或抽象层。]
- [为什么该边界优先于相邻路径。]
- [如何避免把行为扩散到未授权分区。]
- [方案需要保留的实现约束、验证前置或发布/回滚/观测前置。]

### 被拒方案

- [方案名或“不适用”]：[被拒方案或不适用原因。]
  故障模式：[会误改什么、漏掉什么、破坏什么边界。]
  现实场景：[采用该方案后，用户、系统或团队会在什么流程里遇到什么问题。]

### 风险和待裁决

- 风险：
  - [风险项或“不适用”]：[已授权方案内仍可能爆发的问题；不要把未授权分区写成风险。]
    现实场景：[用户、系统或运维在什么操作、状态或时间点会遇到什么后果。]
- 待裁决：
  - [待裁决项或“不适用”]：[未授权同入口变体、范围、行为或取舍；被上层目标覆盖但没有明确排除、且缺少目标行为依赖机制的变体。]
    现实场景：[如果不裁决，用户、产品、实现或验证会在哪个真实流程里卡住或产生歧义。]
- 证据停止点：[哪些证据因为重复、范围过大、不可访问或不改变结论而停止；没有则写“不适用”。]
- 待补证据：[不阻塞本方案结论，但实现、验证或发布前仍需补的代码事实或外部事实；没有则写“不适用”。]
  现实场景：[缺少该证据时，后续实现、测试或发布会在哪一步无法确认。]
- 不能声称：[由于证据等级或关卡限制，本轮不能声称的结论。]
```
