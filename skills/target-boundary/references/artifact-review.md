# 已有目标边界合同复核手册

## 目标

- 判断已有 `boundary.md` 是否仍是可执行的目标边界约束。
- 判断后续 agent 是否可以依据合同进入实现设计或实现。
- 本阶段不更新目标边界合同。
- 本阶段不重新生成边界结论。

## 执行规则

### 合同定位

- 只复核 `.codeartz/<topic>/boundary.md`。
- 同一 `<topic>` 只允许一个有效合同。
- 找不到合同时，报告缺失路径。
- 存在多个有效合同候选时，停止开工判断。
- 多合同冲突需要用户裁定唯一有效合同。

### 合同完整性

- 字段适用性检查
  - 检查项：必需字段是否存在、非空、没有模板占位符并满足枚举合同。
  - 检查项：条件字段是否在适用时填写，不适用时只写“不适用”。
  - 失败条件：以“不适用”掩盖未知、冲突、未授权或待裁决。
  - 处理方式：停止开工判断，并报告字段路径和失败原因。

- 必需章节检查
  - 检查项：是否包含边界结论、证据基线、待裁决、目标合同、分区授权、目标边界、备选与被拒边界、风险、依据与系统现实、开工入口。
  - 放行条件：全部必需章节存在。
  - 失败条件：任一必需章节缺失。
  - 处理方式：停止开工判断，并要求先更新合同。

- 目标合同字段检查
  - 检查项：目标合同是否包含目标行为、输入、输出、状态变化、副作用、失败行为、用户可见变化和兼容约束。
  - 放行条件：八个合同项均已填写。
  - 失败条件：任一合同项缺失、留空或保留模板占位符。
  - 处理方式：停止开工判断，并列出缺失合同项。

- 分区授权检查
  - 检查项：分区授权矩阵是否覆盖全部已知分区。
  - 检查项：每个分区是否包含授权状态、目标行为、证据、处理方式和越界后果。
  - 放行条件：分区授权矩阵完整，且授权状态只使用 `SKILL.md` 规定的值。
  - 失败条件：已知分区缺失、字段缺失或授权状态无效。
  - 处理方式：停止开工判断，并列出缺失或无效分区。

- 结构内容检查
  - 检查项：关键结论、证据索引和开工入口是否存在可执行内容。
  - 放行条件：适用字段已有内容，不适用字段明确写“不适用”。
  - 失败条件：适用字段留空、保留模板占位符或以普通风险替代未授权分区。
  - 处理方式：停止开工判断，并列出损坏内容。

### 依赖有效性

- 基线完整性检查
  - 检查项：合同是否记录仓库、分支、HEAD、工作区状态、相关依赖和分析时间。
  - 放行条件：适用字段均有值。
  - 失败条件：适用字段缺失、为空、写成“未知”或无法解释。
  - 处理方式：判定合同失效，并报告缺失字段。

对每类合同依赖执行同一个有效性门：

| 依赖类型 | 合同记录                                             | 当前复核                         |
| -------- | ---------------------------------------------------- | -------------------------------- |
| 用户授权 | 合同采用的范围、行为和取舍                           | 合同生成后是否出现更高优先级决定 |
| 仓库基线 | 仓库、分支、HEAD、工作区状态                         | 当前值及差异                     |
| 相关依赖 | 锁文件、版本或指纹                                   | 当前值及外部约束                 |
| 关键证据 | “证据索引”和“先读”的定位与语义                       | 对象是否可定位且仍支持对应结论   |
| 过期触发 | 用户决定、需求、代码、依赖、配置、数据或设计资料变化 | 是否命中合同记录的触发条件       |

- 定位合同记录和当前对象；缺失、不可读或身份变化时继续检查其语义影响。
- 变化不会自动导致失效；只有直接证据证明变化不影响当前行为、分区边界、目标授权或关键结论时才放行。
- 变化可能影响任一对应结论时，判定合同失效，并列出变化、受影响结论和更新动作。
- 关键证据无法定位或语义已改变时，判定合同失效，不猜测旧内容。

### 合同状态

- 状态检查
  - 检查项：合同状态是否为“已采纳”。
  - 放行条件：合同状态为“已采纳”。
  - 失败条件：合同状态缺失、保留占位符，或不是“已采纳”。
  - 处理方式：停止开工判断，并报告仍需完成的状态条件。

- 待裁决检查
  - 检查项：合同是否仍有未处理的待裁决项。
  - 放行条件：待裁决为“不适用”。
  - 失败条件：仍有待裁决项，或状态与待裁决内容矛盾。
  - 处理方式：停止开工判断，并列出待裁决项。

### 开工入口

- 先复核检查
  - 检查项：“先复核”中的系统事实、边界和外部约束是否已经重新确认。
  - 放行条件：全部复核通过。
  - 失败条件：存在未复核项或复核失败项。
  - 处理方式：停止开工判断，并列出未通过项。

- 先补检查
  - 检查项：“先补”是否为“不适用”。
  - 放行条件：没有实现前必须补齐的证据。
  - 失败条件：仍有必须补齐的代码事实或外部事实。
  - 处理方式：停止开工判断，并列出待补内容。

### 开工裁定

- 开工放行
  - 放行条件：合同完整性、依赖有效性、合同状态和开工入口检查全部通过。
  - 处理方式：说明可以进入实现设计或实现。
  - 禁止：不把开工放行说成已经实现、验证或可上线。

- 开工拒绝
  - 失败条件：任一检查未通过。
  - 处理方式：说明不能依据当前合同开工。
  - 处理方式：列出失效证据、待补内容或待裁决项。
  - 禁止：不在本阶段静默修订合同。

## 输出要求

- 输出开工结论。
- 输出合同路径与合同状态。
- 输出证据基线比较结果。
- 输出最先复核的关键证据编号和定位。
- 未放行时输出阻塞原因和下一步动作。
- 不输出完整探索链路。
