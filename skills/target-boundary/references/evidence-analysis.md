# 证据与边界分析手册

## 目标

- 把资料假设、仓库事实、外部事实和目标授权分开。
- 枚举语义分区、同入口变体、反证、未知和冲突。
- 形成可追溯的目标合同与最小目标边界。
- 对话分析停在用户要求的最浅结论。
- 本阶段不创建目标边界合同文件。

## 执行规则

### 分析顺序

| 顺序 | 阶段     | 检查动作                                                                 | 阶段结果                             |
| ---- | -------- | ------------------------------------------------------------------------ | ------------------------------------ |
| 1    | 资料假设 | 提取用户目标、明确排除项、拟议行为、冲突点和待验证声明。                 | 可验证的资料假设。                   |
| 2    | 代码事实 | 定位入口、调用链、类型、状态、配置、测试、运行脚本、封装层和副作用归属。 | 当前系统行为及其反证。               |
| 3    | 语义边界 | 枚举已知分区、同入口变体、授权状态和越界后果。                           | 允许工作面、禁止扩散面、未知和冲突。 |
| 4    | 系统现实 | 合并资料假设、代码事实和外部事实。                                       | 已证实、已推翻和仍未知的当前行为。   |
| 5    | 目标合同 | 定义当前行为、目标行为、保持不变和未知。                                 | 可验证的行为合同。                   |
| 6    | 边界结论 | 选择只覆盖已授权分区的最小落点。                                         | 推荐结论、边界、风险和裁决点。       |

- 分析顺序不可跳级。
- 新证据改变前序结论时，回到最早受影响阶段。
- 用户只要求状态时，只输出当前阶段、已知事实、缺口和下一步。

### 来源分类

每条陈述先标记为“当前行为”“目标行为”或“限制与冲突”，再按 `SKILL.md` 的“来源能力”裁定。定位要求如下：

| 来源类型                                     | 定位要求                       |
| -------------------------------------------- | ------------------------------ |
| 用户原话、PRD、spec、issue、会话或旧方案     | 原句、章节、issue 或资料位置   |
| mockup、wireframe、UI/UX spec、截图或设计稿  | 页面、画板、frame 或截图位置   |
| 代码、测试、配置、迁移、类型或运行脚本       | 文件、符号、测试、配置或命令   |
| 官方文档、源码、协议、平台限制或固定版本依赖 | 版本、页面、源码位置或协议章节 |

- 当前代码边界由仓库入口、调用链、类型、配置和测试确认。
- 资料对当前行为的陈述被代码事实推翻时，标记为“已推翻”，并保留对应代码事实和差异。

### 目标授权

- 产物动作与目标行为分离检查
  - 检查项：当前用户是否直接陈述目标行为，或明确采用某份资料中的目标行为。
  - 失败条件：用户只要求读取、分析、创建或更新合同，目标行为仅存在于未确认资料中。
  - 处理方式：把目标行为授权写入待裁决，并停止推荐结论和合同写入。
  - 禁止：不把“根据资料生成合同”解释成“批准资料中的产品行为”。

- 目标授权检查
  - 检查项：每个目标行为是否符合 `SKILL.md`“全局边界”中的授权规则。
  - 失败条件：目标行为只由 `SKILL.md`“不形成目标授权的来源”推导。
  - 处理方式：按“边界不确定性路由”处理。
  - 禁止：不把缺少目标授权的行为写入推荐结论。

### 语义分区

- 按 `SKILL.md` 的“分区分类”裁定每个已知分区。
- 同入口变体包括同一路由、页面、模块、命令、任务或事件下的类型、状态、权限、环境、生命周期、来源、失败上下文和副作用分支。
- 反证分区包括明确不适用、无需参与、不得改变、保持原行为、跳过、例外或直接通过的分区。

- 已授权状态检查
  - 检查项：目标授权证据是否明确允许目标行为作用于该分区。
  - 检查项：仓库证据是否确认该分区身份。
  - 放行条件：目标授权和分区身份检查均通过。
  - 处理方式：把该分区标记为“已授权”。
  - 失败处理：按现有证据标记为“未知”或“冲突”。

### 目标影响面

- 前端/UI：页面、组件、视觉对象、交互、状态、设备、a11y、i18n、设计系统。
- 后端/API：接口、handler、service、schema、错误码、兼容合同。
- 数据：表结构、迁移、索引、旧数据、读写路径、事务。
- 权限/安全：身份、角色、租户、访问控制、隐私、审计。
- 异步任务：队列、cron、worker、重试、幂等、补偿。
- 第三方集成：SDK、webhook、外部 API、限流、版本约束。
- 配置/运行环境：env、feature flag、部署环境、区域差异。
- CI/CD/构建部署：build、test、release、rollback、artifact。
- 系统边界：模块、共享层、服务、仓库、归属方、共享合同、版本和部署依赖。
- 变更演进：一次切换、灰度、双轨、迁移、回填和回滚。
- 运行约束：SLO、容量、成本、日志、指标、监控、合规和数据驻留。
- 只检查被用户目标、资料假设、代码事实或风险命中的影响面。
- 每个命中影响面只记录是否涉及、证据、目标行为和边界处理。
- 无法判断的影响面写入未知或待补证据。

### 复杂变更

- 只检查被目标行为、代码事实、外部事实或风险命中的复杂变更维度。

- 变更拓扑检查
  - 检查项：目标行为是否跨越模块、共享层、服务、仓库或归属方。
  - 检查项：每个命中边界是否存在共享合同、版本差异或部署依赖。
  - 放行条件：每个命中边界都有可定位证据和边界处理。
  - 失败条件：跨边界关系无法确认。
  - 处理方式：按“边界不确定性路由”处理。

- 演进方式检查
  - 检查项：目标行为是否需要一次切换、灰度、双轨、迁移、回填或回滚。
  - 放行条件：每个命中方式都写明触发条件、影响分区、兼容边界和退出条件。
  - 失败条件：数据、公开合同、共享状态或跨服务版本发生变化，但没有演进方式。
  - 处理方式：回到目标合同阶段。

- 运行约束检查
  - 检查项：目标行为是否命中 SLO、容量、成本、合规或数据驻留约束。
  - 放行条件：每个命中约束都写明来源、影响分区和边界限制。
  - 失败条件：运行约束会改变目标合同或推荐结论，但现有证据不足。
  - 处理方式：按“边界不确定性路由”处理。

### 证据获取

- 从用户点名的资料、路径、入口或对象开始。
- 搜索入口、合同、状态、数据、副作用、反证和同入口变体。
- 外部能力影响结论时，先检查仓库固定版本、封装层、调用点、配置、类型和测试。
- 仓库证据仍不能确认外部约束时，再查官方文档、源码或协议。
- 工作过程中记录搜索词、已查路径、未查路径、命中、未命中和停止原因。
- 目标边界合同只保留会改变结论的关键证据。

- 授权分区证据检查
  - 检查项：每个已授权分区是否记录支持证据。
  - 检查项：支持证据是否把目标行为绑定到该分区。
  - 检查项：每个已授权分区是否记录反证搜索结果。
  - 失败条件：任一已授权分区缺少支持证据或反证搜索结果。
  - 处理方式：继续补证据。

- 语义边界证据检查
  - 禁止：不把测试通过当作语义边界的证明。
  - 禁止：不把同一路径、模块或归属方当作语义边界的证明。
  - 禁止：不把框架最佳实践当作语义边界的证明。

- 证据停止检查
  - 放行条件：现有证据已支撑当前结论，新增证据只会重复已有判断。
  - 暂停条件：搜索空间过大，需要先确认下一批范围。
  - 降级条件：仓库、依赖或外部来源不可访问。
  - 继续条件：只有正向证据，尚未搜索反证。
  - 继续条件：尚未确认状态、合同、副作用、测试、配置或现有行为约束。
  - 继续条件：代码与资料冲突尚未处理。
  - 继续条件：高风险分区尚未检查回滚、观测、发布或补偿影响。

### 风险和外部事实

- 检查权限、数据完整性、副作用、异步一致性、兼容性、运行环境、性能、安全、隐私、可访问性、国际化和可观测性风险。
- 每项风险写明影响分区、爆发时机、现实后果、代价归属和边界限制。
- 外部事实影响目标行为、边界或可行性时，回到最早受影响阶段。
- 外部事实不影响推荐结论时，写入待补证据。

### 边界选择

- 备选边界检查
  - 保留条件：备选边界会改变结果、风险、成本、发布、验证或维护方式。
  - 无内容处理：没有实质性备选边界时写“不适用”。
  - 禁止：不为填充产物编造备选边界。

- 被拒边界检查
  - 检查项：每个被拒边界是否写明具体故障模式。
  - 检查项：每项取舍是否写明代价归属和爆发时机。
  - 无内容处理：没有被拒边界时写“不适用”。

- 推荐结论检查
  - 禁止：推荐结论不新增未授权行为。
  - 禁止：推荐结论不扩大已授权分区。
  - 禁止：推荐结论不删除保持原行为的分区。
  - 禁止：推荐结论不改变未经授权的兼容承诺。

### 边界不确定性路由

- 不确定性检查
  - 检查项：证据、分区、范围、行为或取舍的不确定性是否改变目标合同或推荐结论。
  - 阻塞条件：不确定性会改变目标行为、适用分区、兼容约束或推荐结论。
  - 阻塞处理：停止输出推荐结论，回到最早受影响阶段。
  - 裁决条件：不确定性属于用户范围、行为或取舍。
  - 裁决处理：写入待裁决。
  - 待补条件：证据缺口不影响当前推荐结论。
  - 待补处理：写入待补证据。

## 输出要求

- 只输出用户要求的最浅结论。
- 输出区分资料假设、代码事实、外部事实、系统现实和目标授权。
- 代码证据不足且阻止推荐结论时，只输出证据缺口、合同缺口和下一步最小证据动作。
- 禁止用方案骨架、架构方向、实现原则或候选边界代替被证据门阻止的推荐结论。
- 证据支持推荐结论时，输出推荐结论、未知、冲突、待裁决和下一步证据动作。
- 不输出完整探索链路或已查、未查路径。
- 不创建或更新 `.codeartz/<topic>/boundary.md`。
