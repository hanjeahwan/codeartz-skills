# 目标边界合同产物手册

## 目标

- 复核证据与边界分析结果是否足以写入目标边界合同。
- 只维护一个有效的 `.codeartz/<topic>/boundary.md`。
- 记录证据基线、关键证据和开工入口。
- 本阶段不重新执行证据与边界分析。

## 执行规则

### 阶段入口

- 前置检查
  - 检查项：`references/evidence-analysis.md` 要求的分析是否已完成。
  - 放行条件：代码事实、语义边界和目标合同均已形成。
  - 失败条件：任一前置结果缺失或因新证据失效。
  - 处理方式：停止本阶段，回到证据与边界分析阶段。
  - 禁止：不在本阶段补造分析结论。

### 写入前检查

- 字段适用性合同
  - 必需字段必须存在、非空、没有模板占位符，并满足枚举合同。
  - 条件字段适用时遵守同一要求；确认不适用时只能写“不适用”。
  - 未知、冲突、未授权和待裁决不得写成“不适用”。
  - 任一字段失败时停止写入，并报告字段路径和失败原因。

- 全局边界检查
  - 检查项：结论是否符合 `SKILL.md` 的全部全局边界。
  - 失败处理：回到最早受影响阶段。

- 依据和合同检查
  - 检查项：合同是否区分资料假设、代码事实、外部事实和系统现实。
  - 检查项：目标合同是否覆盖目标行为、输入、输出、状态变化、副作用、失败行为、用户可见变化和兼容约束。
  - 放行条件：每项结论都能追溯到对应证据和目标合同字段。

- 证据基线检查
  - 检查项：是否记录仓库、分支、HEAD、工作区状态、相关依赖锁定信息和分析时间。
  - 处理方式：仓库不是 Git 仓库时，对分支、HEAD 和工作区状态写“不适用”。
  - 失败条件：任一适用的基线字段无法读取。
  - 失败处理：在分析结果中把该字段写成“未知”，说明原因，并停止写入合同。
  - 禁止：不推测或补造基线值。

- 复杂变更检查
  - 检查项：合同是否覆盖全部命中的变更拓扑、演进方式和运行约束。
  - 放行条件：每个命中维度都有可定位证据、边界处理和合同限制。
  - 无内容处理：没有命中复杂变更维度时写“不适用”。

- 目标影响面检查
  - 检查项：推荐结论是否覆盖全部命中影响面。
  - 放行条件：每个命中影响面都有可定位证据、目标行为和边界处理。
  - 失败条件：命中影响面缺少代码、配置、测试、脚本或文档证据。
  - 处理方式：回到证据与边界分析阶段补证据。

- 推荐结论范围检查
  - 检查项：推荐结论是否只覆盖已授权分区。
  - 放行条件：未知、冲突、禁止和不适用分区没有被带入目标行为。

- 分区覆盖检查
  - 检查项：分区授权矩阵是否覆盖全部已知分区。
  - 放行条件：反证、未知、冲突和保持原行为分区均已列出。
  - 禁止：不只列被推荐结论修改的分区。

- 同入口变体检查
  - 检查项：入口级目标行为是否已枚举同入口变体。
  - 失败条件：存在未授权变体。
  - 处理方式：写入未知、冲突或待裁决。
  - 禁止：不写成普通风险。

- 粗粒度命中检查
  - 检查项：目标边界是否命中共享入口、共享层、状态分支、类型分支、事件、拦截器、配置或公共抽象。
  - 放行条件：命中面只覆盖已授权分区。
  - 失败条件：命中面会覆盖未知、禁止、不适用、保持原行为或冲突分区。
  - 处理方式：先收窄命中条件。
  - 裁决条件：不能收窄时写入待裁决。

- 风险检查
  - 检查项：已授权边界内仍可能爆发的问题是否写清代价归属、爆发时机和现实后果。

- 待裁决检查
  - 检查项：范围、行为或取舍问题是否已列出。
  - 检查项：每个待裁决项是否写清现实后果。
  - 无内容处理：写“不适用”。

- 证据收尾检查
  - 检查项：证据停止点是否写清停止原因。
  - 检查项：待补证据是否只包含不阻塞推荐结论的代码事实或外部事实。
  - 检查项：不能声称是否覆盖未实现、未验证、未测试或未检查的内容。
  - 无内容处理：没有内容时写“不适用”。

- 开工入口检查
  - 检查项：先读是否引用证据索引中的关键证据编号和定位。
  - 检查项：先复核是否列出开工前必须重新确认的事实。
  - 检查项：先补是否列出实现前必须补齐的证据。
  - 检查项：过期触发是否覆盖用户决定、需求、代码、分支、依赖、配置、数据或设计资料变化。
  - 禁止：不写文件级任务清单。

- 合同写入关卡
  - 放行条件：全部写入前检查通过。
  - 失败处理：不写目标边界合同。
  - 失败输出：只输出事实、缺口、限制、待裁决点和下一步证据动作。

### 产物路径

- 用户可以指定 `<topic>`。
- `<topic>` 使用小写 kebab-case。
- 用户不能改变 `.codeartz/` 根目录。
- 用户不能改变 `boundary.md` 文件名。
- 用户没有指定 `<topic>` 时，从点名的功能、issue、变更、模块、路由或主要对象生成 `<topic>`。
- `<topic>` 无法判断时，使用 `target-boundary`。
- 合同通过写入关卡后，只写入 `.codeartz/<topic>/boundary.md`。
- 禁止把合同写入 ADR、issue、PR description、README 或实现文件。

- 已有文件处理
  - 检查项：`.codeartz/<topic>/boundary.md` 是否存在。
  - 缺失处理：创建 `.codeartz/<topic>/` 目录和合同文件。
  - 已有处理：先读取现有合同，再更新同一个文件。
  - 禁止：不创建第二个有效合同。
  - 禁止：不静默丢弃现有已采纳决定或关键证据。

### 证据基线

- 仓库字段记录仓库根目录或可识别的仓库名称。
- 分支字段记录分析时的分支名称。
- HEAD 字段记录分析时的 commit SHA。
- 工作区状态字段记录 clean 或 dirty。
- 相关依赖字段只记录会影响当前结论的锁文件及其版本或指纹。
- 分析时间字段使用含时区的时间。
- 非 Git 仓库对应字段写“不适用”。

### 合同状态

- 状态裁定按“待裁决 → 可采纳 → 已采纳”的顺序执行。
- 首个满足条件的状态即为本轮唯一合同状态。

- 待裁决状态
  - 触发条件：合同仍有待裁决点。
  - 处理方式：合同状态写“待裁决”。

- 可采纳状态
  - 触发条件：合同没有待裁决点。
  - 触发条件：合同关卡通过。
  - 触发条件：用户未明确同意采用当前合同。
  - 处理方式：合同状态写“可采纳”。

- 已采纳状态
  - 触发条件：待裁决点已处理或明确无待裁决点。
  - 触发条件：合同关卡通过。
  - 触发条件：风险已经展示。
  - 触发条件：用户明确同意当前合同或要求按当前合同进入实现。
  - 处理方式：合同状态写“已采纳”。
  - 禁止：不因用户催促、询问流程或询问是否可以开始而写“已采纳”。

### 降级处理

- 对话降级
  - 触发条件：项目不可写。
  - 处理方式：只在对话输出。
  - 输出要求：说明没有写入项目文件的原因。

- 无代码降级
  - 触发条件：没有仓库或代码不可访问。
  - 处理方式：只输出资料假设、初始分区、候选代码路径和补证据动作。
  - 禁止：不写目标边界合同。

- 写入失败处理
  - 触发条件：合同写入失败。
  - 输出要求：只说明失败原因。
  - 禁止：不把本轮说成已产出目标边界合同。

## 输出要求

- 达到合同写入关卡后，从 Skill 根目录加载 `assets/target-boundary-template.md`。
- 写入合同文件时，使用目标边界合同模板。
- 分区状态和待裁决按 `SKILL.md` 的“分区分类”写入。
- 对话回复只包含合同路径、推荐结论和待裁决摘要。
- 完整目标边界合同以项目文件为准。
- 合同文件只写边界合同、关键证据和开工入口。
- 不把完整探索链路写入合同文件。
- 不把已查路径和未查路径写入合同文件。
