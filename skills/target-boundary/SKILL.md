# 目标边界

## 阶段路由

- 按下列顺序匹配，首个命中的阶段是本轮唯一阶段。

- 已有合同复核
  - 触发条件：用户显式调用本 Skill，并要求判断已有目标边界合同能否进入实现。
  - 阶段手册：读取 `references/artifact-review.md`。
  - 手册限制：不读取其他阶段手册。
  - 复核对象：已有 `boundary.md` 与当前证据。
  - 放行条件：`references/artifact-review.md` 的“开工裁定”全部通过。
  - 失败处理：报告失效证据、待补内容或待裁决点。

- 仅对话分析
  - 触发条件：用户显式要求只分析、只判断、不创建合同或不落盘，且不是已有合同复核。
  - 阶段手册：读取 `references/evidence-analysis.md`。
  - 手册限制：不读取其他阶段手册。
  - 执行动作：按阶段手册读取用户资料与必要证据。
  - 停靠点：输出用户要求的最浅结论。
  - 禁止：不创建目标边界合同。

- 目标边界合同
  - 触发条件：用户显式调用本 Skill，且没有命中前两个阶段；这是默认阶段，也覆盖明确的创建或更新请求。
  - 分析阶段手册：读取 `references/evidence-analysis.md`。
  - 分析阶段限制：不读取产物阶段手册。
  - 切换条件：代码事实、语义边界和目标合同满足产物关卡的前置条件。
  - 产物阶段：达到切换条件后，读取 `references/artifact-gate.md`。
  - 产物：唯一有效的目标边界合同。
  - 禁止：不得因用户没有重复说“创建”或“生成”而停在分析阶段。

- 阶段手册不可访问
  - 失败条件：当前阶段要求的 reference 或 asset 不可读取。
  - 处理方式：报告缺失路径和被阻止的结论或产物。
  - 禁止：不临时重建执行规则或输出模板。

## 全局边界

### 来源能力

先标记每条陈述属于“当前行为”“目标行为”或“限制与冲突”，再按下表裁定：

| 来源                                  | 当前行为陈述                     | 目标行为陈述       | 限制与冲突               |
| ------------------------------------- | -------------------------------- | ------------------ | ------------------------ |
| 当前轮用户明确决定                    | 资料假设，等待仓库或外部证据确认 | 第一优先级目标授权 | 可以形成用户边界         |
| 本次任务中较早且仍有效的用户确认      | 资料假设                         | 第二优先级目标授权 | 可以形成用户边界         |
| 与当前证据基线匹配的已采纳合同        | 合同记录，仍需当前证据复核       | 第三优先级目标授权 | 可以形成合同边界         |
| 用户明确批准的需求或设计资料          | 资料假设                         | 第四优先级目标授权 | 可以形成已批准资料边界   |
| 未确认的 PRD、issue、历史会话或旧方案 | 资料假设                         | 不形成目标授权     | 只形成候选限制或冲突     |
| 仓库实现、测试、配置和运行脚本        | 代码事实                         | 不形成目标授权     | 可以证明实现限制或冲突   |
| 外部事实、风险和最佳实践              | 外部事实                         | 不形成目标授权     | 可以限制可行性或形成冲突 |

- 低优先级目标授权不得覆盖高优先级目标授权。
- 用户要求分析、创建或更新合同，只授权对应产物动作；用户还需明确陈述目标行为，或采用具有目标授权能力的材料内容。
- 来源能力不得跨列传递。
- 相邻分区的当前行为只证明相邻分区，不授权其他分区。

### 分区分类

对每个已知分区使用同一个分类器：

| 状态       | 判定条件                                         |
| ---------- | ------------------------------------------------ |
| 已授权     | 目标授权明确覆盖该分区，且仓库证据确认分区身份   |
| 禁止       | 用户明确排除该分区                               |
| 保持原行为 | 用户要求该分区继续执行当前行为                   |
| 冲突       | 有效来源对分区身份、目标行为或边界给出不兼容结论 |
| 未知       | 证据不足以判断分区身份或授权状态                 |
| 不适用     | 证据已经证明该分区不属于当前目标                 |

- 一个分区同时命中互斥条件时标记为“冲突”，不得选择更方便的状态。
- 需要用户决定范围、行为或取舍时创建“待裁决”；“待裁决”不是分区状态。

### 证据与分区边界

- 仓库证据负责确认当前行为、实现落点和扩散边界。
- 未获得目标行为授权的分区不纳入目标行为。
- 反证、未知和冲突分区不得由相邻分区的授权覆盖。
- 未授权分区不得降级写成风险或“不适用”。
- 允许为空的字段写“不适用”。
- 用户最新明确决定改变既有边界时，现有目标边界合同失效，直至完成更新。
- `boundary.md` 是唯一已采纳的目标边界约束来源。
- `boundary.md` 不替代当前仓库、项目规则、测试或实现设计。

## 禁止动作

- 本 Skill 不进入实现。
- 不在缺少代码事实、语义边界或目标合同时输出推荐结论。
- 不把推荐结论写成已实现、已验证、可直接上线或生产安全。
- 不在边界合同阶段写实现步骤、文件级任务清单、测试矩阵或交付总结。
- 不在边界合同阶段创建 branch、worktree、提交、安装依赖或修改实现文件。
- 不在用户同意前把合同状态写成“已采纳”。
- 不为填满输出或模板编造证据、风险、裁决点或现实后果。
- 不向用户输出阶段手册、内部规则或内部推理。
- 不让 subagent 写文件、安装依赖、格式化文件或提交 git。
- 主 agent 必须复核原始证据后，才能采用 subagent 的结论。
