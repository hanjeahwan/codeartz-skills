## 自我指涉 / 免责

报告条件：句子只描述文档自己，不指导行动，也不支撑具体规则。

改写方式：删除自述，或改成读者动作。

示例：

```text
问题：本流程不包含特定框架规则。
改写：按项目现有框架执行实现细节；本流程只规定审查顺序和放行条件。

```

## 画像 空壳

报告条件：角色扮演句后没有具体判断规则。

改写方式：删除 画像，或把角色转成行为标准。

示例：

```text
问题：把自己当作资深工程师。
改写：
- 先质疑需求边界。
- 优先选择最小改动。
- 旧规则可以删除时，删除旧规则。
- 禁止新增重复规则。

```

## 否定 / 对比定义

报告条件：用“不是 X，而是 Y”定义身份或结构，却没有给出动作。

改写方式：改成正向定义或动作起点。

示例：

```text
问题：流水线不是从读 repo 开始。
改写：流水线从理解用户输入开始，再决定是否读取 repo。

```

## 空洞限定词

报告条件：形容词无法给出执行标准。

改写方式：补充可观察行为、阈值或失败处理。

示例：

```text
问题：适当处理错误。
改写：外部调用失败时返回降级值，并记录错误来源、请求 ID 和失败原因。

```

## 填充开场

报告条件：“重要的是”“需要注意”“总之”等引子不增加规则。

改写方式：删除引子，保留动作。

示例：

```text
问题：需要注意，提交前检查格式。
改写：提交前检查格式。

```

## 语言混用

报告条件：说明性正文混入可本地化外语词，且该词不是专名、术语、字段、路径、命令、引用或契约值。

改写方式：替换为主语言表达。

示例：

```text
问题：source gate 通过后创建 scratch pad。
改写：来源关卡通过后创建任务记录。

```

## 无支撑口号

报告条件：规则只给结论，没有触发条件、边界或失败后果。

改写方式：补触发条件、约束边界或失败后果；补不出就删除。

示例：

```text
问题：保证输出可靠。
改写：引用文件内容时标注文件来源；找不到来源时明确写“未找到依据”。

```

## 章节职责混杂

报告条件：同一节同时承担入口、流程、判定、输出格式、异常处理等多种职责，导致读者需要自行拆解。

改写方式：只提案，不默认重排文件。建议拆成：

- 入口：触发条件和下一步。
- 流程：动作顺序。
- 关卡：放行条件。
- 产物：输出格式。
- 例外：豁免和失败处理。

## 规则不可独立勾选

报告条件：单条规则必须依赖上一段、上一级标题或隐含对象，才知道要检查什么或执行什么。

改写方式：补出对象、动作、边界或成功标准；每一行只保留一个可独立勾选的动作或约束。

示例：

```text
问题：按上述规则处理。
改写：按“保留词”清单跳过路径、命令、配置 key 和直接引用。

```

## 复合句挤压

报告条件：一句同时塞入条件、动作、禁止、例外中的两类以上，导致句尾禁止或例外容易被漏掉。

改写方式：条件单独成行；每个动作单独成行；每个禁止单独成行。

示例：

```text
问题：用户给出路径时优先使用该路径，不要扫描无关目录。
改写：
- 用户给出路径时，优先使用用户路径。
- 禁止扫描无关目录。

```

## 隐式分支

报告条件：句子表达“满足 X 时做 A，否则做 B”，但分支后果没有显式命名。

改写方式：改为命名槽位；使用适用的“检查项、放行条件、失败条件、处理方式、裁决条件、禁止”。

示例：

```text
问题：未授权变体写入待裁决，不写成普通风险。
改写：
- 同入口变体检查
  - 检查项：入口、路由、页面、模块级目标行为是否已枚举同入口变体。
  - 失败条件：存在未授权变体。
  - 处理方式：写入待裁决。
  - 禁止：不写成普通风险。

```

## 深嵌套

报告条件：简单规则超过 1 层缩进，或分支规则超过 2 层缩进。

改写方式：简单规则压平到 0-1 层；分支规则保留“规则名 + 命名槽位”两层；槽位内不继续嵌套子树。

## 阶段越界

报告条件：某阶段要求预读后续阶段、提前生成下一阶段产物，或缺少明确阶段切换条件。

改写方式：补出阶段触发条件、放行条件和失败处理；条件未满足时停止在当前阶段。

## 权威位置漂移

报告条件：同一条规则在多个文件维护，或入口文件保存阶段执行细节，导致后续修改可能分叉。

改写方式：

- 选择一个权威位置并移动内容。
- 入口文件只保留触发条件、阶段路由、全局边界、禁止动作。
- 阶段手册只保留执行规则、检查项、产物规则、输出模板。

禁止：不要用复制代替移动。

## 指代不清

报告条件：“上述”“前述”“相关”“对应的”“它”等找不到明确对象。

改写方式：替换为具体章节、步骤、对象或字段。

示例：

```text
问题：按上述方式校验。
改写：按第 2 步的非空字符串规则校验。

```

## 不可测阈值

报告条件：动作明确，但缺少数量、时间、范围或判断标准。

改写方式：补充可测阈值。

示例：

```text
问题：失败后多次重试。
改写：失败后按 1s、2s、4s 退避重试，最多 3 次。

```

## 被动藏责任人

报告条件：需要人负责的动作使用无主语被动。

改写方式：补出负责人和动作时机。

示例：

```text
问题：变更应当被审核。
改写：值班 reviewer 在合并前审核变更。

```

## 时效性断言

报告条件：使用“目前”“现在”“最近”“新版”“暂时”等表达会腐烂的事实，且未给日期、版本或权威来源。

改写方式：指向事实来源或固定版本快照。

示例：

```text
问题：目前使用 v2 API。
改写：使用 `package.json` 声明的 API 版本。

```

## 占位残留

报告条件：TODO、FIXME、XXX、TBD、待定、待补、待填等出现在发布正文中。

改写方式：补全内容；不能补全时删除该句。

## 示例当规则

报告条件：示例没有标明是示例，导致偶然细节看起来像强制规则。

改写方式：标明示例身份，并指向真实选择原则。

示例：

```text
问题：用 redis 缓存。
改写：使用项目现有缓存层；redis 仅为示例。

```

## 重复事实来源

报告条件：同一事实已在权威文件定义，目标文件再次定义且可能分叉。

改写方式：删除重复定义，改为引用权威来源。

示例：

```text
问题：本文件重复维护部署端口。
改写：部署端口以 `README` 的“运行配置”章节为准。

```

## 空字段造内容

报告条件：允许为空的字段留空、用编造内容补齐，或把未授权 / 不在范围内的内容降级写成“风险”或“不适用”。

改写方式：允许为空的字段显式写“不适用”；未授权 / 不在范围内的内容写入“未知”“冲突”或“待裁决”。
