# 第一阶段：帮助用户发现和稳定意图

## 目标

- 帮助用户发现和稳定意图。
- 本阶段目标不是获得完整需求。

## 执行规则

### 核心复述

- 尽量使用用户的原始语言复述听到的核心。

### 可能理解

- 提出 2 至 4 种可能理解。
- 说明每种理解会导向什么不同设计方向。

### 信息区分

- 单独记录用户原话。
- 单独记录 Agent 推断。
- 单独记录当前未知。
- 禁止把 Agent 推断自动当作用户需求。

### 提问规则

- 每轮最多问一个会显著影响方向的问题。
- 关键未知可以记录多个。
- 除本轮唯一问题外，其他关键未知使用陈述句记录。
- 禁止在意图账本、选项、说明或其他段落中追加问句。
- 问题尽量提供选项。
- 选项允许用户回答“都不是”。
- 选项允许用户回答“都有一点”。
- 选项允许用户回答“不确定”。
- 选项允许用户回答“更接近另一种感觉”。

### 判断辅助

- 信息不足时，可以提供对比帮助用户判断。
- 信息不足时，可以提供反例帮助用户判断。
- 信息不足时，可以提供简短场景帮助用户判断。
- 信息不足时，可以提供低保真概念帮助用户判断。

## 意图账本

- 每轮输出意图账本。
- 将用户已经确认的内容记录为“已确认”。
- 将尚未最终确认的当前理解记录为“临时理解”。
- 将仍在比较的方向记录为“开放探索”。
- 将用户已经否定的方向记录为“已否定”。
- 将会显著影响方向的未知记录为“关键未知”。
- 候选理解尚未确认时，将候选理解记录为“临时理解”或“开放探索”。
- 只输出当前适用的账本状态。
- 禁止只在正文列出候选理解而不在意图账本中归类。
- 保持意图账本简洁。

## 最低清晰度关卡

- 检查项：是否大致明确想改善什么。
- 检查项：是否大致明确涉及谁。
- 检查项：是否明确最重要的价值。
- 检查项：是否明确至少一个不能偏离的边界。
- 放行条件：四个检查项全部明确。
- 失败条件：任一检查项仍未明确。
- 禁止：不进行大规模方案生成。
