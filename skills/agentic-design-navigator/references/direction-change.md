# D 模式：方向变化

## 目标

区分明确转向、局部修正、并行分支和真实意图漂移，更新主题状态后回到适用模式。

## 分类

### 明确转向

- 触发条件：用户明确要求替换主体、情境、目的、价值、边界、成功标准或实现方向。
- 处理方式：将新方向设为当前基线，将被替换内容记录为历史方向。
- 禁止：不得把用户明确转向标记为意图漂移。

### 局部修正

- 触发条件：用户修正当前理解，但未替换核心目标方向。
- 处理方式：只更新被修正内容，保留其他有效状态。

### 并行分支

- 触发条件：用户明确要求保留当前主线并探索另一方向。
- 处理方式：保留当前主线，创建命名分支并分别维护基线。
- 禁止：不得用新分支覆盖当前主线。

### 真实意图漂移

- 触发条件：Agent 或新方案未经用户确认替换目的、违反仍有效的边界，或重新引入已否定方向。
- 处理方式：指出发生替换或冲突的位置。
- 处理方式：恢复最近一次有效基线。
- 处理方式：将新内容标记为“暂定”或“有争议”。

## 更新步骤

1. 比较变化前后的主体、情境、目的、价值、边界、成功标准和实现方向。
2. 只列出新增、删除、替换或弱化的内容。
3. 保留没有受到变化影响的已确认内容。
4. 更新当前基线、历史方向或命名分支。
5. 按更新后的清晰度返回 A、B 或 C 模式。

## 冲突处理

- 用户新输入明确撤销旧边界时，以新输入更新当前基线，并保留变更记录。
- 新输入与旧边界冲突但未明确撤销时，将冲突标记为“有争议”。
- 冲突会改变执行结果时，使用 `SKILL.md` 的单问题边界请求用户裁决。

## 完成关卡

- 检查项：变化已经分类，受影响状态已经更新，未受影响状态仍然有效。
- 路由条件：用户要求继续探索或比较受影响方向。
- 路由检查：命中路由条件时，受影响方向已经按清晰度进入 A、B 或 C 模式。
- 放行条件：状态更新完成；命中路由条件时，后续路由也已完成。
- 失败处理：继续完成缺失的状态更新或后续路由。

## 前台响应

- 明确说明本轮属于转向、局部修正、并行分支或意图漂移。
- 按 `SKILL.md` 的“前台投影”展示变化；用户要求继续探索或比较时，补充新方向的下一判断或下一动作。
- 只有真实意图漂移才使用“意图漂移”措辞。
- 禁止每次变化都重复完整历史基线。
