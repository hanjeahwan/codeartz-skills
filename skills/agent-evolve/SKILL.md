# Agent Evolve

硬路由：用户手动点名 Agent Evolve 时，无论请求最终是否越界、重复、冲突或没有候选，都必须读取 `references/workflow.md`，并在工作流要求时读取 `references/validation.md`；禁止从本页直接返回正式结果。

## 目标

把当前项目对话中会影响未来 Agent 决策、但会随本轮任务结束而丢失的判断转成项目规则。主动发现候选，避免用户重复纠正和未来 Agent 重犯同类错误。

## 触发与排除

- 上下文存在 `AGENT EVOLVE ACTIVE — mode: safe|review` 时，在任务过程中和发送当前任务的最终回复前检查是否出现额外的未来 A→B 规则；不要求先发生故障或用户提及沉淀。
- 用户手动调用或点名 Agent Evolve、明确要求当前项目沉淀、评估可能成为项目规则的反馈，或审核精确候选时，直接进入工作流；即使请求最终因作用域、重复或未决状态不能沉淀，也不得在入口提前返回。用户要求暂不写入只改变结果状态，不取消评估。
- 自动加载前先分离当前任务交付：关闭 Agent Evolve 后，当前任务本来仍须交付的条件、决策后果和作用域只属于任务交付；目标文件、写入路径或是否修改规则源不同不能证明独立。
- 独立候选不要求与当前修复主题不同：同一根因或项目约束还能指导当前交付范围之外的未来同类实现时，只把有依据的额外条件、后果或作用域送入工作流。
- 当前任务同时产生独立的未来规则时，只把超出当前交付的规则范围送入工作流。
- 候选必须包含可观察条件、明确决策后果、可靠依据和未来决策差异。
- 项目合同或现有约束直接排除或优先一个可行选择，并会影响未来同类决策时，即使没有故障也形成候选。
- 发送最终回复前逐项检查本轮每个独立的未来决策；发现一条候选不能替代其他候选。
- 空泛目标缺少用户或任务证据直接支持的条件与后果时，入口即判为无候选，不得读取 reference 后再替用户补造规则。
- 只有当前事实、局部细节、一次性要求、单次成功、普通否定、无决策差异的失败、无独立证据的 Agent 观察、用户明确禁止泛化，或个人级与跨项目目标时，不自动加载工作流或输出回执。

## 模式路由

- `safe`：模式本身是当前项目自动写入的预授权。任何来源的候选通过安全门后直接沉淀；候选来源只影响证据判断和回执，不追加确认门。
- `review`：主动发现候选但只输出提案；用户随后明确批准精确规则和目标位置时，重新通过安全门后写入。
- `off`：禁止主动发现和持久化；只有用户手动调用 `$agent-evolve` 时进入工作流。
- 用户当前消息明确禁止写入、限制为一次性或要求保持文件不变时，该边界优先于 `safe`。

## 渐进路由

- 自动发现的候选通过入口判断后，读取相对 `references/workflow.md`；用户明确调用或要求评估时直接读取，由工作流判断作用域、重复、冲突和结果状态。
- 进入工作流时禁止预读 `references/validation.md`。
- 工作流完成候选、模式、落点和冲突判断并进入安全验证时，读取相对 `references/validation.md`。
- 没有候选时立即停止，不读取 reference，不输出 Agent Evolve 回执。
- `SKILL.md` 只负责入口与路由；任何正式 Agent Evolve 结果状态都必须经过工作流和安全验证，不得根据本页摘要直接作答。

## 全局边界

- 只处理当前项目规则，只在与用户对话的主会话中自动运行。
- 先完成当前用户任务，再处理不会阻塞任务的规则候选。
- 候选与有效项目规则冲突时，在修改会选定冲突方向的实现或测试前进入工作流并停止选边。
- 禁止写入用户级、全局、其他项目或第三方规则源。
- 禁止扫描完整对话记录做会后复盘、使用模型记忆补造反馈、创建反馈收集箱、自动提交 git commit 或启动额外 agent 轮次。
