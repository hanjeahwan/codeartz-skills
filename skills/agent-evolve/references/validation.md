# Agent Evolve 安全验证

## 目标

在写入、提案或拒绝前核对独立性、模式权限、项目作用域与实际结果，并用可识别的 Agent Evolve 回执区分任务交付和规则沉淀。

## 自动写入安全门

直接写入必须全部满足：

- 候选通过交付独立性和全部候选关卡。
- 候选状态是可沉淀。
- 目标只属于当前项目。
- 当前用户没有禁止写入、泛化或修改目标文件。
- 当前模式是 `safe`，或用户已在 `review`、`off` 手动调用中明确批准精确规则和目标位置。
- 决策记录的来源、依据、确定性和规则关系已准确记录；每条依据都已做反事实检查，所有会改变选择的依据变化都进入失效条件。
- 最终规则是该记录的无损投影：包含全部适用条件、必须与禁止动作、最窄作用域和非空失效条件；写入后允许与禁止的动作集合没有变化。
- 已确定唯一权威落点和未来 Agent 读取路径。
- 目标文件可写，写入前已重新读取。
- 没有相同规则、未解决冲突或无法合并的并发变化。
- 抽象规则与回执不包含敏感内容。

任一安全门失败时禁止写入，选择已有规则覆盖、待审核、未沉淀或处理失败。

## 来源与证据

- `候选来源` 使用：用户反馈、项目规则、测试、工具、代码或 diff、审查、Agent 观察（独立证据）。
- 来源只说明规则依据，不代表用户授权，也不改变 `safe` 模式权限。
- 证据只保留直接决定条件、后果和作用域的项目事实。
- 自动写入时必须能指出实际规则位置和重新读取结果。
- 仓库资料不得表述为当前用户反馈。
- 没有候选时，来源标记为不适用。

## 敏感内容门

- 禁止在规则、证据和回执中保存或复述密钥、私有 URL、邮箱、客户名、ticket、事故名称、私有项目标识、长日志或原始堆栈。
- 只保留仍会改变未来决策的脱敏语义。
- 实际错误包含敏感内容时，先脱敏再引用证据。

## 结果状态

- `已沉淀`：规则已成功写入并重新读取确认。
- `已有规则覆盖`：现有规则已覆盖候选条件、动作和禁止项，文件保持不变。
- `待审核`：候选精确，但模式、用户边界、冲突、未决状态、落点或读取路径阻止写入。
- `未沉淀`：手动调用没有形成独立、精确、当前项目内的候选。
- `处理失败`：读取、写入或重读失败，实际状态无法确认。

- 自动模式没有候选时保持静默，不输出 `未沉淀`。
- 当前任务交付不输出 Agent Evolve 回执。
- 多条候选分别输出，不用一条结论覆盖不同状态。

## 默认回执

没有冲突或处理失败时，每条候选只输出一行：

```text
Agent Evolve｜已沉淀（<候选来源>）｜规则：<决策记录的无损投影>｜位置：<项目相对路径与可识别锚点>
Agent Evolve｜已有规则覆盖｜规则：<决策记录的无损投影>｜位置：<项目相对路径与可识别锚点>
Agent Evolve｜待审核（<候选来源>）｜规则：<决策记录的无损投影>｜下一步：<解除当前阻塞的最小动作>
Agent Evolve｜未沉淀｜<没有独立候选或超出当前项目作用域的直接原因>
```

- 默认回执字段是封闭 schema：规则字段只放规则投影，位置字段只放权威位置，下一步字段只放动作；禁止向任一字段追加证据、覆盖差异或解释。
- 位置使用能唯一识别权威规则的最窄稳定锚点，例如 Markdown 路径加章节；只有文件本身就是最小单位时才只写路径。
- `已沉淀` 只用于实际写入成功，禁止用它描述当前任务文件修改。
- `下一步` 只解除已经识别的阻塞条件；不能解除全部安全门时，不得声称普通批准后即可写入。
- 冲突的下一步必须命名待裁决的决策轴和互斥选项；禁止把“批准候选文本”当作设计裁决。
- 默认回执不重复罗列完整证据，不在回执前后复述同一结论。
- Agent Evolve 状态只渲染一次。用户当前任务仅是评估或沉淀时，最终回复就是回执；存在其他任务交付时，回执只附在任务报告末尾，回执后不得追加 Agent Evolve 解释。

## 展开回执

候选存在未解决冲突，或反馈处理失败时使用：

```text
Agent Evolve｜<待审核 | 处理失败>
候选来源：<来源 | 不适用>
直接原因：<冲突或失败>
证据：<决定结论的项目事实>
目标位置：<项目相对路径 | 不适用>
建议规则：<精确提案 | 不适用>
下一步：<仅待审核；解除阻塞所需的裁决或动作>
```

- `处理失败` 不输出下一步；直接原因和证据说明已知恢复条件，未知时如实标记。
- 读取失败时不得声称完成查重或查冲突。
- 写入或重读失败时不得声称规则已沉淀。
- Agent Evolve 失败不得阻止当前用户任务继续完成。
