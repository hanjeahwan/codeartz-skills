# Agent Evolve 安全验证

在写入、提案或拒绝前核对工作流结果，并用 Agent Evolve 回执区分任务交付与规则沉淀。

## 安全门

直接写入前必须全部满足：

- `workflow.md` 已产生一份通过全部关卡、状态为可沉淀的决策记录。
- 决策记录已完成所有权检查，不是对 `当前交付所有者` 已负责决定的遵循、执行纠正或原位修改；已有规则覆盖、约束证据和权威冲突已经按各自分支处理。
- 当前模式允许写入，目标只属于当前项目，用户未禁止写入、泛化或修改目标文件。
- 决策记录的来源、依据、确定性、规则关系和失效条件完整。
- 规则包含全部条件、后果、最窄作用域和非空失效条件，且投影前后允许与禁止的动作集合不变。
- 唯一权威落点已有未来读取路径。
- 目标可写，且写入前已经重读。
- 重读后没有相同规则、未解决冲突或无法安全合并的并发变化。
- 规则、证据和回执已经脱敏。

任一条件失败时禁止写入，返回已有规则覆盖、待审核、未沉淀或处理失败。

## 来源与敏感内容

- `候选来源` 直接使用决策记录的 `来源`；来源不改变模式权限，仓库资料不得标为用户反馈。
- 证据只保留直接决定条件、后果和作用域的项目事实。
- 自动写入必须能指出实际规则位置和重读结果。没有候选时来源为不适用。
- 禁止保存或复述密钥、私有 URL、邮箱、客户名、ticket、事故名称、私有项目标识、长日志或原始堆栈。
- 错误包含敏感内容时，先脱敏再引用，只保留会改变未来决策的语义。

## 结果状态

- `已沉淀`：规则已成功写入并重读确认。
- `已有规则覆盖`：现有规则覆盖全部条件、动作、禁止项和失效条件，文件保持不变。
- `待审核`：模式、用户边界、冲突、未决状态、落点或读取路径阻止写入。
- `未沉淀`：手动调用没有形成独立、精确、当前项目内的候选。
- `处理失败`：读取、写入或重读失败，实际状态无法确认。

- 自动模式没有候选时保持静默。
- 当前任务交付不输出 Agent Evolve 回执。
- 多条候选分别输出状态，不用一条结论覆盖不同状态。

## 默认回执

没有冲突或处理失败时，每条候选只输出一行：

```text
Agent Evolve｜已沉淀（<候选来源>）｜规则：<无损投影>｜位置：<项目相对路径与稳定锚点>
Agent Evolve｜已有规则覆盖｜规则：<无损投影>｜位置：<项目相对路径与稳定锚点>
Agent Evolve｜待审核（<候选来源>）｜规则：<无损投影>｜下一步：<解除当前阻塞的最小动作>
Agent Evolve｜未沉淀｜<没有独立候选或超出当前项目作用域的直接原因>
```

- 回执字段是封闭 schema：规则只放规则投影，位置只放权威位置，下一步只放解除当前阻塞的动作。
- 禁止向回执字段追加证据、覆盖差异或解释。
- 位置使用最窄稳定锚点，例如 Markdown 路径加章节；文件本身是最小单位时才只写路径。
- `已沉淀` 只描述实际写入成功，不描述当前任务文件修改。
- 下一步不能解除全部安全门时，禁止声称普通批准后即可写入。
- 冲突时，下一步必须命名决策轴和互斥选项。
- 默认回执不重复罗列完整证据，回执前后不复述评估、覆盖关系或同一结论。
- 状态只渲染一次：纯评估或沉淀任务的最终回复只能是回执；另有任务交付时，把回执放在任务报告末尾，回执后不再解释 Agent Evolve。

未解决冲突或处理失败时，停止生成默认回执并读取 `failure-receipt.md`；其他状态禁止读取该文件。
