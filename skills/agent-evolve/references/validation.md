# Agent Evolve 安全验证

## 目标

在更新、提案或不沉淀前验证安全性，并为每条反馈候选生成可核对的原因与证据回执。

## 安全门

直接写入必须同时满足：

- 工作流已把当前用户内容判定为可复用反馈候选。
- 工作流已生成不依赖私有事故上下文的抽象规则。
- 工作流已提供唯一权威落点。
- 工作流已提供未来 agent 读取路径证据。
- 目标文件可写。
- 工作流已判定没有相同规则需要标记为 `已有规则覆盖`。
- 工作流已判定没有未解决冲突。
- 抽象规则与证据不包含敏感内容。
- 工作流已判定当前模式允许写入。
- 当前用户消息没有禁止写入。

- 任一安全门失败时禁止直接写入。

## 敏感内容门

规则与证据都不得保存或复述：

- 密钥。
- 私有 URL。
- 邮箱。
- 客户名。
- ticket 编号。
- 事故名称或事故细节。
- 私有项目标识。
- 长日志或原始堆栈。

- 只保留经过抽象、仍会改变未来决策的语义。
- 实际错误包含敏感内容时，先脱敏再写入证据。
- 整份回执及其前后说明都不得复述敏感原文。

## 证据门

- `原因` 写明本次决策的直接原因。
- `证据` 引用用户消息中会改变未来决策的语义，不复制敏感原文。
- 文件证据写明文件和标题或行号。
- 声明目标唯一时写明检查过的候选范围。
- 声明没有重复时写明查重范围。
- 声明没有冲突时写明检查过的相邻规则、反向规则或候选文件。
- 声明后续会话会受益时写明自动加载机制或现有项目指令路由。
- 更新成功时引用实际 diff 或重新读取结果。
- 测试或审查参与判断时写明命令、结果或结论位置。
- 禁止只写“已检查”“符合要求”或“判断安全”。

## 决策选择

- 安全写入成功：`已更新`。
- 已有规则语义相同：`已有规则覆盖`。
- 当前模式、权威落点、读取路径或冲突条件不允许直接写，但存在精确建议：`已提案`。
- 内容只适用于当前任务，或不适合作为长期规则：`不沉淀`。
- 已尝试的读取、状态、写入或验证动作失败：`处理失败`。

- 同一轮存在多条反馈候选时，每条候选分别输出 `反馈结论`、`原因` 和 `证据`。
- 多条回执可以放在同一个紧凑区块中。
- 禁止用一条汇总结论覆盖不同候选的处理结果。
- 所有回执的字段名必须精确使用 `反馈结论`、`原因`、`证据`、`目标位置` 与 `建议变更`。
- `已提案` 必须额外使用精确字段名 `推荐操作`。
- `推荐操作` 必须针对工作流识别的阻塞条件给出推荐动作。
- 项目证据不足以推荐唯一方案时，`推荐操作` 必须写明需要用户裁决的具体选项。
- 推荐动作不能解除全部安全门时，禁止声称用户普通批准后即可写入。
- 禁止翻译、改写或追加括号说明。

## 回执模板

### 已更新

```text
反馈结论：已更新
原因：<为什么该反馈会改变未来项目决策>
证据：<为什么该位置唯一、未来 agent 会读取、无重复且无冲突；以及实际 diff 证据>
目标位置：<文件和标题>
建议变更：<写入后的抽象规则>
```

### 已有规则覆盖

```text
反馈结论：已有规则覆盖
原因：不应生成第二份相同规则。
证据：<现有文件、标题或行号，以及查重范围>
目标位置：<已有规则的文件和标题>
建议变更：不适用
```

### 已提案

```text
反馈结论：已提案
原因：<为什么当前模式或安全门不允许直接写入>
证据：<候选位置、未来读取路径、冲突或缺失证据>
目标位置：<建议文件和标题>
建议变更：<精确提案>
推荐操作：<解除当前阻塞条件的最小下一步；无法推荐唯一方案时写明待裁决选项>
```

### 不沉淀

```text
反馈结论：不沉淀
原因：<为什么反馈不适合成为长期项目规则>
证据：<当前用户消息中的范围证据或项目证据>
目标位置：不适用
建议变更：不适用
```

### 处理失败

```text
反馈结论：处理失败
原因：<失败动作>
证据：<脱敏后的实际错误或验证结果>
目标位置：<尝试修改的文件或不适用>
建议变更：不适用
```

## 失败边界

- 反馈处理失败不得阻止当前用户任务继续完成。
- 读取目标失败时不声称已完成查重或查冲突。
- 写入失败时不声称规则已更新。
- `git diff` 验证失败时使用 `处理失败` 回执。
