# Agent Evolve 安全验证

## 目标

在更新、提案或不沉淀前验证安全性，并为每条反馈候选生成可核对的原因与证据回执。

## 安全门

直接写入必须同时满足：

- 当前用户内容是可复用反馈候选。
- 候选直接来自用户。
- 候选状态是独立收敛或合并收敛。
- 候选不处于暂定、待验证、冲突或未确认的多轮归约状态。
- 已生成不依赖私有事故上下文的抽象规则。
- 已确定唯一权威落点。
- 已证明未来 agent 会读取目标规则。
- 目标文件可写。
- 没有相同规则。
- 没有未解决冲突。
- 抽象规则与证据不包含敏感内容。
- 当前模式允许写入。

- 失败条件：任一安全门不通过。
- 处理方式：选择 `已有规则覆盖`、`已提案`、`不沉淀` 或 `处理失败`。
- 禁止：不直接写入。

## 敏感内容门

- 规则、证据和回执前后说明不得保存或复述密钥、私有 URL、邮箱、客户名、ticket 编号、事故名称或细节、私有项目标识、长日志或原始堆栈。
- 只保留经过抽象、仍会改变未来决策的语义。
- 实际错误包含敏感内容时，先脱敏再写入证据。

## 证据门

- `原因` 写明本次决策的直接原因。
- `证据` 只展示直接决定结论的用户语义和项目事实。
- 文件证据写明相关文件和标题或行号。
- 读取路径影响结论时，写明自动加载机制或现有项目指令路由。
- 更新成功时，引用重新读取结果。
- 测试、审查或现有 diff 影响结论时，写明结果或结论位置。
- 禁止只写“已检查”“符合要求”或“判断安全”。

## 决策与字段

- 决策映射：安全写入成功用 `已更新`；语义相同用 `已有规则覆盖`；存在精确建议但因语义状态或操作安全门不能写入用 `已提案`；没有精确可复用候选或用户明确禁止泛化用 `不沉淀`；执行动作失败用 `处理失败`。

- 每条候选分别输出 `反馈结论`、`原因`、`证据`、`目标位置` 与 `建议变更`。
- `已提案` 额外输出 `推荐操作`。
- `已有规则覆盖` 与 `不沉淀` 没有变更时，`建议变更` 写 `不适用`。
- 没有目标位置时，`目标位置` 写 `不适用`。
- 多条候选可以放在同一个紧凑区块中。
- 禁止用一条汇总结论覆盖不同候选。
- 禁止翻译字段名、改写字段名或追加括号说明。

## 回执格式

```text
反馈结论：<已更新 | 已有规则覆盖 | 已提案 | 不沉淀 | 处理失败>
原因：<直接原因>
证据：<决定结论的用户语义和项目事实>
目标位置：<文件和标题 | 不适用>
建议变更：<写入结果或精确提案 | 不适用>
推荐操作：<仅已提案；解除阻塞条件的最小下一步>
```

- `推荐操作` 必须针对工作流识别的阻塞条件。
- 项目证据不足以推荐唯一方案时，`推荐操作` 列出需要用户裁决的选项。
- 推荐操作不能解除全部安全门时，禁止声称用户普通批准后即可写入。

## 失败边界

- 反馈处理失败不得阻止当前用户任务继续完成。
- 读取目标失败时不声称已完成查重或查冲突。
- 写入失败时不声称规则已更新。
