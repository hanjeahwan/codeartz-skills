# Agent Evolve 安全验证

## 目标

在更新、提案或不沉淀前验证安全性，并为每条候选生成最短可执行回执；只有冲突或处理失败时展开审计依据。

## 安全门

直接写入必须同时满足：

- 当前用户内容或任务证据形成可复用候选。
- 候选来源已准确记录为用户反馈或具体任务证据。
- 用户反馈候选直接来自用户；证据派生候选已获得用户对精确语义的确认。
- 候选状态是独立收敛或合并收敛。
- 候选不处于暂定、待验证、冲突或未确认的多轮归约状态。
- 候选是低风险变更，或高风险变更已获得用户对精确持久化动作的批准。
- 已生成包含可判定条件、明确决策后果和证据支持作用域的抽象规则；依赖可变前提时已保留复审条件。
- 已确定唯一权威落点。
- 已证明未来 agent 会读取目标规则。
- 目标文件可写。
- 没有相同规则。
- 没有未解决冲突。
- 抽象规则与证据不包含敏感内容。
- 当前模式允许写入。

- 失败条件：任一安全门不通过。
- 处理方式：选择 `已有规则覆盖`、`已提案`、`不沉淀` 或 `处理失败`。
- 禁止：不直接写入。

## 敏感内容门

- 规则、证据和回执前后说明不得保存或复述密钥、私有 URL、邮箱、客户名、ticket 编号、事故名称或细节、私有项目标识、长日志或原始堆栈。
- 只保留经过抽象、仍会改变未来决策的语义。
- 实际错误包含敏感内容时，先脱敏再写入证据。

## 证据门

- `候选来源` 写明用户反馈、项目规则、测试、工具、代码或 diff、审查、获得独立证据支持的 Agent 观察；没有精确候选时写 `不适用`。
- `原因` 写明本次决策的直接原因。
- `证据` 只展示直接决定结论的用户语义和项目事实。
- 证据派生候选写明可核对的命令、失败结果、文件位置或审查事实；主观观察同时写明独立支持证据。
- 禁止把任务证据、仓库资料或 Agent 判断表述为用户反馈或用户批准。
- 文件证据写明相关文件和标题或行号。
- 读取路径影响结论时，写明自动加载机制或现有项目指令路由。
- 更新成功时，引用重新读取结果。
- 测试、审查或现有 diff 影响结论时，写明结果或结论位置。
- 禁止只写“已检查”“符合要求”或“判断安全”。

## 决策与字段

- 决策映射：安全写入成功用 `已更新`；语义相同用 `已有规则覆盖`；存在精确建议但因语义状态或操作安全门不能写入用 `已提案`；没有精确可复用候选或用户明确禁止泛化用 `不沉淀`；执行动作失败用 `处理失败`。
- 安全判断始终核对候选来源、原因、证据、目标位置和建议变更；默认回执不重复展示不会改变用户下一步的信息。
- 默认每条候选只输出一行：结论、精确变更，以及仅 `已提案` 需要的最小下一步。
- 证据来源会改变确认或写入权限时，把来源紧跟在结论后，例如 `已提案（测试）`；不得因此展开完整字段。
- 默认回执的精确变更只写实际写入、已覆盖规则、提案规则或不沉淀原因；禁止把省略的原因、证据和目标位置塞进该字段或下一步。
- 多条候选分别输出一行。
- 候选存在未解决冲突，或反馈处理失败时，改用展开回执，保留来源、原因、证据、目标位置、建议变更和必要的推荐操作。
- 禁止用一条汇总结论覆盖不同候选。
- 禁止在默认回执前后再复述同一结论、原因或证据。

## 回执格式

默认回执：

```text
<已更新 | 已有规则覆盖 | 已提案 | 已提案（候选来源） | 不沉淀>｜<写入结果、覆盖位置、精确提案或不沉淀原因>[｜下一步：<仅已提案；解除阻塞条件的最小动作>]
```

冲突或处理失败的展开回执：

```text
反馈结论：<已提案 | 处理失败>
候选来源：<用户反馈 | 项目规则 | 测试 | 工具 | 代码或 diff | 审查 | Agent 观察（独立证据） | 不适用>
原因：<直接原因>
证据：<决定结论的用户语义和项目事实>
目标位置：<文件和标题 | 不适用>
建议变更：<写入结果或精确提案 | 不适用>
推荐操作：<仅已提案；解除阻塞条件的最小下一步>
```

- 默认回执的 `下一步` 与展开回执的 `推荐操作` 必须针对工作流识别的阻塞条件。
- `处理失败` 的展开回执不输出 `推荐操作`；原因和证据直接说明可恢复条件，未知时如实标记。
- 项目证据不足以推荐唯一方案时，`推荐操作` 列出需要用户裁决的选项。
- 推荐操作不能解除全部安全门时，禁止声称用户普通批准后即可写入。

## 失败边界

- 反馈处理失败不得阻止当前用户任务继续完成。
- 读取目标失败时不声称已完成查重或查冲突。
- 写入失败时不声称规则已更新。
