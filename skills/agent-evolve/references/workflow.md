# Agent Evolve 工作流

## 目标

把当前项目中独立于任务交付、会改变未来 Agent 行动的判断转成可执行规则候选，并按模式沉淀或提案。

## 输入拆分

把当前消息和任务证据拆成以下语义单元，分别判断：

- 当前任务动作与局部参数。
- 用户直接给出的未来规则。
- 由代码、测试、工具、项目约束、审查或任务推理支持的证据派生候选。
- 持久化范围、禁止写入和其他当前授权边界。
- 多轮纠正中的保留、替代、撤销和未决关系。

禁止把整条消息、整个任务或文件类型直接标记为候选。

每个独立决策分别进入候选关卡。发现一条候选不能结束本轮检查，也不能覆盖来源、条件或后果不同的其他候选。

## 交付独立性

- 显式沉淀请求：用户手动调用 Agent Evolve 或明确要求把规则写入当前项目时，沉淀本身就是本次授权动作，继续候选判断。
- 普通任务：关闭 Agent Evolve 后，列出完成当前任务本来仍须交付的决策语义，并与候选按适用条件、决策后果和作用域逐项比较。目标文件、写入路径或是否修改规则源不能证明独立。
- 失败条件：候选的适用条件、决策后果和作用域全部包含在当前任务本来仍须交付的语义中。
- 处理方式：把该语义保留为当前任务交付，不进入候选、落点或回执流程。
- 放行条件：候选至少有一项依据支持的条件、决策后果或作用域超出当前交付，并会带来未来决策变化。
- 判断边界：独立指当前交付之外的语义增量，不要求候选与当前修复使用不同原理。同一根因、失败机制或项目约束能够指导未来同类实现时，可以同时支持当前修复和独立候选。
- 混合情况：当前任务和未来规则同时存在时，只处理额外的规则范围。禁止把任务参数、任务文件改写或验证动作重复记作沉淀。

## 决策记录

每个候选只建立一份“决策记录”，作为判断、写入和回执的唯一语义来源：

- `适用条件`：未来 Agent 能从项目事实判断的触发条件。
- `决策后果`：条件成立时全部必须、禁止、优先或等待的动作，以及必须避免的失败后果。
- `作用域`：证据支持的最窄当前项目组件或任务类型。
- `来源`：用户反馈、项目规则、测试、工具、代码或 diff、审查、获得独立证据支持的 Agent 观察。
- `依据`：直接支持条件、决策和作用域的事实。
- `未来差异`：没有该规则时，未来哪个选择会不同。
- `确定性`：已收敛，或具体的暂定、待验证、待裁决部分。
- `规则关系`：与有效规则的关系，只能是新增、已有规则覆盖或冲突；冲突时记录决策轴和互斥选项。
- `失效条件`：逐项改变或删除依据后，所有会使当前选择变化、需要重新判断的前提变化；没有时标为不适用。

后续关卡只读取和更新这份记录，不重新生成另一版候选语义。任何输出都不得丢失会改变未来允许动作集合的非空字段。

## 候选关卡

决策记录必须全部通过：

- 交付独立性：候选不与普通当前任务交付重复。
- 决策差异：未来至少一类情境会因此选择不同动作。
- 可判定：适用条件可以从可观察事实判断。
- 可执行：规则能推出明确动作、禁止项、优先级或等待条件。
- 可迁移：适用于当前项目内一类未来任务，不只适用于当前实例。
- 有依据：用户直接反馈或可核对任务证据支持条件、全部决策后果和最窄作用域。
- 语义闭合：对每条依据做反事实检查；改变或删除该依据会改变选择时，对应变化已进入失效条件。
- 作用域可收敛：不依赖无证据外推，不扩展到其他项目。

任一关卡不通过时判为无候选。关键词、文件类型、工程领域和“以后”“必须”“沉淀”等措辞不能替代关卡。

## 来源与证据

- 用户反馈候选：适用条件、决策后果或稳定依据直接来自当前用户。
- 证据派生候选：代码、测试、工具结果、项目规则、diff 或审查直接证明条件、后果和最窄作用域。
- Agent 观察：必须同时引用独立、可核对的项目证据；主观判断本身不能成为依据。
- 仓库资料：只能作为项目证据，不能冒充当前用户反馈或授权。
- 工具失败：只有定位出可复用决策函数时才形成候选；超时、网络错误、缺少依赖或偶发环境现象本身不是候选。
- 运行环境限制：当前 runner、sandbox、临时凭证或会话能力只有获得项目内配置、代码或规则支持时才是项目证据；禁止把宿主限制直接写成当前项目规范。
- 当前事实、单次成功和现有技术选型：只有同时证明选择边界与未来决策差异时才形成候选。
- 项目合同与现有约束：直接排除或优先一个可行选择，并会影响未来同类任务时，可以单独证明决策边界。

候选来源只影响证据判断和回执，不改变 `safe` 模式的写入权限。没有失败发生不影响候选判断。

## 状态判断

### 可沉淀

- 候选通过全部关卡。
- `确定性` 已收敛且 `规则关系` 不是冲突。
- 当前用户没有禁止写入或泛化。
- 证据足以支持最窄作用域。

### 待审核

命中任一条件：

- 当前模式是 `review`，用户尚未批准精确写入。
- 用户要求先提案、暂不写入或保持目标文件不变。
- `确定性` 仍有暂定、待验证、待裁决或未收敛分区。
- `规则关系` 是冲突。
- 缺少唯一权威落点、现有读取路径或写入条件。

### 无候选

命中任一条件：

- 只是当前任务交付、事实、局部细节或一次性动作。
- 必须补造条件、后果或作用域才能形成规则。
- 用户明确禁止泛化。
- 目标是个人级、全局或其他项目规则。

自动模式遇到无候选时保持静默。用户手动调用时进入安全验证并返回未沉淀原因。

## 连续纠正

- 每轮先处理能独立成立的候选，不等待整份当前产物验收。
- 多轮关系需要合并时，记录保留、收窄、替代、撤销和未决分区。
- 用户完整确认时只保留最终关系；部分确认时只收敛对应分区。
- 沉默、任务切换、测试通过和 Agent 自证不能确认未决关系。
- 稳定交集满足全部候选关卡且没有用户写入禁令时，可以按当前模式处理；否则保持待审核。

## 模式决策

| 状态   | `safe`                 | `review`                     | `off` 手动调用                   |
| ------ | ---------------------- | ---------------------------- | -------------------------------- |
| 可沉淀 | 通过安全门后自动写入   | 输出提案；用户明确批准后写入 | 用户明确要求写入时才写，否则提案 |
| 待审核 | 输出提案和最小阻塞条件 | 输出提案和最小阻塞条件       | 输出提案和最小阻塞条件           |
| 无候选 | 静默返回当前任务       | 静默返回当前任务             | 返回未沉淀原因                   |

- 当前消息的禁止写入边界优先于模式授权。
- `safe` 是当前项目自动写入预授权，不再按候选来源增加确认步骤。
- `review` 的后续批准只覆盖用户明确确认的规则、作用域和目标位置。

## 原则抽象

- 把规则写成决策记录的无损投影：保留适用条件、决策后果、作用域和非空失效条件。投影后未来允许与禁止的动作集合必须与记录一致。
- 同一字段包含多个独立条件、动作、禁止项或失效路径时逐项保留；禁止用较弱的概括替代其中任一项。
- 删除日期、客户名、ticket、私有路径、原始日志、测试文件名和偶然示例值。
- 当前任务参数与通用规则并存时，只抽象额外的未来规则。
- 遵循目标规则源已有语言、结构和术语。

## 权威落点

按以下顺序寻找当前项目的唯一权威落点：

1. 用户明确指定且属于当前项目的现有规则源。
2. 项目已有且会被未来 Agent 自动读取的指令文件，例如 `AGENTS.md` 或 `CLAUDE.md`。
3. 已由项目读取入口明确路由的规则文件。
4. 项目拥有并纳入版本控制的 Skill；仅当额外候选直接约束该 Skill，且不与当前 Skill 修改任务重复时使用。

唯一权威落点必须满足：

- 文件承担长期规则职责。
- 候选与目标段落属于同一决策主题。
- 规则可以最小合并，不制造第二权威位置。
- 未来 Agent 已有读取路径。
- 没有同等强度的反向规则。

- 已知信息不足时，用 `rg --files` 有边界地查找项目内 `AGENTS.md`、`CLAUDE.md` 和已知规则路径。
- 禁止扫描用户目录、全局配置、其他项目或第三方 Skill。
- `safe` 模式不得自动创建新的规则源或读取入口；缺少落点时只提案最小入口和规则文本。

## 查重、冲突与脏状态

- 在目标段落、相邻规则和同主题项目规则源中查找相同语义，并更新规则关系。
- 现有规则覆盖条件、全部动作、禁止项和失效条件时，返回已有规则覆盖。
- 只有宽泛目标或同主题名词时，不得当作完整覆盖。
- 冲突时记录具体决策轴、现有选项和候选选项；禁止覆盖旧规则，也不得通过修改实现或测试让任一方向成为既成事实。
- 解决冲突需要用户先裁决决策轴；仅批准候选文本或替换文件不是独立的设计裁决。
- 写入前重新读取目标文件并再次查重、查冲突。
- 目标在检查后变化时重新判断；无法安全合并时停止写入。

## 写入与提案

### 写入

1. 完成当前用户任务中不受冲突阻塞的部分。
2. 重新读取唯一权威落点。
3. 通过查重、冲突、敏感内容和模式权限检查。
4. 最小合并抽象规则，不改无关内容。
5. 重新读取修改文件并核对实际结果。

### 提案

- 从同一决策记录给出精确规则、候选来源、直接阻塞条件和解除阻塞所需的最小下一步。
- 项目证据不足以确定唯一方案时，列出待用户裁决的具体差异。
- 禁止用提案修改规则文件或扩大当前授权。

完成候选、模式、落点和冲突判断后，进入安全验证阶段并读取相对 `validation.md`。最终结果只使用 `validation.md` 定义的回执格式。
