# Agent Evolve 工作流

## 输入与交付分离

把输入拆成当前任务动作与局部参数、用户直接给出的未来规则、证据派生候选、持久化边界，以及多轮纠正中的保留、替代、撤销和未决关系。每个独立决策分别判断。

禁止把整条消息、整个任务或文件类型直接标记为候选。

- 显式沉淀：用户手动调用 Agent Evolve 或要求把规则写入当前项目时，继续候选判断。
- 普通任务：关闭 Agent Evolve 后，列出当前任务本来仍须交付的决策语义，并与候选逐项比较适用条件、决策后果和作用域。三项均已包含时，只完成当前交付，不进入候选落点或回执。目标文件、写入路径或是否修改规则源不同不能证明独立。
- 混合输入：只处理当前交付之外、有依据且会改变未来选择的语义增量。同一根因、失败机制或项目约束可以同时支持当前修复和独立候选。
- 禁止把任务参数、文件改写或验证动作重复记作沉淀。

## 决策所有权

为每个可能的 `条件 → 后果` 先判断它与当前有效权威的关系，再建立候选。权威职责按适用范围和有效状态判断，不按文件名、路径或文件类型判断。

- 有效权威已经负责相同条件、后果和作用域，且用户只要求遵循它、修复执行偏差或按当前任务修改原权威内容时，标为 `当前交付所有者`：不形成候选，也不得复制到另一规则源。当前指令只拥有本次交付范围；用户明确给出的额外未来作用域不因此被排除。
- 现有长期规则已经覆盖同一未来决定时，标为 `已有规则覆盖`，继续查重流程并返回覆盖状态，不新增落点。
- 项目证据只提供约束、事实或失败机制，但没有直接规定待选动作时，可以支持由证据派生的独立未来决策；不得把“证据能支持选择”误判成“原权威没有拥有该决定”。
- 同一决定存在多个声称有效的所有者时，记录权威冲突并停止新增落点；禁止任选其一或再创建第三份规则。
- 移除 `当前交付所有者` 负责的语义后，只有仍然存在条件、后果或作用域增量时，才继续候选判断。

## 决策记录

每个候选只维护一份决策记录，供后续判断、写入和回执共同使用：

- `适用条件`：未来 Agent 可从项目事实判断的触发条件。
- `决策后果`：条件成立时全部必须、禁止、优先或等待的动作，以及要避免的失败后果。
- `作用域`：依据支持的最窄当前项目组件或任务类型。
- `来源`：用户反馈、项目规则、测试、工具、代码或 diff、审查、具有独立证据的 Agent 观察。
- `依据`：直接支持条件、后果和作用域的事实。
- `未来差异`：没有该规则时会改变的选择。
- `所有权关系`：当前交付所有者、已有规则覆盖、仅提供约束、无现有所有者或权威冲突；记录对应有效来源和作用域。
- `确定性`：已收敛，或具体的暂定、待验证、待裁决部分。
- `规则关系`：新增、已有规则覆盖或冲突；冲突时记录决策轴和互斥选项。
- `失效条件`：改变或删除依据后会使选择变化、需要重新判断的前提；没有时标为不适用。

后续步骤只更新这份记录。任何产物都必须保留会改变未来允许动作集合的非空字段。

## 候选关卡

决策记录必须全部通过：

- `交付独立性`：不与当前任务交付重复。
- `非权威内交付`：不是对 `当前交付所有者` 已负责决定的遵循、执行纠正或原位修改。
- `决策差异`：删除候选但保留当时有效的用户指令和权威内容后，至少一类未来情境会选择不同动作；动作不变时无候选。
- `可判定`：适用条件能在执行决策前从可观察事实判断，且不是决策后果、验收结果或待执行动作的改写。
- `可执行`：能推出明确动作、禁止项、优先级或等待条件。
- `可迁移`：适用于当前项目内一类未来任务，不只适用于本次实例。
- `有依据`：用户直接反馈或可核对证据支持全部条件、后果和最窄作用域。
- `语义闭合`：逐项反事实检查依据，并把所有会改变选择的变化写入失效条件。
- `作用域可收敛`：不依赖无证据外推，不扩展到其他项目。

任一关卡失败即无候选。关键词、文件类型、工程领域以及“以后”“必须”“沉淀”等措辞不能替代关卡。

## 证据边界

- 用户反馈直接支持的条件、后果或稳定依据，来源标为用户反馈。
- 代码、测试、工具、项目规则、diff 或审查直接证明决策记录时，来源标为对应项目证据。
- 仓库资料不得冒充当前用户反馈或授权。
- Agent 观察必须引用独立、可核对的项目证据。主观判断本身不是依据。
- 超时、网络错误、缺少依赖、偶发工具失败、runner、sandbox、临时凭证和其他宿主现象，只有项目证据证明了可复用决策边界时才形成候选。
- 当前事实、单次成功或现有选型，只有同时证明选择边界和未来差异时才形成候选。
- 项目合同提供的约束能够推出独立未来选择时，可以成为依据，不要求先失败；合同已经直接规定该选择时，它是权威所有者，不得把同一决定再次沉淀。

候选来源只影响证据判断和回执，不改变 `safe` 权限。

## 候选状态

- `可沉淀`：通过全部关卡；确定性已收敛；无冲突；证据支持最窄作用域；用户未禁止写入或泛化。
- `待审核`：模式或用户边界不允许写入，或存在未收敛分区、冲突、缺少唯一落点、读取路径或写入条件。
- `无候选`：只包含任务交付、事实、局部细节或一次性动作，必须补造语义，用户禁止泛化，或超出当前项目。

处理权限以 `SKILL.md` 的模式规则为准；当前消息边界优先。

## 连续纠正

- 每轮处理能独立成立的候选，不等待当前产物整体验收。
- 合并多轮关系时记录保留、收窄、替代、撤销和未决分区。
- 完整确认只保留最终关系；部分确认只收敛对应分区。
- 沉默、任务切换、测试通过和 Agent 自证不能确认未决关系。
- 稳定交集通过全部关卡且没有写入禁令时按模式处理，否则保持待审核。

## 语义抽象与规则投影

1. 把候选输入拆成实例定位、稳定判据、决策后果和作用域；实例定位只选择当前对象，不直接进入长期规则。
2. 对每个拟保留条件做反事实替换：直接证据表明替换后决策不变时删除该条件；证据表明决策会变化时保留；证据没有说明时标为未知，禁止自行补造边界。
3. 稳定判据必须在决策前可观察。禁止用执行结果、验收结论、待执行动作，或“用户要求执行某动作”反向定义同一动作的适用条件。
4. 找不到有依据的稳定判据时判为无候选；手动入口按未沉淀返回，自动入口静默停止。
5. 规则只保留适用条件、全部后果、最窄作用域和非空失效条件。删除日期、客户名、ticket、私有路径、原始日志、测试文件名和偶然示例值，并核对投影前后允许与禁止的动作集合不变。

遵循目标规则源的语言、结构和术语。

## 候选落点

只有决策所有权检查确认没有现有所有者后，才按顺序选择当前项目的唯一权威落点：

1. 用户指定且属于当前项目的现有规则源。
2. 未来 Agent 自动读取的项目指令文件，例如 `AGENTS.md` 或 `CLAUDE.md`。
3. 已由项目读取入口路由的规则文件。
4. 项目拥有并纳入版本控制的 Skill。只承载直接约束该 Skill、且不与当前 Skill 修改任务重复的额外候选。

落点必须全部满足：

- 承担长期规则职责。
- 与候选属于同一决策主题。
- 可最小合并且不会制造第二权威位置。
- 已有未来读取路径。
- 没有同等强度的反向规则。

- 信息不足时，只用 `rg --files` 有边界地查找项目内 `AGENTS.md`、`CLAUDE.md` 和已知规则路径。
- 禁止扫描用户目录、全局配置、其他项目或第三方 Skill。
- `safe` 不得自动创建规则源或读取入口。缺少落点时只提案最小入口和规则文本。

## 查重、冲突与脏状态

- 在目标段落、相邻规则和同主题项目规则源中查重，并更新决策记录的规则关系。
- 现有规则覆盖全部条件、动作、禁止项和失效条件时，返回已有规则覆盖。宽泛目标或同主题名词不算覆盖。
- 冲突时记录决策轴、现有选项和候选选项。
- 禁止覆盖旧规则，或修改实现、测试让任一方向成为既成事实。
- 用户必须先裁决冲突决策轴。批准候选文本或替换文件不等于设计裁决。
- 写入前重读目标并重新查重、查冲突。
- 目标发生变化时重新判断；无法安全合并时停止写入。

## 写入与提案

### 写入

1. 完成当前任务中不受冲突阻塞的部分。
2. 重读唯一权威落点，通过查重、冲突、敏感内容和模式权限检查。
3. 最小合并规则，不改无关内容。
4. 重读修改文件并核对实际结果。

### 提案

- 从同一决策记录输出精确规则、候选来源、直接阻塞条件和解除阻塞的最小下一步。
- 证据不足以确定唯一方案时，列出待用户裁决的具体差异。
- 禁止用提案修改文件或扩大授权。

完成候选、模式、落点和冲突判断后，读取 `validation.md`。最终结果只使用该文件定义的状态和回执。
