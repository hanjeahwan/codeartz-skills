# Agent Evolve 工作流

## 目标

把用户直接反馈转成可复用项目规则，写入未来 agent 已有读取路径；无法安全写入时给出可核对的提案或不沉淀原因。

## 输入

- 当前用户消息。
- 当前可见对话。
- 当前项目已有规则、规范、手册、文档和配置。
- 当前任务产生的代码事实、diff、测试结果与审查结论。
- 注入头中的当前模式。

## 反馈判断

候选反馈必须同时满足：

- 直接来自用户。
- 会改变未来项目任务中的 agent 决策。
- 能抽象为代码模式、架构、规范、边界或实践规则。
- 不依赖事故名称、日期、客户名、ticket 或私有上下文才能成立。

以下内容不是候选反馈：

- 普通问题。
- 单次任务需求。
- 只适用于当前文件的一次性选择。
- 子 agent 产生的观察。
- Agent 自己的总结或建议。
- 没有得到用户确认的审查发现。
- Agent Evolve 模式控制命令。

- 不要求用户使用“以后”“记住”“不要再”或“写进规则”。
- 不因消息出现上述词语就判定为候选。
- 使用当前用户消息、可见对话、代码事实、diff、测试和审查证据判断。

## 原则抽象

- 保留会改变未来决策的判断条件、动作与禁止项。
- 删除事故名称、日期、客户名、ticket、私有路径与原始日志。
- 把只描述一次失败的抱怨改写成跨任务可执行的规则。
- 一条规则只表达一个可独立检查的动作或约束。
- 条件、动作与禁止分别写成独立规则。
- 存在互斥后果时使用命名槽位表达分支。

## 落点发现

按以下优先级找候选：

- 用户明确指定的位置。
- 当前任务正在编辑或审查的规则文件。
- 项目已有 agent 指令文件，例如 `AGENTS.md` 或 `CLAUDE.md`。
- 项目已有 docs、手册、规范、policy、guide、manual、convention 或 instruction 文件。

- 项目级 Skill 候选
  - 放行条件：Skill 由当前项目拥有并纳入版本控制。
  - 放行条件：反馈直接约束该 Skill 的行为。
  - 放行条件：未来项目 agent 会加载或按现有路由读取该 Skill。
  - 禁止：不修改用户级、全局安装或第三方 Skill。

- 先用当前上下文与已知规则源定位。
- 已知来源不足时，使用以下有边界的文件发现命令：

```bash
rg --files -g 'AGENTS.md' -g 'CLAUDE.md' -g 'README.md' -g 'docs/**' -g '.github/copilot-instructions.md' -g '.cursor/rules/**' -g '.windsurf/rules/**' -g '.clinerules'
```

- 项目级 Skill 候选的放行条件全部满足时，才追加以下候选：

```bash
rg --files -g 'skills/**/SKILL.md' -g 'skills/**/references/**'
```

- 从用户反馈提取具体主题词，再对候选文件做 `rg -n` 查找。
- 禁止默认扫描全库。
- 禁止把 grep 命中直接当成权威落点。
- 禁止因为本技能位于 skills 仓库，就默认把其他项目反馈写进 `SKILL.md`。

唯一权威落点必须同时满足：

- 文件是项目长期规则源。
- 命中段落与抽象规则属于同一决策主题。
- 新规则能合并进已有段落，或创建最小相邻段落。
- 没有同等强度的反向规则。
- 不需要用户裁决才能选择位置。
- 有证据证明未来 agent 会自动加载该文件，或会通过现有项目指令路由读取该文件。

- 找不到唯一权威落点时不随机写入 README。
- 找不到唯一权威落点时不随机创建 docs 文件。
- 找不到未来 agent 读取路径证据时不自动写入。
- 无权威落点时给出建议的最小目标、精确规则文本与缺失证据。

## 查重与查冲突

- 在目标段落、相邻规则与同主题候选文件中查找相同语义。
- 反馈与已有规则语义相同时，判为 `已有规则覆盖`。
- 重复时不追加第二份规则。
- 原规则含糊时，只收紧原规则。
- 反馈只是已有规则的例子时，不新增规则。
- 查找同主题的反向规则、例外与优先级约束。
- 冲突时不覆盖旧规则。
- 冲突时记录文件、标题或行号，以及需要用户裁决的具体差异。

## 模式处理

### 安全沉淀模式

- 把候选、抽象规则、唯一权威落点、未来读取路径、查重结果与冲突结果交给当前上下文中的 `# Agent Evolve 安全验证`。
- 全部安全门通过时直接更新项目已有规则源。
- 用户明确要求本轮不修改文件时只输出精确提案。
- 安全门失败分支
  - 失败条件：任一安全门不通过。
  - 处理方式：输出精确提案或不沉淀回执。
  - 禁止：不写入。

### 审核后沉淀模式

- 生成目标位置、精确提案、原因与证据。
- 用户明确批准后才进入写入步骤。
- 批准只覆盖用户确认的提案，不扩展到其他候选或位置。

### 关闭模式下的手动调用

- 只处理用户本次手动指定的反馈。
- 用户明确要求写入或批准精确变更时，才进入写入步骤。
- 写入仍须通过当前上下文中的 `# Agent Evolve 安全验证` 全部安全门。
- 用户未明确要求写入或批准精确变更时，只输出精确提案。

## 提案推荐

- 每个 `已提案` 必须识别阻止本轮写入的直接条件。
- `推荐操作` 只推荐解除该阻塞条件所需的最小下一步。
- 推荐动作必须引用当前项目证据。
- 推荐动作完成后，重新验证全部安全门。
- 禁止承诺推荐动作完成后必然写入。

- 只缺批准
  - 失败条件：精确提案已满足其他安全门，当前模式或用户指令不允许本轮写入。
  - 推荐动作：请用户批准当前 `目标位置` 与 `建议变更`。
- 唯一权威落点不明确
  - 失败条件：存在多个同等强度的候选位置，或没有长期规则源权威落点。
  - 推荐动作：有项目证据支持一个候选时，推荐该候选并说明优先依据。
  - 裁决条件：项目证据无法区分候选时，请用户在列明的候选间指定权威落点。
- 未来读取路径不明确
  - 失败条件：无法证明未来 agent 会自动加载目标，或会通过现有项目指令路由读取目标。
  - 推荐动作：指出需要确认或补足的最小读取机制。
- 规则冲突
  - 失败条件：新规则与既有规则、例外或优先级约束冲突。
  - 推荐动作：项目证据支持一侧时，推荐保留或替换该侧并说明依据。
  - 裁决条件：项目证据不足以选择时，请用户裁决列明的具体差异。
- 多个阻塞条件
  - 处理方式：按解除依赖的顺序给出一个紧凑的推荐动作。
  - 禁止：不把“批准提案”写成权威落点、读取路径或冲突问题的充分解决方案。

## 写入流程

- 修改前重新读取目标文件。
- 比较目标内容是否从落点分析后发生变化。
- 目标变化时重新查重。
- 目标变化时重新查冲突。
- 无法安全合并并发变化时不写入。
- 只编辑唯一权威落点。
- 优先合并或收紧已有规则。
- 不制造第二权威位置。
- 保留工作区已有用户改动。
- 禁止回退、覆盖或清理无关用户改动。
- 写入后重新读取目标规则。
- 使用 `git diff -- <target>` 验证实际 diff。
- 最后按当前上下文中的 `# Agent Evolve 安全验证` 选择并输出每条候选的回执。
