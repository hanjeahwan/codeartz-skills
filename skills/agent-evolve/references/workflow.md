# Agent Evolve 工作流

## 目标

把用户直接反馈转成可复用项目规则，写入未来 agent 已有读取路径；无法安全写入时给出可核对的提案或不沉淀原因。

## 语义单元与范围

先把用户消息拆成当前动作、局部参数、反馈原则、持久化意图和收敛信号。分别判断每个语义单元，禁止用整条消息或整个任务的标签代替。

- 当前动作只执行一次时，只把动作和局部参数限制在当前任务。
- 用户同时给出跨任务原因、失败机制或决策边界时，单独评估该反馈原则。
- 用户明确禁止泛化某个语义单元时，将该单元排除为长期规则候选。
- 候选必须直接来自用户；禁止把 Agent 建议、子 Agent 观察、测试结果或审查发现改写成用户反馈。

## 候选关卡

反馈候选必须同时满足：

- 来源直接来自用户。
- 内容会改变未来项目任务中的 Agent 决策。
- 内容能抽象为不依赖私有上下文的代码模式、架构、规范、边界或实践规则。

重复纠正或具体抱怨只有在能从用户原话提取精确候选规则时才进入后续判断。纯粹的“还是不对”、要求继续修改或情绪表达不构成候选。

- 排除普通问题、只有当前产物细节的要求、用户明确限定为不适用于后续任务的选择和模式控制命令。
- 使用当前用户消息、可见对话、项目规则、代码事实、diff、测试和审查证据判断，不依赖触发关键词。

## 收敛关卡

为每条候选选择一个状态：

- 独立收敛：用户原话已经给出可复用触发条件、动作或禁止项，不依赖当前产物或其他反馈验收才能成立。
- 合并收敛：多轮反馈的保留、替代、撤销和未决关系已经得到用户确认。
- 精确但受阻：能生成精确候选规则，但仍处于暂定、待验证、冲突或未确认的多轮归约状态。
- 无精确候选：反馈没有稳定交集，或必须补造用户未表达的边界才能形成规则。

### 连续纠正

- 每轮先判断是否存在独立收敛原则；存在时当轮处理，不等待整个产物验收。
- 需要合并、替代或归约前序反馈时，在当前可见上下文中建立纠正链。
- 按可独立验收的分区记录保留、收窄、替代、撤销和未决关系。
- 用户完整肯定时确认整链；部分肯定时只确认对应分区；继续否定时保持相关分区未收敛。
- 沉默、任务切换、测试通过、Agent 自证和要求继续修改均不构成多轮归约确认。
- 未确认的多轮归约存在精确稳定交集时，保留为精确但受阻候选。
- 未确认的多轮归约没有精确稳定交集时，判为无精确候选。

## 反馈决策

- 可写入候选
  - 放行条件：候选状态是独立收敛或合并收敛。
  - 处理方式：继续执行原则抽象、落点发现、查重、冲突检查和模式处理。

- 提案候选
  - 放行条件：候选状态是精确但受阻。
  - 处理方式：生成精确规则文本，并把未确认、待验证、冲突或权限条件写成直接阻塞原因。
  - 禁止：任何模式都不得直接写入。

- 不沉淀候选
  - 放行条件：候选状态是无精确候选，或用户明确禁止泛化。
  - 处理方式：跳过原则抽象、落点发现和查重，进入安全验证并说明缺少稳定规则或用户范围限制。
  - 禁止：不得补造规则文本。

## 原则抽象

- 保留会改变未来决策的判断条件、动作与禁止项。
- 删除事故名称、日期、客户名、ticket、私有路径与原始日志。
- 当前任务参数与通用原则并存时，只抽象通用原则。
- 遵循目标规则源已有的语言和结构。

## 落点发现

### 权威落点

按以下优先级找权威落点：

- 用户明确指定的位置。
- 当前任务正在编辑或审查的规则文件。
- 项目已有 agent 指令文件，例如 `AGENTS.md` 或 `CLAUDE.md`。
- 已由项目读取入口明确路由的规则文件。

- 项目级 Skill 候选
  - 放行条件：Skill 由当前项目拥有并纳入版本控制。
  - 放行条件：反馈直接约束该 Skill 的行为。
  - 放行条件：未来项目 agent 会加载或按现有路由读取该 Skill。
  - 禁止：不修改用户级、全局安装或第三方 Skill。

- 未路由规范文档候选
  - 放行条件：文档由当前项目维护。
  - 放行条件：文档承担长期规范或约束职责。
  - 放行条件：文档主题与反馈相同。
  - 放行条件：文档路径稳定且没有同等强度的其他候选。
  - 禁止：不把介绍型 README、实施计划、临时 spec、调研记录、生成文件或第三方文档当成权威落点。

- 先用当前上下文与已知规则源定位。
- 已知来源不足时，用 `rg --files` 有边界地查找 `AGENTS.md`、`CLAUDE.md`、`README.md`、`docs/**` 和已知 agent 规则路径。
- 项目级 Skill 候选全部放行时，追加查找 `skills/**/SKILL.md` 与 `skills/**/references/**`。

- 从用户反馈提取具体主题词，再对候选文件做 `rg -n` 查找。
- 禁止默认扫描全库。
- 禁止把 grep 命中直接当成权威落点。
- 禁止因为本技能位于 skills 仓库，就默认把其他项目反馈写进 `SKILL.md`。

唯一权威落点必须同时满足：

- 文件是项目长期规则源。
- 命中段落与抽象规则属于同一决策主题。
- 新规则能合并进已有段落，或创建最小相邻段落。
- 没有同等强度的反向规则。
- 不需要用户裁决才能选择位置。

- 找不到唯一权威落点时，不随机写入 README 或创建 docs 文件。
- 无权威落点时给出建议的最小目标、精确规则文本与缺失证据。

### 读取入口

- 读取入口是宿主会自动加载的项目指令文件。
- 读取入口只保存规则路由，不复制权威落点中的规则。

- 读取路径分支
  - 已有权威落点与读取路径：使用该权威落点。
  - 已有权威落点但缺少读取路径：提案写入权威落点，并在现有读取入口增加最小路由。
  - 已有规范文档但没有读取入口：提案创建当前宿主的最小读取入口，并路由到该规范文档。
  - 没有权威落点也没有读取入口：提案创建当前宿主的最小读取入口，并由该入口承担权威规则源。
  - 禁止：安全沉淀模式不自动增加路由。
  - 禁止：没有项目证据时不为多个宿主同时创建入口。

- 新建读取入口或增加路由必须获得用户批准。
- 找不到未来 agent 读取路径证据时不自动写入。

## 查重与查冲突

- 在目标段落、相邻规则与同主题候选文件中查找相同语义。
- 反馈与已有规则语义相同时，判为 `已有规则覆盖`。
- 既有规则覆盖候选的判断条件、动作、禁止项和验证方式时，才判为 `已有规则覆盖`。
- 既有规则只表达宽泛目标、质量愿景或同主题名词时，优先收紧既有规则。
- 禁止因主题相同就判为 `已有规则覆盖`。
- 重复、仅为示例或可通过收紧原规则处理时，不新增规则。
- 查找同主题的反向规则、例外与优先级约束。
- 冲突时不覆盖旧规则。
- 冲突时记录文件、标题或行号，以及需要用户裁决的具体差异。

## 模式处理

- 模式路由
  - 安全沉淀：可写入候选已有权威落点与读取路径且全部安全门通过时，直接更新项目已有规则源。
  - 安全沉淀：可写入候选缺少权威落点、读取路径或写入授权时，只输出精确提案。
  - 安全沉淀：提案候选只输出精确提案。
  - 审核后沉淀：可写入候选与提案候选都先生成目标位置、精确提案、原因与证据。
  - 审核后沉淀：批准只覆盖用户确认的提案。
  - 不沉淀候选：任何模式都不得写入或生成虚假提案。
  - 关闭模式手动调用：只处理用户本次指定的反馈。
  - 关闭模式手动调用：用户明确要求写入或批准精确变更时，才进入写入步骤。
  - 通用放行条件：写入必须通过 `# Agent Evolve 安全验证` 全部安全门。
  - 禁止：不扩展到其他候选或位置。

## 提案推荐

- 每个 `已提案` 必须识别阻止本轮写入的直接条件。
- `推荐操作` 只推荐解除该阻塞条件所需的最小下一步。
- 项目证据不足以推荐唯一方案时，列出需要用户裁决的选项。
- 推荐操作完成后重新验证全部安全门。

## 写入流程

- 修改前重新读取目标文件，并比较目标内容是否发生变化。
- 目标变化时重新查重和查冲突。
- 无法安全合并并发变化时不写入。
- 默认只编辑唯一权威落点。
- 用户批准建立读取路径时，额外编辑所需的最小读取入口。
- 优先合并或收紧已有规则。
- 不制造第二权威位置。
- 保留工作区已有用户改动。
- 禁止回退、覆盖或清理无关用户改动。
- 写入后重新读取所有修改文件。
- 完成候选判断、落点发现、查重与模式处理后，进入安全验证阶段。
- 进入安全验证阶段时，读取相对 `validation.md`。
- 安全验证结果只使用 `validation.md` 定义的回执格式。
- 禁止用自由摘要替代安全验证回执。
