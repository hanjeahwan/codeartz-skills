# 项目知识奠基

## 目标

根据现有代码建立或刷新可追溯的项目知识基础。先识别会改变未来 agent 决策的知识；需要继续处理时维护可跨会话跟踪的草稿与待裁决项，用户批准最终内容后再合并为正式文件。

## 全局边界

- 需要草稿时固定使用 `.codeartz/project-foundation/draft.md` 跟踪本次奠基工作。
- 固定使用 `docs/project-foundation-baseline.json` 保存上一次成功落地的长期基线。
- 草稿是工作产物，不是正式规则源。
- 长期基线是维护状态，不是正式规则源。
- 用户最终批准前，只允许创建或更新草稿。
- 用户最终批准前，不创建或修改草稿中的正式目标文件。
- 不把代码中存在的写法直接判定为项目希望长期保持的规范。
- 不预设项目必须拥有固定目录或固定文档集合。
- 不创建没有仓库证据或明确读取情境的知识文件。
- 不修改用户级、全局安装或第三方资料。
- 保留工作区已有改动。
- 禁止覆盖、回退或清理无关改动。

## 阶段路由

下游阶段只消费身份未变化的上游产物。消费前重新读取并重算产物身份；身份变化时，使旧验证、批准或合并资格失效，并返回最早拥有该输入的阶段。用户批准只绑定精确草稿 hash；正式目标变化必须重新执行冲突检查。

### 第一阶段：准备草稿与处理裁决

- 触发条件：草稿不存在，草稿尚未完成，或用户要求刷新项目知识。
- 读取手册：`references/prepare.md`。
- 放行条件：待裁决项均已解决，草稿包含全部目标文件的唯一精确最终内容。
- 失败处理：在同一份草稿中记录诊断受限范围、失败证据、不确定性或未完成裁决，继续停留在第一阶段。

### 第二阶段：验证草稿

- 触发条件：第一阶段放行条件满足。
- 读取手册：`references/validate.md`。
- 放行条件：确定性验证与语义验证均通过，用户明确批准已验证 hash 对应的唯一最终内容。
- 失败处理：报告实际失败证据并返回第一阶段。

### 第三阶段：合并正式文件

- 触发条件：第二阶段放行条件满足。
- 读取手册：`references/merge.md`。
- 完成条件：已合并批准内容，重新读取正式文件，更新长期基线，并在草稿中记录合并与验证结果。
- 失败处理：记录实际阻塞；发生内容冲突时返回第一阶段。

## 禁止动作

- 禁止预读后续阶段手册。
- 禁止用聊天回复代替草稿产物。
- 禁止用草稿替代正式项目知识文件。
- 禁止把未裁决内容合并进正式文件。
- 禁止把普通讨论、模糊同意或部分裁决当成最终批准。
- 禁止在未获明确请求时运行付费的新 Agent 会话验证。
