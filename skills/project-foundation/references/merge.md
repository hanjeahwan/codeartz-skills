# 第二阶段：合并正式文件

## 目标

把用户批准的最终草稿安全合并进正式项目知识文件，并留下可核对的合并与验证结果。

## 批准门

- 检查草稿的当前阶段是 `待最终批准`。
- 检查草稿的批准状态是 `已批准`，或当前用户消息明确批准精确目标与完整内容。
- 检查全部待裁决项已经解决。
- 检查批准范围覆盖准备写入的每个目标。
- 任一检查失败时返回第一阶段。
- 禁止用普通同意、部分裁决或早期草稿的批准替代最终批准。

## 合并前检查

- 重新读取草稿。
- 重新读取全部正式目标；目标不存在时记录不存在。
- 重新读取直接支持批准内容的关键仓库证据。
- 比较正式目标与草稿基线后的变化。
- 检查批准内容与正式目标是否重复。
- 检查批准内容与正式目标是否冲突。
- 无法安全合并变化时停止写入。
- 内容冲突时把差异写入新的待裁决项。
- 内容冲突时把当前阶段改为 `待裁决`。

## 合并规则

- 只创建或修改批准范围内的正式目标。
- 正式目标不存在时写入批准的完整内容。
- 正式目标存在时按其现有结构合并批准内容。
- 读取入口只写项目级指令与任务路由。
- 详细规则只写入其正式知识文件。
- 保留正式目标中的无关内容和用户改动。
- 禁止顺手创建未批准的目录或知识文件。

## 长期基线

- 固定目标为 `docs/project-foundation-baseline.json`。
- 长期基线是 Project Foundation 的维护状态。
- 禁止把长期基线加入 `AGENTS.md` 或其他正式读取路由。
- 创建或更新长期基线必须在最终批准范围内。
- 长期基线路径被 ignore 规则命中时停止更新。
- 长期基线路径被 ignore 规则命中时写入待裁决。
- 全部正式目标成功合并并重新读取后，才更新长期基线。
- 任一正式目标失败时保留上一份长期基线。

长期基线使用以下 JSON 合同：

```json
{
  "version": 1,
  "updatedAt": "<ISO 8601>",
  "repository": {
    "gitCommit": null,
    "checkedScopes": ["<path>"],
    "uncheckedScopes": ["<path>"]
  },
  "knowledgeFiles": ["<path>"],
  "knowledge": [
    {
      "id": "K1",
      "location": "<file#section>",
      "summary": "<abstract project knowledge>",
      "evidence": ["<file#location>"],
      "lastVerifiedAt": "<ISO 8601>"
    }
  ]
}
```

- 非 Git 仓库将 `gitCommit` 写为 `null`。
- Git 仓库将 `gitCommit` 写为完整提交标识。
- `checkedScopes` 与 `uncheckedScopes` 必须显式记录。
- `knowledgeFiles` 只列成功合并并重新读取的正式知识文件。
- `knowledge` 只保存抽象结论、正式位置、证据位置与上次验证时间。
- 禁止在长期基线中保存长代码、长日志、密钥或敏感原文。

## 验证与收尾

- 写入后重新读取每个正式目标。
- 检查正式目标包含批准内容。
- 检查读取入口中的路由指向实际存在的目标。
- 运行与文档结构直接相关的低成本静态验证。
- 只有用户明确请求时才启动新的 Agent 会话验证读取效果。
- 在草稿中记录每个目标的合并结果。
- 在草稿中记录实际验证及其结果。
- 更新后重新读取 `docs/project-foundation-baseline.json`。
- 长期基线更新成功时记录本轮已建立增量刷新能力。
- 长期基线未更新时说明下次必须进行有边界的全量诊断。
- 全部目标成功时把当前阶段改为 `已落地`。
- 全部目标成功时记录 `批准状态：已批准`。
- 长期基线在批准范围内但未成功更新时，禁止把当前阶段改为 `已落地`。
- 合并失败时记录实际错误，不声称已落地。
- `已落地` 草稿只保留为工作记录，不加入正式读取路由。
