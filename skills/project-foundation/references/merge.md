# 第三阶段：合并正式文件

## 目标

把用户批准的唯一最终内容合并进正式项目知识，验证后更新最小长期基线。

## 批准门

- 检查待裁决项已全部解决。
- 检查草稿包含唯一最终内容。
- 检查用户明确批准精确目标与最终内容。
- 任一内容或批准检查失败时返回第一阶段。
- 运行 `node <skill-root>/scripts/validate-draft.ts .codeartz/project-foundation/draft.md`。
- 检查确定性验证通过。
- 检查当前 `draftSha256` 与用户批准的 hash 一致。
- 任一验证或 hash 检查失败时返回第二阶段。
- 禁止用普通同意、部分裁决或早期草稿批准替代最终批准。

## 合并前检查

- 重新读取草稿。
- 重新读取全部正式目标。
- 重新读取直接支持批准内容的关键仓库证据。
- 比较正式目标与草稿建立后的变化。
- 检查批准内容与正式目标是否重复或冲突。
- 无法安全合并时停止写入。
- 发生冲突时把差异写回待裁决。

## 合并规则

- 只创建、修改或删除批准范围内的正式目标。
- 新文件写入批准的完整内容。
- 现有文件按批准的锚点与精确内容合并。
- 读取入口只写项目级指令与任务路由。
- 保留正式目标中的无关内容和用户改动。
- 禁止顺手创建未批准的目录或文件。

## 验证与长期基线

- 写入后重新读取每个正式目标。
- 按草稿中的验证计划执行低成本静态验证。
- 检查读取入口中的路由指向实际存在的项目共享路径。
- 全部正式目标成功合并并重新读取后，写入已批准的 `docs/project-foundation-baseline.json`。
- 任一正式目标或验证失败时，保留上一份长期基线。
- 长期基线路径被 ignore 规则命中时停止写入。
- 重新读取已写入的长期基线。
- 在草稿中记录合并与验证结果。
- 只有用户明确请求时才启动新 Agent 会话验证。
- 合并或验证失败时不声称已落地。
- 成功落地后，草稿可删除或归档为工作记录。
- 保留的草稿不加入正式读取路由。
