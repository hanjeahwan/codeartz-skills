# 第一阶段：诊断项目

## 目标

从仓库事实建立诊断基线、证据清单和项目知识地图。刷新时同时对账已有知识并发现新知识。

## 执行规则

### 诊断范围

- 先读取用户指定的范围与已有草稿。
- 使用 `rg --files` 了解仓库结构。
- 检查依赖清单、构建配置、运行脚本、入口、源码目录与测试目录。
- 检查仓库已有的 agent 指令、规范、设计资料和局部 README。
- 根据仓库技术栈追加相关配置与代表性实现。
- 优先抽样能证明边界的文件，不默认逐文件读取全库。
- 未读取的关键范围写入草稿。
- 读取失败的范围写明实际错误。

### 诊断类型

- 首次奠基
  - 触发条件：项目没有可靠的正式知识基础。
  - 处理方式：按仓库结构、代表性证据、异常与冲突进行分层诊断。
- 项目知识刷新
  - 触发条件：项目已有正式知识，需要根据当前代码检查漂移。
  - 处理方式：对账已有知识。
  - 处理方式：发现代码演进产生的新知识。

### 项目知识刷新

- 已有知识对账
  - 检查项：正式知识的证据是否仍然存在。
  - 处理方式：将结论标记为仍成立、已失效、已漂移、冲突或待裁决。
- 新知识发现
  - 检查项：基线后是否出现新模块、新依赖、新配置、新入口、新测试或新知识主题。
  - 处理方式：按知识主题、适用范围和读取情境建立新候选。

- 可靠基线存在
  - 处理方式：优先检查基线后新增、修改和删除的文件。
  - 处理方式：追踪变化影响的依赖方与正式知识。
- 结构复查
  - 检查项：顶层目录、应用、包、依赖、构建命令、测试命令、部署入口与读取路由。
  - 处理方式：结构出现新能力时再对对应区域做定向诊断。
- 基线无法可靠比较
  - 失败条件：基线不存在、基线无法读取、正式知识缺少证据索引，或仓库发生广泛重构。
  - 处理方式：回到有边界的全量诊断。
  - 禁止：不假设未变更区域仍然成立。

### 证据分类

- 已证明事实
  - 放行条件：配置、目录、依赖、测试或代码能够直接证明结论。
  - 处理方式：记录来源文件与具体位置。
- 稳定模式
  - 放行条件：多个独立位置采用一致做法，但没有证据证明它是明确规范。
  - 处理方式：标为候选规则。
- 设计推断
  - 放行条件：结论依赖对结构、依赖或命名意图的解释。
  - 处理方式：记录推断链并进入待裁决。
- 冲突信号
  - 失败条件：同一主题存在不兼容做法或反向证据。
  - 处理方式：列出差异并进入待裁决。
- 知识缺口
  - 失败条件：代码无法回答会影响正式知识结构的问题。
  - 处理方式：进入待裁决。
- 明显技术债
  - 失败条件：测试、注释、迁移状态或项目资料证明当前实现不是目标状态。
  - 处理方式：记录为排除证据。
  - 禁止：不升级为规范。

### 知识地图

- 为每个候选知识单元记录知识主题。
- 为每个候选知识单元记录适用范围。
- 为每个候选知识单元记录未来 agent 的读取情境。
- 知识主题包括但不限于架构、领域、开发约定、用户界面与用户体验（UI/UX）、视觉设计、国际化（i18n）、数据、接口（API）、安全、运维、测试与工作流程。
- 主题列表只用于识别，不作为必须创建的目录模板。
- 同一读取情境的少量内容优先合并进一个目标。
- 内容形成稳定且独立的知识职责时才建议拆分文件。

### 目标选择

- 按知识职责和真实项目边界建议目标文件。
- 优先沿用项目已有命名与目录结构。
- `AGENTS.md` 或 `CLAUDE.md` 只在当前宿主或项目证据支持时作为读取入口候选。
- 读取入口只负责项目级指令和任务路由。
- 详细知识写入相应的正式知识文件草稿。
- 路径选择会改变长期知识结构时进入待裁决。
- 项目证据不足时不同时建议多个宿主入口。
- 检查 `docs/project-foundation-baseline.json` 是否被 ignore 规则命中。
- 长期基线路径可追踪时，将建立或更新该文件列入建议目标。
- 长期基线路径被 ignore 规则命中时，将持久化方式写入待裁决。

## 阶段产物

- 创建或更新 `.codeartz/project-foundation/draft.md`。
- 写入状态、诊断范围、项目知识现状、证据清单和项目知识地图。
- 写入建议目标文件及每个目标的职责、依据和读取情境。
- 写入本轮建议的长期基线完整 JSON 草稿。
- 尚未生成完整内容的目标标记为 `待生成`。
- 将当前阶段标记为 `草稿待审` 或 `待裁决`。
