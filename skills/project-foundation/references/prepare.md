# 第一阶段：准备草稿与处理裁决

## 目标

诊断项目、维护完整草稿并处理用户裁决，直到形成可被精确批准的最终内容。

## 诊断

### 诊断范围

- 先读取用户指定的范围、已有草稿与长期基线。
- 使用 `rg --files` 了解仓库结构。
- 检查依赖清单、构建配置、运行脚本、入口、源码目录与测试目录。
- 检查仓库已有的 agent 指令、规范、设计资料和局部 README。
- 检查受版本控制的符号链接与悬空入口。
- 根据仓库技术栈追加相关配置与代表性实现。
- 优先抽样能证明边界的文件，不默认逐文件读取全库。
- 未读取的关键范围写入草稿。
- 读取失败的范围写明实际错误。

### 诊断类型

- 首次奠基
  - 触发条件：项目没有可靠的正式知识基础。
  - 处理方式：按仓库结构、代表性证据、异常与冲突进行分层诊断。
- 项目知识刷新
  - 触发条件：项目已有正式知识，需要根据当前代码检查漂移。
  - 处理方式：对账已有知识。
  - 处理方式：发现代码演进产生的新知识。

### 项目知识刷新

- 已有知识对账
  - 检查项：正式知识的证据是否仍然存在。
  - 处理方式：将结论标记为仍成立、已失效、已漂移、冲突或待裁决。
- 新知识发现
  - 检查项：基线后是否出现新模块、新依赖、新配置、新入口、新测试或新知识主题。
  - 处理方式：按知识主题、适用范围和读取情境建立新候选。
- 可靠基线存在
  - 处理方式：优先检查基线后新增、修改和删除的文件。
  - 处理方式：追踪变化影响的依赖方与正式知识。
- 结构复查
  - 检查项：顶层目录、应用、包、依赖、构建命令、测试命令、部署入口与读取路由。
  - 处理方式：结构出现新能力时再对对应区域做定向诊断。
- 基线无法可靠比较
  - 失败条件：基线不存在、基线无法读取、正式知识缺少证据索引，或仓库发生广泛重构。
  - 处理方式：回到有边界的全量诊断。
  - 禁止：不假设未变更区域仍然成立。

## 证据与知识地图

### 证据分类

- 已证明事实
  - 放行条件：配置、目录、依赖、测试或代码能够直接证明完整结论。
  - 处理方式：每条只写一个可直接核对的陈述。
  - 处理方式：记录来源文件与具体位置。
- 事实夹带解释
  - 失败条件：同一陈述同时包含可直接证明的现状与对意图、产品性质或长期方向的解释。
  - 处理方式：拆成已证明事实与设计推断。
- 稳定模式
  - 放行条件：多个独立位置采用一致做法，但没有证据证明它是明确规范。
  - 处理方式：标为候选规则。
- 设计推断
  - 放行条件：结论依赖对结构、依赖、命名或实现意图的解释。
  - 处理方式：记录推断链并进入待裁决。
- 规范性结论
  - 检查项：结论是否包含“唯一权威”“已废弃”“必须”“禁止新增”或等价含义。
  - 放行条件：已有项目规则或用户裁决直接支持该结论。
  - 失败条件：只有使用量、目录位置、依赖数量或 Skill 存在证据。
  - 处理方式：降级为设计推断或待裁决。
- 冲突信号
  - 失败条件：同一主题存在不兼容做法或反向证据。
  - 处理方式：列出差异并进入待裁决。
- 知识缺口
  - 失败条件：代码无法回答会影响正式知识结构的问题。
  - 处理方式：进入待裁决。
- 明显技术债
  - 失败条件：测试、注释、迁移状态或项目资料证明当前实现不是目标状态。
  - 处理方式：记录为排除证据。
  - 禁止：不升级为规范。

### 知识地图

- 为每个候选知识单元记录知识主题。
- 为每个候选知识单元记录适用范围。
- 为每个候选知识单元记录未来 agent 的读取情境。
- 知识主题包括但不限于架构、领域、开发约定、用户界面与用户体验（UI/UX）、视觉设计、国际化（i18n）、数据、接口（API）、安全、运维、测试与工作流程。
- 主题列表只用于识别，不作为必须创建的目录模板。
- 同一读取情境的少量内容优先合并进一个目标。
- 内容形成稳定且独立的知识职责时才建议拆分文件。

### 目标选择

- 按知识职责和真实项目边界建议目标文件。
- 优先沿用项目已有命名与目录结构。
- `AGENTS.md` 或 `CLAUDE.md` 只在当前宿主或项目证据支持时作为读取入口候选。
- 读取入口只负责项目级指令和任务路由。
- 详细知识写入相应的正式知识文件草稿。
- 路径选择会改变长期知识结构时进入待裁决。
- 项目证据不足时不同时建议多个宿主入口。
- 检查 `docs/project-foundation-baseline.json` 是否被 ignore 规则命中。
- 长期基线路径可追踪时，将建立或更新该文件列入建议目标。
- 长期基线路径被 ignore 规则命中时，将持久化方式写入待裁决。

## 草稿合同

固定使用以下一级区块并保持顺序：

```markdown
# 项目知识奠基草稿

## 状态

## 诊断范围

## 基线来源

## 项目知识现状

## 项目知识地图

## 证据清单

## 刷新结果

## 目标文件

## 待裁决

## 用户裁决

## 最终批准

## 合并记录
```

### 状态与基线来源

- `当前阶段` 只使用：`诊断中`、`草稿待审`、`待裁决`、`待最终批准`、`合并中`、`已落地`。
- 首次奠基时写明长期基线不存在。
- 刷新时读取 `docs/project-foundation-baseline.json`。
- 记录本轮使用的基线提交、已检查范围与未检查范围。
- 长期基线缺失、损坏或版本不支持时，记录回到有边界全量诊断的原因。
- Git 仓库可以记录提交与工作区状态。
- 非 Git 仓库记录已检查文件及其当前内容事实。
- Git 不是使用本技能的前置条件。

### 证据、刷新结果与目标

- 每条证据使用稳定编号 `E1`、`E2`。
- 每个目标使用稳定编号 `T1`、`T2`。
- 每个目标记录建议路径、知识职责、路径依据、读取情境与状态。
- 每个目标包含准备写入正式文件的完整精确内容。
- 正式内容只引用已证明事实、用户已确认规则和已裁决推断。
- 稳定模式在用户确认前保持候选状态。
- 对每条已有正式知识记录仍成立、已失效、已漂移、冲突或待裁决。
- 对每个新知识候选记录变化证据。
- 将保留、修改、删除、新增与待裁决分开记录。
- 将 `docs/project-foundation-baseline.json` 的完整 JSON 列为目标草稿。
- 禁止在目标草稿中保留占位符。

## 裁决与批准

### 待裁决

- 每个待裁决项使用稳定编号 `D1`、`D2`。
- 每项只处理一个会改变正式内容或知识结构的问题。
- 每项记录问题、证据、推荐、选项和影响的目标。
- 推荐必须说明仓库证据。
- 证据无法支持唯一方向时，不伪造推荐。
- 没有待裁决项时写 `不适用`。

### 用户裁决

- 逐项记录用户对 `D*` 的原始语义与采用结果。
- 用户裁决后更新受影响的完整目标内容。
- 用户只完成部分裁决时保留其他待裁决项。
- 禁止把一个裁决扩展到未提及的目标或问题。

### 最终批准

- 全部待裁决项解决后，将当前阶段改为 `待最终批准`。
- 展示每个目标路径及其完整最终内容。
- 列出准备创建、修改和保持不变的文件。
- 请求用户批准精确目标与精确内容。
- 用户要求调整时更新草稿，并继续停留在第一阶段。
- 只有明确批准最终草稿时才记录 `批准状态：已批准`。

## 写入边界

- 本阶段只创建或更新 `.codeartz/project-foundation/draft.md`。
- 本阶段不创建草稿中列出的正式目标。
- 本阶段不修改已经存在的正式目标。
- 草稿中保留未检查范围和不确定性。
- 禁止为了让草稿看起来完整而补造项目事实。
