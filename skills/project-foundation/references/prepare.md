# 第一阶段：准备草稿与处理裁决

## 目标

从仓库证据提出项目知识变更，按顺序解决待裁决项，再生成唯一最终草稿供用户批准。

## 知识识别

- 开始诊断前完整读取 `discover.md`。
- 使用 `discover.md` 识别候选知识并完成放行检查。
- 首次诊断没有候选通过知识门槛时，按 `discover.md` 返回没有奠基变更并停止，不创建草稿。
- 识别结果直接写入本手册规定的五个一级区块。
- 禁止为识别结果创建自定义一级区块。
- 候选知识未通过放行检查时继续维护同一份五段草稿。

## 诊断路由

### 首次奠基

- 触发条件：项目没有可靠的正式知识基础。
- 处理方式：按仓库结构、代表性证据、异常与冲突进行分层诊断。

### 项目知识刷新

- 触发条件：项目已有正式知识。
- 处理方式：使用长期基线的 `sourceCommit` 检查后续代码变化。
- 处理方式：重新读取长期基线中的 `knowledgeFiles`。
- 处理方式：将已有知识标记为仍成立、已失效、已漂移、冲突或待裁决。
- 处理方式：检查新模块、新依赖、新配置、新入口、新测试与新知识主题。
- 处理方式：每个新知识主题必须在草稿中记录建议变更、待裁决或不变理由。
- 长期基线不存在、无法读取或无法比较时，回到有边界的全量诊断。

## 草稿产物

开始维护草稿前加载 `assets/draft-template.md`。该模板是一级区块名称与顺序的唯一权威；只填写区块内容，不新增一级区块。

- 禁止新增其他一级区块。
- 对账结论写入“检查范围”。
- 证据与不确定性写入对应的“建议变更”或“待裁决”。
- 合并状态只在第三阶段写入现有区块，不新增一级区块。

### 建议变更

- 每个建议记录目标、动作、精确内容、依据与不确定性。
- 每个建议沿用第一阶段确定的知识类型与证据范围。
- 动作只使用创建、修改或删除。
- “保持不变”不是建议动作。
- 没有合格变更证据的现有文件只在最终文件清单中列为保持不变。
- 每项建议必须满足本手册的“可直接应用合同”。
- 待裁决项未解决时，建议只展示会被裁决改变的内容，不生成组合文件变体。
- 禁止把未裁决方向写成已采用的正式内容。

### 待裁决

- 先处理会改变目标路径、知识职责或文件结构的裁决。
- 结构裁决完成后，再生成相关目标内容。
- 内容仍存在分歧时，再继续裁决精确文本。
- 每项记录问题、证据、建议、选项和受影响的目标。
- 用户只处理部分裁决时，保留其他未解决项。
- 不影响本轮目标或合并决策的问题不写入待裁决。
- 证据无法支持唯一方向时，不伪造建议。

### 最终内容

- 生成最终内容前执行最终内容门。
- 每项最终内容必须通过“可直接应用合同”。
- 长期基线使用唯一的完整 JSON。
- 禁止在最终内容中保留占位符、未裁决分支或待生成内容。
- 列出准备创建、修改、删除和保持不变的文件。
- 用户要求调整时更新同一份草稿。
- 唯一最终内容生成完成后进入第二阶段。

### 最终内容门

- 检查草稿是否仍有未解决的待裁决项。
- 仍有待裁决项时，“最终内容”只写等待裁决。
- 仍有待裁决项时，跳过可应用正文与长期基线 JSON。
- 待裁决项全部解决时，才生成唯一最终内容。
- 禁止选择默认选项代替用户裁决。

### 可直接应用合同

| 动作 | 必需内容                                     |
| ---- | -------------------------------------------- |
| 创建 | 完整文件内容                                 |
| 修改 | 唯一可定位的标题或原文锚点，以及完整替换内容 |
| 删除 | 精确删除边界                                 |

- 修改目录树、列表或表格时，同时提供维持结构正确所需的相邻内容。
- 依赖省略、占位、隐含步骤或 Agent 自行补全的内容不得进入建议或最终内容。
- 任一检查失败时继续停留在第一阶段。

### 验证计划

- 列出每个目标合并后的重新读取与低成本静态验证。

## 长期基线

`docs/project-foundation-baseline.json` 使用以下合同：

```json
{
  "version": 1,
  "sourceCommit": "<full commit or null>",
  "knowledgeFiles": ["<path>"]
}
```

- `sourceCommit` 记录本轮诊断所使用的代码快照。
- Git 仓库将 `sourceCommit` 写为完整提交标识。
- 非 Git 仓库将 `sourceCommit` 写为 `null`。
- `knowledgeFiles` 只列最终内容中的正式项目知识文件。
- 长期基线只在待裁决项全部解决后生成。
- 检查 `docs/project-foundation-baseline.json` 是否被 ignore 规则命中。
- 长期基线路径被 ignore 规则命中时，将持久化方式写入待裁决。

## 写入边界

- 需要继续处理合格候选或已有草稿时，本阶段只创建或更新 `.codeartz/project-foundation/draft.md`。
- 本阶段不修改草稿中列出的正式目标。
- 草稿中保留未检查范围和不确定性。
- 禁止为了让草稿看起来完整而补造项目事实。
