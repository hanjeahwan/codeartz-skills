# 第二阶段：生成草稿与处理裁决

## 目标

在同一工作包中维护完整目标内容、证据、待裁决项与用户裁决，直到形成可被精确批准的最终草稿。

## 草稿结构

使用以下一级区块并保持顺序：

```markdown
# 项目知识奠基草稿

## 状态

## 诊断范围

## 基线来源

## 项目知识现状

## 项目知识地图

## 证据清单

## 刷新结果

## 目标文件

## 待裁决

## 用户裁决

## 最终批准

## 合并记录
```

### 状态

- `当前阶段` 只使用：`诊断中`、`草稿待审`、`待裁决`、`待最终批准`、`合并中`、`已落地`。
- 记录草稿建立时的仓库基线。
- Git 仓库可以记录提交与工作区状态。
- 非 Git 仓库记录已检查文件及其当前内容事实。
- Git 不是使用本技能的前置条件。

### 基线来源

- 首次奠基时写明长期基线不存在。
- 刷新时读取 `docs/project-foundation-baseline.json`。
- 记录本轮使用的基线提交、已检查范围与未检查范围。
- 长期基线缺失、损坏或版本不支持时，记录回到有边界全量诊断的原因。

### 刷新结果

- 对每条已有正式知识记录仍成立、已失效、已漂移、冲突或待裁决。
- 对每个新知识候选记录变化证据。
- 将保留、修改、删除、新增与待裁决分开记录。
- 删除正式知识与新增正式知识使用相同的裁决与批准门。

### 证据与目标

- 每条证据使用稳定编号 `E1`、`E2`。
- 每个目标使用稳定编号 `T1`、`T2`。
- 每个目标记录建议路径、知识职责、路径依据、读取情境与状态。
- 每个目标包含准备写入正式文件的完整精确内容。
- 正式内容只引用已证明事实、用户已确认规则和已裁决推断。
- 稳定模式在用户确认前保持候选状态。

### 待裁决

- 每个待裁决项使用稳定编号 `D1`、`D2`。
- 每项只处理一个会改变正式内容或知识结构的问题。
- 每项记录问题、证据、推荐、选项和影响的目标。
- 推荐必须说明仓库证据。
- 证据无法支持唯一方向时，不伪造推荐。
- 没有待裁决项时写 `不适用`。

### 用户裁决

- 逐项记录用户对 `D*` 的原始语义与采用结果。
- 用户裁决后更新受影响的完整目标内容。
- 用户只完成部分裁决时保留其他待裁决项。
- 禁止把一个裁决扩展到未提及的目标或问题。

### 最终批准

- 全部待裁决项解决后，将当前阶段改为 `待最终批准`。
- 展示每个目标路径及其完整最终内容。
- 列出准备创建、修改和保持不变的文件。
- 请求用户批准精确目标与精确内容。
- 用户要求调整时更新草稿，并继续停留在第二阶段。
- 只有明确批准最终草稿时才记录 `批准状态：已批准`。

## 写入边界

- 本阶段只创建或更新 `.codeartz/project-foundation/draft.md`。
- 本阶段不创建草稿中列出的正式目标。
- 本阶段不修改已经存在的正式目标。
- 草稿中保留未检查范围和不确定性。
- 禁止为了让草稿看起来完整而补造项目事实。
