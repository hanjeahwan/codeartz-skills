# Agentic Design Navigator 设计规范

> 状态:已批准,待实施。
> 日期:2026-07-11。

## 目标

把用户给出的模糊想法、感觉、问题或关键词稳定成可继续设计的意图,再分析设计条件并生成可追溯方案。

- Agent 负责结构化、分析和维护上下文。
- 用户只需用“接近、不接近、哪里不对、哪个更像”校准方向。
- 信息不足时不自动补全成完整项目。
- 最低清晰度未达到时不生成大规模方案。
- 每个方案都能追溯到用户原始表达。
- 讨论发生明显变化时主动检测意图漂移。

## Skill 名称与触发

- 目录名:`skills/agentic-design-navigator/`。
- Skill 名:`agentic-design-navigator`。
- 展示名:`Agentic Design Navigator`。
- description 只描述触发条件,不摘要执行流程。
- 触发面覆盖模糊想法、感觉、问题、关键词、意图不稳、理解冲突和可能发生意图漂移的设计讨论。

## 文件职责

### `SKILL.md`

- 只放触发条件、阶段路由、全局边界和禁止动作。
- 定义三个阶段的进入条件和退出条件。
- 定义最低清晰度关卡。
- 定义持续偏离检测的触发条件和固定输出。
- 要求 Agent 只读取当前阶段手册。
- 禁止预读后续阶段手册。
- 禁止把推断自动升级为用户需求。
- 禁止在最低清晰度前生成大规模方案。

### `references/stabilize-intent.md`

- 保留第一阶段的全部意图发现与稳定规则。
- 保留原始语言复述、2 至 4 种可能理解和单轮单问规则。
- 明确区分用户原话、Agent 推断和当前未知。
- 维护“已确认、临时理解、开放探索、已否定、关键未知”意图账本。
- 定义最低清晰度的四个检查项:改善对象、涉及主体、最重要价值和不可偏离边界。
- 最低清晰度未满足时继续第一阶段。
- 最低清晰度满足时才允许进入第二阶段。

### `references/analyze-design.md`

- 保留第二阶段九个分析维度。
- 分析主体、情境、目的、行动可能、转化条件、结果、价值与代价、公平性和验证。
- 保留“设计要素 → 改变信息、行为、权限或环境 → 提供行动可能 → 在转化条件下被使用 → 产生直接结果 → 形成长期价值或代价”的因果链。
- 区分已知事实、合理推断和未验证假设。
- 为关键假设记录支持证据、可能反证、验证方式和负面指标。
- 第二阶段理解足以约束方案时才允许进入第三阶段。

### `references/generate-options.md`

- 保留第三阶段每个方案的十二项必填合同。
- 每个方案说明原始表达、主体、情境、目的、行动可能、转化条件、结果、价值、代价或风险、公平性、最小验证和偏离关系。
- 至少生成四类方案:直接贴近、轻度增强、不同但有启发、无新增功能的替代方案。
- 偏离关系只使用“直接实现、补充增强、替代实现、延伸探索、可能偏离”。
- 允许字段无内容时显式写“不适用”。
- 禁止为了填满模板补造内容。

## 阶段路由

### 第一阶段:稳定意图

- 触发条件:用户只给出模糊想法、感觉、问题或关键词。
- 读取手册:`references/stabilize-intent.md`。
- 放行条件:已明确大致改善对象、涉及主体、最重要价值和至少一个不可偏离边界。
- 失败处理:继续维护意图账本,每轮只问一个会显著影响方向的问题。
- 禁止:不读取第二或第三阶段手册。

### 第二阶段:建立设计理解

- 触发条件:第一阶段最低清晰度关卡通过。
- 读取手册:`references/analyze-design.md`。
- 放行条件:已形成足以约束方案的主体、情境、目的、行动可能、转化条件、结果、价值代价、公平性和验证判断。
- 失败处理:回到缺失维度继续分析或向用户提出一个关键校准问题。
- 禁止:不读取第三阶段手册。

### 第三阶段:生成并检查方案

- 触发条件:第二阶段理解足以约束方案。
- 读取手册:`references/generate-options.md`。
- 放行条件:四类方案齐全,每个方案完成十二项合同并标注偏离关系。
- 失败处理:补齐缺失合同或回流到最早受影响阶段。
- 禁止:不把未经确认的推断写成需求或成功标准。

## 持续偏离检测

讨论明显变化时检查以下变化:

- 主体是否改变。
- 情境是否改变。
- 原始目的是否被替换。
- 手段是否变成目标。
- 是否新增未经确认的成功标准。
- 是否忽略转化条件。
- 是否为了创新、完整或商业价值偏离用户重点。
- 是否重新引入用户已经否定的方向。

命中偏离时固定输出“这里可能出现了意图漂移。”,并列出:

- 最初的理解。
- 当前的理解。
- 新增、删除、替换或弱化的内容。
- 回到最近一次确认方向的方法。

## 内容保留原则

- 用户提供的规则语义全部保留。
- 原有稳定句子尽量保留原文。
- 只为阶段路由、命名槽位和可独立勾选性调整结构。
- 同一条规则只保留一个权威位置。
- 从 `SKILL.md` 移入阶段手册的详细规则不在入口重复。
- 示例只在能消除歧义时保留一个。

## 验证设计

### RED 基线

在新 Skill 不存在或不加载时运行至少三个压力场景:

1. 用户只说“我想让团队开会没那么累”,同时催促 Agent 直接给完整产品方案。
2. 用户给出模糊教育产品想法,随后否定“提高效率”,再用商业化压力诱导 Agent 重新引入该方向。
3. 用户提出照护场景,要求快速列功能,但没有明确谁获益、谁承担额外劳动或谁无法退出。

记录 Agent 是否过早补全需求、遗漏意图账本、一次询问多个问题、忽略转化条件、遗漏公平性或重新引入已否定方向。

### GREEN 验证

- 对同一批场景显式加载新 Skill。
- 检查 Agent 是否停留在正确阶段。
- 检查每轮是否最多提出一个关键问题。
- 检查推断是否与用户原话、未知分开。
- 检查最低清晰度关卡是否阻止过早方案生成。
- 检查方案是否完成四类方向和十二项合同。
- 检查意图漂移是否使用固定提示和四项差异说明。

### REFACTOR 验证

- 从失败输出提取新的合理化理由。
- 只在对应权威位置补规则。
- 复测相同场景和至少一个变化场景。
- 连续一次完整验证无关键违规后停止。

## 完成标准

- Skill 目录和文件结构符合本设计。
- Frontmatter 通过 skill 验证器。
- 三个阶段手册结构一致。
- 用户原始规则逐条可追溯。
- 没有跨文件重复规则。
- 阶段手册没有超过两层的规则嵌套。
- RED、GREEN 和 REFACTOR 证据完整。
- 仓库测试、格式检查和类型检查通过。
